
Spring 的**三级缓存机制**是 Spring 核心容器（`BeanFactory`）用来解决单例 Bean 循环依赖（Circular Dependency）的核心设计。

简单来说，当 Bean A 依赖 Bean B，而 Bean B 又依赖 Bean A 时，如果不做处理，程序就会陷入死循环。Spring 通过这三层缓存，成功地在“对象实例化”和“对象完全初始化”之间拉开了一个空间，解决了这个难题。

## 什么是三级缓存？

在 `DefaultSingletonBeanRegistry` 类中，定义了这三个 Map 作为缓存：

|**缓存层级**|**属性名称**|**存储的内容**|**作用**|
|---|---|---|---|
|**一级缓存**|`singletonObjects`|**完全初始化好的、成熟的 Bean**|日常业务中直接使用的单例 Bean，拿来即用。|
|**二级缓存**|`earlySingletonObjects`|**半成品的、未完全初始化的 Bean**|已经实例化（分配了内存），但还没进行属性赋值和初始化的 Bean。主要为了避免重复创建代理对象。|
|**三级缓存**|`singletonFactories`|**Bean 对象的 ObjectFactory（工厂 Lambda 表达式）**|用来生成半成品 Bean（或其 AOP 代理对象）的工厂。**这是解决 AOP 循环依赖的核心**。|

## 循环依赖的解决流程（以 A, B 互相依赖为例）

假设 A、B 都是单例（Singleton），且 A 先被加载：

```
A 实例化 -> 放入三级缓存 -> A 注入属性 B -> 发现 B 不存在 -> 去创建 B
B 实例化 -> 放入三级缓存 -> B 注入属性 A -> 去缓存找 A
```

此时的详细流转如下图所示：

1. **A 实例化**：Spring 先把 A 实例化（执行了构造函数，此时是半成品）。
    
2. **A 放入三级缓存**：将 A 包装成一个 `ObjectFactory` 放入**三级缓存**。
    
3. **A 注入 B**：A 发现需要依赖 B，于是触发 `getBean(B)`。
    
4. **B 实例化并放入三级缓存**：B 走同样的流程，实例化后把自己的工厂放入**三级缓存**。
    
5. **B 注入 A（关键点）**：B 发现需要依赖 A，触发 `getBean(A)`。
    
    - **查找缓存**：B 去一级缓存找 A（没有） -> 去二级缓存找 A（没有） -> 去**三级缓存**找到了 A 的 `ObjectFactory`。
        
    - **升级到二级缓存**：B 调用该工厂的 `getObject()` 方法，拿到 A 的引用（如果 A 需要 AOP，此时拿到的是 A 的代理对象）。
        
    - 将 A 放入**二级缓存**，并从**三级缓存**中移除 A。
        
6. **B 完成初始化**：B 成功注入了 A 的引用，继续走完自己的生命周期，最后**进入一级缓存**。
    
7. **A 完成初始化**：B 创建完成后返回给 A，A 成功注入 B，也走完生命周期，从二级缓存移除，**进入一级缓存**。
    

## 核心思考：为什么非要“三级”缓存？两级不行吗？

没问题，之前的解释确实有点绕。我们抛开那些复杂的术语，用最接地气的“大白话+打比方”来解释。

其实核心问题就一个：**如果没有三级缓存（也就是只有两级缓存），Spring 就会变成一个“一刀切”的死板系统。**

我们来对比一下两种情况：

### 1. 正常情况：没有循环依赖

在没有循环依赖时，Spring 创造一个 Bean 的流程就像在工厂流水线上组装一辆汽车：

1. **组装车架**（实例化）
    
2. **安装发动机和轮胎**（属性赋值/依赖注入）
    
3. **最后喷漆、贴膜，让它变帅**（**AOP 代理**，比如加上日志、事务功能）
    
4. **新车出厂**（存入一级缓存，正式使用）
    

这是最完美的顺序。**AOP（喷漆）发生在最后一步。**

### 2. 冲突点：发生了循环依赖，且只有两级缓存

现在，A 和 B 互相依赖了。

A 刚组装完车架（还没安装发动机，也没喷漆），B 就急着用 A。

如果这时候**只有两级缓存**，Spring 只有两个选择：

- **一级缓存**：放完全装好的车。
    
- **二级缓存**：放刚组装好的裸车架。
    

这时候 B 跑过来找 A，如果 Spring 直接把二级缓存里的“裸车架”给 B。等到后面 A 终于走到最后一步喷漆（AOP 代理）时，完蛋了——**A 变成了另外一个形象（代理对象），但 B 手里拿到的还是那个旧的“裸车架”！**

为了解决这个问题，如果只有两级缓存，Spring 就必须被迫修改规则：

> **“所有汽车，只要一完成第一步（组装车架），不管是啥情况，立刻给我喷漆（AOP代理），然后塞进二级缓存！”**

### 3. 为什么 Spring 觉得“一刀切”不行？（为什么必须用三级缓存）

如果按照上面的两级缓存做法，Spring 就能解决循环依赖，但代价是：**所有 Bean 哪怕没有循环依赖，也必须在刚出生的时候就进行 AOP 代理。**

这打破了 Spring 的生命周期原则。Spring 希望 Bean 是“先成为一个完整的自己（属性赋值完），最后才被 AOP 包装”。

#### 💡 三级缓存是怎么当“聪明人”的？

三级缓存不存车架，也不存喷好漆的车，它存的是一个“喷漆锦囊”（ObjectFactory）。

当 A 刚出生时，Spring 把它放进三级缓存（给它一个锦囊）。

- **如果没有循环依赖**：这个锦囊永远不用打开。A 慢悠悠地装发动机、最后正常喷漆，完美！
    
- **如果发生了循环依赖**：B 急着要 A。B 就会跑到三级缓存，把 A 的“锦囊”拆开。锦囊会立刻判断：“A 需不需要喷漆（AOP）？需要是吧？好，我现在提前给 A 喷漆，把喷好漆的 A 代理对象给 B！”
    

### 总结：用一句话看懂

- **二级缓存的坏处**：不管有没有发生冲突，所有 Bean 一出生就必须被迫提前做 AOP 代理，太死板。
    
- **三级缓存的好处（延迟代理）**：平时大家还是按部就班，到了最后一步再做 AOP；**只有当特殊发生循环依赖时**，才通过三级缓存的“锦囊”，“提前”把 AOP 代理对象做出来给对方。
    

这就叫：**不撞南墙不回头，撞了南墙（循环依赖）再提前做 AOP。**
