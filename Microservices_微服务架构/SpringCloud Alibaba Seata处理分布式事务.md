
# 分布式事务问题如何产生？

要理解**分布式事务问题**是如何产生的，我们得先看看在“以前的好日子”里，事务是怎么顺理成章地工作的，以及当我们把系统拆分后，底层的规则发生了什么颠覆性的变化。

简单来说，分布式事务问题的本质就是：**“网络把原本的一家人，变成了各过各的陌生人。”**

## 一、 曾经的温床：单体架构与本地事务

在单体架构时代，你的订单、库存、账户代码都写在同一个项目里，并且连接着**同一个数据库**。

当用户下单时，你的代码可能长这样：

``` java
@Transactional // 开启本地事务
public void createOrder() {
    orderDao.insert();    // 1. 生成订单
    stockDao.deduct();    // 2. 扣减库存
    accountDao.pay();     // 3. 扣减余额
}
```

这时候，生活非常美好。因为 Spring 的 `@Transactional` 底层依赖的是数据库（如 MySQL）的**本地事务机制**。MySQL 会用它强大的原子性（Atomicity）和隔离性（Isolation）向你保证：这三步操作要么**一起成功**，要么只要有一步报错，前面做过的所有修改都会**一起回滚（Rollback）**。数据绝对不可能变脏。

## 二、 惨剧的发生：微服务拆分打破了边界

随着业务发展，你把单体系统拆成了微服务：订单代码进了“订单服务”，库存代码进了“库存服务”，它们各自连着**独立的数据库**。

这时候，同样的下单逻辑变成了这样：

``` java
@Transactional // ❌ 注意：这个本地注解在此刻已经部分失效了！
public void createOrder() {
    orderDao.insert();               // 1. 本地数据库：生成订单 (成功)
    stockFeignClient.deduct();       // 2. 远程调用：扣减库存 (成功)
    accountFeignClient.pay();        // 3. 远程调用：扣减余额 (❌ 突然报错：余额不足！)
}
```

当第三步“扣减余额”由于余额不足抛出异常时，戏剧性的一幕发生了：

1. **订单服务**感知到异常，触发本地回滚，刚刚生成的订单记录在订单库里**消失了**（这很正确）。
    
2. **但是！库存服务**所在的数据库已经高高兴兴地执行完了 `commit`（提交）。因为对库存库来说，它刚才接到的 Feign 调用是一个独立的、完全合法的请求，它根本不知道“远方的账户服务”后来发生了什么。
    

> **结果：** 钱没扣，订单没了，但**库存却平白无故被扣掉了**！数据彻底失去了 consistency（一致性）。这就是最经典的**分布式事务问题**。

## 三、 归根结底，分布式事务问题是由这三大矛盾引起的

为什么我们没办法轻易解决这个问题？因为分布式系统天然存在以下三个不可逆转的物理客观事实：

### 1. 数据库连接与本地事务的失效

原生的数据库事务（如 MySQL 的 Innodb 引擎）是基于数据库连接（Connection）**和**预写日志（WAL/Undo Log）来实现的。当你的操作跨越了多个数据库实例，甚至跨越了网络时，单一的数据库连接根本无法控制别人的核心日志，原本的本地事务锁瞬间鞭长莫及。

### 2. 网络调用的“三分法”状态

在单机环境下，一个方法调用只有两种结果：**成功** 或 **失败**。

但在分布式环境下，由于网络丢包、延迟、机房断电，一次 RPC/HTTP 远程调用有三种状态：

- **成功**
    
- **失败**
    
- **超时（未知状态）：** 你不知道请求是压根没发过去，还是发过去处理完了但在返回时网络断了。这种“未知”是数据一致性的噩梦。
    

### 3. CAP 定理的紧箍咒

分布式领域有一个著名的 **CAP 定理**（一致性 Consistency、可用性 Availability、分区容错性 Partition tolerance）。定理指出，在网络可能出现分区（P）的现实世界里，你**不可能同时完美兼顾**强一致性（C）和高可用性（A）。为了系统的整体吞吐量，我们往往无法采用传统的、让所有人死等的大锁（如 2PC 强一致性），这就必然导致了“短时间内数据不一致”的风险。

## 四、 总结：产生问题的核心场景

在日常开发中，只要满足以下两个条件之一，分布式事务问题就会如期而至：

1. **跨服务调用：** 像上面聊到的，一个业务流程由多个独立的微服务、多个独立的数据库协同完成。
    
2. **单服务跨多库（分库分表）：** 哪怕你没有拆分微服务，代码还在一个项目里，但是由于数据量太大，你把订单库拆成了 `order_db_01` 和 `order_db_02`。只要一个事务里同时操作了这两个库，本地事务同样当场瘫痪。

# Seata简介

接续上一个关于分布式事务的痛点，阿里开源的 **Seata（Simple Extensible Autonomous Transaction Architecture）** 就是为了彻底解决这个问题而诞生的**一站式分布式事务解决方案**。

用一句话来概括 Seata 的江湖地位：**它让你能够像使用本地事务 `@Transactional` 一样，近乎无感地搞定复杂的微服务分布式事务。**

## 一、 Seata 的“三剑客”核心架构

Seata 之所以能掌控全局，是因为它在整个微服务集群中安插了三个核心角色。理解了这三个缩写，你就看懂了 Seata 的大半：

- **TC (Transaction Coordinator) - 事务协调器：**
    
    - **角色：** 独立运行的**中央大团长**（需要单独部署的服务器集群）。
        
    - **职责：** 维护全局事务的运行状态，负责接收 TM 的指令，并最终拍板决定让所有 RM 是一起提交（Commit）还是全网回滚（Rollback）。
        
- **TM (Transaction Manager) - 事务管理器：**
    
    - **角色：** 嵌入在你的**核心业务微服务**（如订单服务）中的代码组件。
        
    - **职责：** 它是全局事务的**发起者**。它负责向 TC 申请：“报告团长，我要开启一个全局事务了（生成一个全局唯一的 XID）”，并在业务结束时通知 TC 是成功了该提交，还是失败了该回滚。
        
- **RM (Resource Manager) - 资源管理器：**
    
    - **角色：** 嵌入在**每一个分支微服务**（如库存服务、账户服务）中的代码组件。
        
    - **职责：** 负责控制分支服务自己的本地数据库。它向 TC 汇报自己这边的分支事务执行得怎么样了，并死死听从 TC 的最终调配（执行本地提交或回滚）。
        

## 二、 Seata 的四大硬核武器（模式）

面对不同的业务场景，Seata 提供了四种截然不同的解决思路（事务模式）。其中 **AT 模式**是无代码侵入的行业明星：

### 1. AT 模式（Automatic Transaction）—— 绝对的主角

- **怎么做：** 几乎不需要你改写任何业务代码，只要在方法上加个 `@GlobalTransactional` 注解。
    
- **底层黑科技：** Seata 在底层拦截了你的 SQL 语句。在你的本地数据库提交前，它会自动帮你把数据修改前后的样子（Before Image / After Image）自动存到一张叫 `undo_log` 的快照表里。
    
    - 如果全局成功：后台异步删掉快照，性能极高。
        
    - 如果有人报错：利用 `undo_log` 的反向 SQL 逆向把数据还原回去。
        
- **适用场景：** 90% 的传统商业、电商系统，只要你用的是关系型数据库（如 MySQL、Oracle）。
    

### 2. TCC 模式（Try-Confirm-Cancel）—— 性能怪兽，手工定制

- **怎么做：** 需要你硬编码手动实现三个方法：`Try`（检查并冻结资源）、`Confirm`（真正执行业务）、`Cancel`（释放冻结的资源）。
    
- **优点：** 不依赖底层数据库的锁和快照，全部在内存和业务层完成，性能高到飞起。
    
- **适用场景：** 金融级、高并发核心链路（如话费充值、银行转账）。
    

### 3. Saga 模式 —— 长事务与老系统的救星

- **怎么做：** 针对每一个正向的业务操作，你都必须配一个反向的“补偿操作”。如果前面某一步卡住了，Seata 会倒序调用补偿操作把状态冲正。
    
- **适用场景：** 业务流程特别长（比如要跑好几天）、或者需要对接第三方外部系统（如调用外部银行接口，你管不到人家的数据库）的场景。
    

### 4. XA 模式 —— 强一致性洁癖者

- **怎么做：** 利用各大数据库原生对 XA 国际标准的分布式事务支持。
    
- **缺点：** 锁持有的时间太长，高并发下性能极差。
    
- **适用场景：** 对数据一致性有极度严苛追求，且并发量不高的核心后台。
    

## 三、 四大模式怎么选？（速查表）

|**模式**|**侵入性（改代码程度）**|**性能**|**一致性级别**|**核心痛点/缺点**|
|---|---|---|---|---|
|**AT**|**无侵入**（最省心）|高|最终一致|存在一小段短暂的数据不一致窗口|
|**TCC**|**高侵入**（写代码写到吐）|极高|最终一致|开发成本极大，需要自己写回滚逻辑|
|**Saga**|高侵入|高|最终一致|无法保证中间状态的隔离性|
|**XA**|无侵入|低|**强一致**|数据库资源锁太久，并发稍微上去就瘫痪|

## 四、 总结：为什么选 Seata？

在 Seata 出现之前，解决分布式事务我们要么手写一堆复杂的 MQ 消息和状态表（非常容易漏掉边界情况），要么用昂贵笨重的传统架构。而 Seata 将这层复杂的逻辑直接**从业务中抽离了出来**，让普通的 Java 程序员只要认识 `@GlobalTransactional`，就能稳妥地守护好微服务的数据安全。


# Seata工作流程简介

如果把分布式事务比作一场需要多方精细协作的“高难度战役”，那么 Seata 的工作流程就是一套精密配合的“两阶段指挥艺术”。

在 Seata 中，虽然有四种模式，但它们的核心骨架都逃不开 **两阶段提交（2PC）** 的底座。我们以最常用、最核心的 **AT 模式**为例，来看看当一个用户下单时，TC、TM 和 RM 是如何各司其职、完成这一套大招的。

## 阶段一：一阶段（准备与就地提交阶段）

在一阶段，各个微服务各过各的，快速执行本地业务，**不长期锁死数据库连接**。这是 Seata 性能高出传统分布式事务的关键。

### 1. TM 开启全局事务

- 当请求来到订单服务（TM），准备执行带有 `@GlobalTransactional` 的方法时，TM 会立刻向中央协调器（TC）汇报：“大团长，我要开打全局事务了！”
    
- TC 收到后，会生成一个在全网唯一的事务身份证，叫做 **XID**。
    

### 2. XID 全网传递

- 订单服务在通过 OpenFeign 远程调用“库存服务”和“账户服务”时，Seata 会偷偷在 HTTP Header（或者 RPC 协议头）里塞入这个 **XID**，像接力棒一样传给下游。
    

### 3. RM 拦截 SQL，保存快照

- 下游的库存服务（RM）收到了请求和 XID。
    
- 在执行你写的那句业务 SQL（比如 `UPDATE stock SET count = count - 1`）之前，Seata 的数据源代理会默默站出来，解析你的 SQL，并自动去数据库里把**数据修改前长什么样**查出来。这个旧数据快照叫 **Before Image（前镜）**。
    

### 4. 本地直接提交

- RM 执行你的业务 SQL，紧接着再次查询修改后的数据，生成 **After Image（后镜）**。
    
- 随后，RM 把前镜、后镜数据打包成一条 JSON 记录，塞进你本地数据库的 `undo_log` 表里。
    
- **高明之处来了：** RM 会把**业务数据的修改**和 **`undo_log` 的插入**，在同一个本地事务里**直接 commit 提交掉！** 提交成功后，向 TC 汇报：“我的一阶段任务打完，安全撤离。”
    

> 💡 **为什么要急着提交本地事务？** > 传统的分布式事务要求大家执行完 SQL 后死等最终指令，谁都不准提交，这会导致数据库连接和行锁被死死卡住。而 Seata 让大家在一阶段就各自落袋为安、释放连接，极大地解放了系统的并发吞吐量。

## 阶段二：二阶段（决议与冲正完成阶段）

当所有微服务的一阶段都跑完后，TM 会根据大家的汇报结果，指示 TC 做出最终裁决。这时候流程会走向两个分叉：

### 情况 A：全网大捷 $\rightarrow$ 全局提交（Global Commit）

如果订单、库存、账户在第一阶段**全部执行成功**：

1. **TM 吹响胜利号角**：通知 TC：“兄弟们都很给力，全局提交！”
    
2. **TC 异步下发指令**：TC 收到后，立刻向所有人下发提交指令。由于一阶段大家已经把业务数据写进库里了，所以这里的提交其实是个“擦屁股”的工作。
    
3. **RM 悄悄删日志**：RM 收到指令后，甚至**不需要同步阻塞**核心业务，而是把任务丢给一个后台异步线程池。线程池慢条斯理地把 `undo_log` 表里对应 XID 的那条快照记录彻底删掉就行了。整个事务宣告完美结束。
    

### 情况 B：有人翻车 $\rightarrow$ 全局回滚（Global Rollback）

如果一阶段里，账户服务突然报错（比如余额不足），或者网络超时导致调用失败：

1. **TM 申请全网撤退**：TM 发现大事不妙，立刻向 TC 报告：“有人拉胯了，赶紧全局回滚！”
    
2. **TC 命令全军撤回**：TC 紧急向所有执行成功的 RM（比如库存服务）发送回滚指令。
    
3. **RM 本地反向补偿**：
    
    - 库存服务（RM）收到回滚指令，开启一个新的本地事务。
        
    - 它通过 XID 去自己的 `undo_log` 表里找到当时存下的那张“死兆星照片”。
        
    - **防御性校验**：RM 会拿快照里的 `After Image`（修改后的样子）和现在数据库里该行的实际数据进行对比。如果一致，说明这期间数据没有被别人偷摸改过（没有脏写）。
        
    - **魔术还原**：校验通过后，Seata 会利用 `Before Image`（修改前的老数据）**在底层自动生成反向的补偿 SQL**（你之前减了 1 个库存，它现在就帮你 `SET count = count + 1`），执行并提交。
        
    - 数据完美退回到下单前的状态，RM 向 TC 汇报回滚成功。
        

## 三、 核心彩蛋：并发安全靠什么？—— 全局锁（Global Lock）

你可能会问：“既然一阶段大家都把本地事务提交了，那要是另一个不归 Seata 管的线程，趁着二阶段还没开始，把数据改了怎么办？那回滚不就乱套了吗？”

为了解决这个问题，Seata 在 TC 端维护了一个 **全局锁（Global Lock）**：

- 在一阶段 RM 准备提交本地事务前，它必须先去 TC 那里申请拿这行数据的“全局锁”（以 `表名 + 主键ID` 作为 Key）。
    
- 只有拿到了全局锁，本地事务才允许 `commit` 提交。
    
- 如果是全局提交（情况 A），这个全局锁会在二阶段异步删除日志时被释放；如果是全局回滚（情况 B），则必须等 RM 把数据反向还原成功后，才会释放全局锁。
    

通过这种“本地锁由数据库管，全局锁由 TC 管”的精妙设计，Seata 既保障了分布式环境下的写隔离，又榨干了微服务架构的运行性能。
















# Seata原理小总结

回顾了 Seata 的架构和详细步骤后，我们可以把 Seata 的核心原理浓缩为“一个中心、两个阶段、三大角色、四种武器”。

这套原理总结不仅能帮你理清知识骨架，也是面对面试官时最完美的“高分小抄”。

## 📌 一个核心：两阶段提交（2PC）的变种

Seata 所有的故事，都是围绕 **2PC（Two-Phase Commit）** 展开的。传统的 2PC 因为长期锁死数据库连接，属于“强一致性”但“低并发”。Seata（尤其是 AT 模式）对其进行了天才般的改良：

- **一阶段：** 敢于**提早释放本地锁**。各微服务直接把本地事务 commit 提交掉，不耽误别人，并发吞吐量飙升。
    
- **二阶段：** 绝大多数情况下都是全网成功，所以二阶段变成了**异步删除快照日志**，几乎不占业务时间；只有万一失败时，才通过日志进行**反向补偿回滚**。
    

## 📌 三大角色：各司其职的“铁三角”

整个分布式事务的运转，全靠这三个组件打配合：

- **TM (事务管理器)：** 老大。贴了 `@GlobalTransactional` 的服务，负责**开启**和**结束**（提交/回滚）全局事务。
    
- **RM (资源管理器)：** 小弟。各分支微服务，负责**执行**本地事务、记录 `undo_log` 快照，并听从 TC 的最终调配。
    
- **TC (事务协调器)：** 裁判。独立部署的 Seata 核心服务器集群，负责**维护**全局事务状态，下发最终的 Commit 或 Rollback 指令。
    

## 📌 核心黑科技（以 AT 模式为例）

### 1. 镜像机制（Before / After Image）

RM 会在底层利用数据库代理（Data Source Proxy）拦截你的业务 SQL：

- 执行前：查出旧数据，生成 **Before Image**。
    
- 执行后：查出新数据，生成 **After Image**。
    
- 组合打包，塞进本地库的 `undo_log` 表。
    

### 2. 全局锁（Global Lock）—— 解决脏写死穴

为了防止在“一阶段各自提交、二阶段尚未决议”的真空期里，数据被其他不归 Seata 管的线程偷偷修改（脏写），Seata 引入了**全局锁**：

- 只要加入了全局事务，RM 在一阶段本地提交前，必须先去 TC 注册并拿到该行数据的**全局锁**（基于表名 + 主键）。
    
- 拿不到全局锁就必须等待，从而在不依赖数据库原生重锁的情况下，实现了全局的**写隔离**。
    

## 📌 四种武器（模式）大局观

不同的业务场景，Seata 允许你切换不同的武器来应对：

``` plain
Seata 模式选择 ──┬──> 关系型数据库 + 无代码侵入  ──────> AT 模式 (首选)
                  ├──> 高并发 + 金融级 + 性能控   ──────> TCC 模式 (手动 Try-Confirm-Cancel)
                  ├──> 流程超长 + 对接第三方老系统  ────> Saga 模式 (正向业务 + 反向补偿)
                  └──> 强一致性洁癖 + 不计并发代价  ───> XA 模式 (原生2PC)
```

### 💡 终极一句话小结：

> **Seata 的原理就是：一阶段借着本地事务直接提交，通过数据库代理自动留存“反悔快照（undo_log）”并加全局锁；二阶段若成功则异步删快照，若失败则利用快照自动生成反向 SQL 把数据“逆向倒带”回去。**
