
Spring Cloud Alibaba Sentinel 可以说是微服务架构中非常尽职尽责的“保安”，专门用来保护系统不被突发的流量冲垮，或者被下游故障的依赖拖死。

相比于老牌的 Hystrix，Sentinel 更加轻量级，控制台功能更丰富，且对业务代码的侵入性极低。它的核心就是两个词：**限流（Flow Control）** 和**熔断（Circuit Breaking）**。

# Sentinel 能干嘛

简单来说，Sentinel 就是微服务在生产环境下的“防弹衣”**和**“保险丝”。

在实际开发和线上运维中，它主要用来解决以下几个让人睡不着觉的痛点场景：

### 1. 搞定“秒杀/大促”的突发洪峰（限流与排队）

- **痛点：** 比如搞一个抢购活动，平时系统每秒只有 50 个人访问，活动开始瞬间飙到 5000。如果不加控制，数据库连接池瞬间被死锁，整个系统直接卡死，所有人都刷不出页面。
    
- **Sentinel 能干嘛：** 你可以设置一个死命令：这个接口每秒最多只能进 1000 个请求。超过的怎么办？可以用**排队等待**模式，让请求像沙漏一样匀速通过，把大流量“整形”成平稳的小流量。多余的请求在队列里排队，超时就告诉用户“前方拥挤，请稍后再试”，从而死死保住后端的数据库。
    
### 2. 防止“第三方接口宕机”拖垮你（熔断降级）

- **痛点：** 你的订单系统需要调用第三方的“短信验证码”或者“物流查询”接口。今天这个第三方服务刚好大面积延迟，平时 10 毫秒就返回的接口，现在要 5 秒才响应。结果就是你系统里的核心线程全卡在等待第三方返回上，导致你自己的核心业务（比如浏览商品、下单）也跟着瘫痪。
    
- **Sentinel 能干嘛：** 它会盯着这个第三方接口。一旦发现最近 10 秒内，这个接口的报错率超过了 50%，或者平均响应时间长得离谱，Sentinel 就会“啪”地一下拉闸（熔断）。在接下来的冷却期内，只要有用户调用这个接口，Sentinel 直接在前端拦截并返回一个默认的友好提示（比如“短信发送失败，请稍后”），**绝不让别人的垃圾服务拖垮你自己的服务器**。

### 3. 精准防刷、防黄牛（热点参数限流）

- **痛点：** 正常的流量系统能抗住，但突然有黄牛用脚本疯狂刷某一个特定的商品 ID，或者某个恶意 IP 在疯狂爬你的数据。如果直接关闭整个接口，正常用户就没办法用了。
    
- **Sentinel 能干嘛：** 它能识别请求里的参数。你可以配置：对 `/goods/detail` 接口整体放行，但如果**同一个商品 ID** 每秒被访问超过 100 次，或者**同一个用户 ID** 每秒请求超过 5 次，那就只把这个特定的人或商品给限流了。这就叫精准打击，不伤及无辜。
    
### 4. 拯救刚重启的服务器（冷启动/预热）

- **痛点：** 刚发布了新版本，服务器刚重启。这时候 JVM 还没热身（各种类还没加载，缓存也是空的）。如果这时候直接把上万的流量切过来，服务器会因为瞬间高 CPU 负载而直接瘫痪。
    
- **Sentinel 能干嘛：** 开启 **Warm Up（预热）** 模式。刚重启的前 3 分钟，Sentinel 只允许 20% 的流量进来，让系统暖暖身子、加载下缓存；随着时间推移，3 分钟后才把流量闸门完全打开放行 100%。
	
### 5. 最后的“自适应”兜底保命（系统自适应保护）

- **痛点：** 有时候你根本不知道流量从哪来的，可能是遭遇了 DDOS 攻击，也可能是某个业务写了死循环，反正服务器的 CPU 已经飙到了 90%，眼看就要死机死锁了。
    
- **Sentinel 能干嘛：** 它可以作为最后的防线，直接监控服务器的硬指标（如 CPU 使用率、系统整体 Load、响应时间）。一旦发现 **CPU 超过 80%**，它会开启自适应保护，自动拒绝掉后续进来的非核心请求，把 CPU 压下去，**优先保住服务器的命**，避免整机宕机。
    

> **一句话总结：**
> 没有 Sentinel，系统遇到大流量或依赖故障就是“全凭天意，说挂就挂”；有了 Sentinel，你能做到“局部故障、整体可用，流量再大、系统不垮”。

## 服务雪崩

**服务雪崩（Service Avalanche）**，听名字就能感受到那种画面感：山顶上仅仅掉落了一小块积雪（一个微服务出故障），最后却演变成把山脚下整座村庄都吞没的惊天灾难（整个微服务集群瘫痪）。

在微服务架构中，它是最致命的“连锁反应”或“多米诺骨牌效应”。

### 一、 雪崩是怎么发生的？（四步演变史）

假设你的电商系统有三个服务：**用户服务 $\rightarrow$ 订单服务 $\rightarrow$ 支付服务**。平时大家都开开心心地搬砖，直到某一天：

1. **起因：下游变慢**
    
    支付服务因为数据库死锁或者第三方通道卡顿，原本 10 毫秒就能处理完的请求，现在需要 **5 秒**才能响应。
    
2. **蔓延：上游线程被死死卡住**
    
    订单服务需要等支付服务返回结果。因为支付变慢了，调用订单服务的核心线程不能释放，只能在这儿傻傻地等。
    
3. **爆发：线程池耗尽**
    
    用户还在疯狂地点击下单。订单服务的线程池（比如一共 200 个线程）在几秒钟内就被新来的、等待的请求**全部占满**。此时，订单服务再也腾不出任何一个线程去处理别的请求了。
    
4. **雪崩：全线崩溃**
    
    因为订单服务瘫痪了，依赖订单服务的“用户服务”也开始被拖垮；同时，由于订单服务无法响应，前端网关（Gateway）的排队请求也爆满。最终，**整个电商系统大面积报错 502/504，彻底瘫痪**。
    

> **核心本质：**
> 
> 局部资源的故障，由于**请求的不断积压**，导致上游服务的**线程资源被耗尽**，从而将故障级联放大到整个链路。

### 二、 导致雪崩的三大“罪魁祸首”

1. **基础设施故障：** 网线被挖断、机房断电、高并发下数据库连接池爆满、慢 SQL。
    
2. **业务流量激增：** 突发的抢购大促、或者遭受恶意的 DDOS 攻击。
    
3. **代码及配置问题：** 代码里写了死循环、网络调用没写超时时间（Timeout）导致无限期等待。
    

### 三、 怎么拦截雪崩？（微服务续命四大招）

面对雪崩，架构师们总结出了四条黄金法则，这也是 Sentinel 等框架的核心功能：

- **超时机制（Timeout）：**
    
    绝对不能让线程无限期等待！调用下游时，设置“最多等 1 秒，不回我就挂断”。虽然这次请求失败了，但**及时释放了线程**，保住了自己。
    
- **舱壁隔离/线程池隔离（Bulkhead）：**
    
    借鉴造船的原理，把船舱隔成一个一个独立的吃水舱，一个舱漏水不会沉船。在微服务里，给每个下游依赖分配**独立的、固定大小的线程池**。比如给支付服务分配 10 个线程，它就算卡死，也最多占满这 10 个线程，订单服务剩下的 190 个线程还能正常去处理退款、查询等业务。
    
- **熔断降级（Circuit Breaking）：**
    
    当发现支付服务已经连续失败或者超时好几次了，直接“拉闸”。后续请求不再真正去调用支付服务，而是直接在订单服务内部“走快捷通道”返回失败提示。给下游充足的时间去“回血”和修复。
    
- **流量控制（Rate Limiting）：**
    
    在最前端把好关，只放系统扛得住的流量进来，多余的直接拒之门外，不给系统添乱。

## 服务降级

如果说“服务雪崩”是微服务面临的灭顶之灾，那么服务降级（Service Degradation）**就是应对这场灾难时，明智的**“丢车保帅”策略。

简单来说，服务降级就是：**当系统压力过大、或者下游服务出现故障时，为了保证核心业务（如支付、下单）的绝对可用，主动关闭、简化或者替代一些非核心业务（如评论、推荐、历史账单）的机制。**

### 一、 一个生活中的经典比喻

想象一下每年的“双11”或者“年货节”大促，电商平台的流量是平时的几十倍：

- **不降级（硬抗）：** 所有人不仅能下单，还能看高清评价视频、写长篇评论、查询五年前的订单。结果导致服务器 cup 爆满，最后连“最核心的付款”功能也卡死报错。
    
- **服务降级（聪明人的做法）：** 平台为了保住“付款”这个赚钱的核心功能，主动做出了牺牲——暂时关闭“发表评论”功能，把高清视频评价降级为只显示文字，历史订单只能查最近3个月的。等大促高峰期过去，再把这些功能恢复。
    

### 二、 常见的服务降级策略

在微服务架构中，研发和运维人员通常会使用以下几种手段来进行降级：

1. **返回默认托底数据（Fallback）：**
    
    当调用“商品推荐服务”超时或报错时，不再继续死等，而是直接返回一组硬编码的默认商品（比如销量最高的爆款），保证页面不出现空白或报错。
    
2. **功能降级（关闭非核心业务）：**
    
    在高流量期间，直接在网关或配置中心关闭掉一些消耗资源、但不影响主流程的功能。比如：暂时关闭日志审计、关闭个性化推荐、关闭复杂的报表查询。
    
3. **缓存降级：**
    
    平时用户的个人主页数据需要实时查询数据库。当数据库压力过大时，切换为直接读取 Redis 缓存中的旧数据，哪怕数据有几分钟的延迟，也比系统卡死强。
    
4. **异步降级（延迟写入）：**
    
    平时用户下单后，系统要同步去加积分、发短信、扣库存。降级模式下，把加积分、发短信等操作直接扔进消息队列（MQ）里异步处理，让接口快速响应，减少用户等待时间。

## 服务熔断

**服务熔断（Circuit Breaking）** 是微服务架构中实现“自我保护”的核心机制。如果把“服务雪崩”比作传染病，那么“服务熔断”就是**最果断的隔离措施**。

它的灵感完全来源于我们现实生活中的**电闸保险丝**（或空气开关）：当家里的电器越来越多，电流过大或者发生短路时，保险丝会瞬间熔断拉闸，从而保护整个房子的电路不被烧毁。在微服务中，它保护的是你的服务器核心资源。

### 一、 熔断器的核心：三大状态机

服务熔断不是简单地把接口关掉就完事了，它有一套非常严密的**自动恢复状态机**。理解了这三种状态，你就理解了熔断的精髓：

#### 1. 🟢 Closed（关闭状态）—— 正常放行

- **表现：** 此时系统一切正常，熔断器处于关闭状态。所有的请求都会被放行，正常调用下游服务。
    
- **背后的动作：** Sentinel 等框架会在后台默默统计这段时间内的请求数据（比如成功率、报错率、响应时间）。
    

#### 2. 🔴 Open（打开状态）—— 跳闸隔离

- **触发条件：** 当后台统计的数据达到了你设定的“危险阈值”（比如最近 10 秒内有 50% 的请求都在报错）。
    
- **表现：** 熔断器果断跳闸！**所有后续请求不再调用下游服务**，直接在本地被拦截，瞬间返回报错或者执行预设的降级逻辑（Fallback）。
    
- **目的：** 1. 彻底断开与故障下游的连接，**保住上游自己的线程资源**。
    
    2. 给下游故障服务腾出“喘息和修复”的时间，避免无休止的流量继续轰炸它。
    

#### 3. 🟡 Half-Open（半开状态）—— 试探恢复

- **触发条件：** 熔断器打开后，会进入一个临时的“冷却期”（比如设定为 5 秒）。冷却时间一到，熔断器会自动变成**半开状态**。
    
- **表现：** 此时，系统会小心翼翼地只放行少量请求（通常是 1 个）去真正调用下游服务。
    
- **结果走向：**
    
    - **如果这个试探请求成功了：** 说明下游服务已经康复！熔断器会自动闭合，回到 **Closed** 状态，全线恢复通车。
        
    - **如果这个试探请求又失败了（或超时）：** 说明下游还在“ICU”里没抢救过来。熔断器会立马重新变成 **Open** 状态，并**重置冷却时间**（再等 5 秒）。
        

### 二、 怎么判断下游“不行了”？（三大熔断策略）

在 Sentinel 中，你不能凭感觉拉闸，需要配置具体的规则。主要有以下三种判断标准：

1. **慢调用比例 (Slow Call Ratio)：**
    
    - **逻辑：** 你规定一个请求如果超过 500ms 就算“慢调用”。如果最近一分钟内，系统有超过 60% 的请求都是慢调用，直接熔断。
        
    - **场景：** 适合用来对付下游服务虽然没挂、但是响应极慢（比如数据库死锁、TCP 拥堵）的情况。
        
2. **异常比例 (Exception Ratio)：**
    
    - **逻辑：** 最近 100 个请求里，如果调用下游接口报错（抛异常）的比例超过了 50%，直接熔断。
        
    - **场景：** 适合网络抖动、对方代码有 Bug 导致大面积报错的场景。
        
3. **异常数 (Exception Count)：**
    
    - **逻辑：** 不管比例，只要最近 1 分钟内，调用下游接口抛出异常的次数达到了 20 次，直接熔断。
        

## 服务限流

如果说服务熔断是发现下游生病后的“紧急隔离”，那么**服务限流（Rate Limiting / Flow Control）**就是大门口手握计数器的**“硬核保安”**。

它的核心思想非常简单：**“任你流量暴雨倾盆，我自定额限量放行。”** 无论外部有多少请求涌进来，系统只接受自己能力范围内的流量，超出的部分直接拒之门外或排队等待，以此来死守服务器的底线。

### 一、 为什么要限流？（防患于未然）

在微服务中，限流主要用来对付以下三种让人头疼的场景：

1. **突发业务洪峰：** 比如中午 12 点准时开抢的连花清瘟、或者突发的明星热点新闻，流量在 1 秒内翻了几百倍。
    
2. **无节制的恶意流量：** 羊毛党、抢票爬虫、或者恶意的 DDoS 攻击，用高频请求疯狂消耗你的服务器资源。
    
3. **保护脆弱的下游：** 你的核心网关抗并发能力极强（比如 10 万 QPS），但后端的底层数据库（MySQL）最多只能抗 2000 QPS。如果不限流，网关会直接把数据库冲垮。
    

### 二、 核心四大限流算法（面试必问）

怎么精准地控制流量？业界和 Sentinel 内部主要有四种经典的算法。

#### 1. 固定窗口算法（Fixed Window）

- **原理：** 把时间划分为固定的大小（比如 1 秒为一个窗口）。每个窗口内有一个计数器，来一个请求计数器加 1，达到阈值（如 100）后，后续请求全部拒绝。下个秒数到来时，计数器清零。
    
- **缺点（致命硬伤）：** **临界点暴击**。假设在第 0.9 秒来了 100 个请求，第 1.1 秒又来了 100 个请求。虽然两个单独的 1 秒窗口都没超标，但在第 0.9 秒到 1.1 秒这短短 **0.2 秒内，系统其实硬抗了 200 个请求**，可能会瞬间崩盘。
    

#### 2. 滑动窗口算法（Sliding Window）

- **原理：** 它是固定窗口的升级版。把一个大时间窗口（1秒）拆分成很多个小格子（比如 10 个 100ms 的格子）。随着时间流逝，窗口像传送带一样向右滑动。计数器统计的是**当前滑动窗口内所有小格子的请求总和**。
    
- **优点：** 完美解决了固定窗口的“临界点暴击”问题，时间颗粒度越细，限流越平滑。**Sentinel 的默认限流统计就是基于滑动窗口（LeapArray）实现的。**
    

#### 3. 漏桶算法（Leaky Bucket）

- **原理：** 把系统想象成一个**底部有小孔的桶**。不管外面的水（流量）进来的多猛、多大，桶底的小孔都以**绝对固定的速度**往外漏水（处理请求）。如果水进来的太快，桶满了，多余的水就直接漫出来（拒绝请求）。
    
- **优点：** 绝对平滑，能完美强行整形流量。
    
- **缺点：** 缺乏弹性。如果系统当前很闲，突然来了几个并发请求，漏桶依然会让他们排队匀速通过，无法应对**突发激增流量（Burst Traffic）**。
    

#### 4. 令牌桶算法（Token Bucket）

- **原理：** 系统的“无形之手”以固定的速度往桶里放“令牌”（Token）。桶是有容量上限的，满了就不放了。每个请求进来，必须先去桶里**拿走一个令牌**才能被处理。如果桶里没令牌了，说明流量超标，请求被拒绝或排队。
    
- **优点：** 既能限制平均流量，又能**完美应对突发流量**。如果系统闲了一段时间，桶里攒满了令牌（比如 100 个），突然来了 100 个并发请求，它们可以瞬间拿走所有令牌同时通过，非常人性化。
    


## 服务隔离

如果说限流是微服务的“大门保安”，熔断是“紧急电闸”，那么**服务隔离（Service Isolation）** 就是系统内部的 **“防火墙”** 和 **“安全隔舱”**。

它的核心思想是：**把原本混在一起的系统资源（如线程、内存、CPU、数据库连接）按业务或者渠道“切分”成一个个独立的区域。哪怕其中某一个区域由于故障彻底瘫痪，火势也绝对蔓延不到其他区域。**

### 一、 经典源头：造船厂的“舱壁隔离”（Bulkhead Pattern）

服务隔离的概念完全来自于传统的造船业。像泰坦尼克号这样的现代远洋轮船，船舱内部并不是一个空旷的大肚子，而是被厚厚的钢板隔成了十几个**彼此独立、完全密封的“吃水舱”**。

- **不隔离：** 如果船体破了一个洞，水会流满整个船舱，船直接沉没（对应微服务中的服务雪崩）。
    
- **有隔离：** 如果一两个舱进水了，只需要把这两个舱的闸门一关，水就在这里被锁死了。虽然这两个舱废了，但其他十几个舱还是安全的，船依然能平稳航行。

### 二、 微服务中的两大“硬核”隔离手段

在开发中（尤其是结合 Sentinel 或 Hystrix 等框架时），我们最常挂在嘴边的服务隔离，指的是**线程级/调用级的资源隔离**。主要分为两种派系：

#### 1. 线程池隔离（Thread Pool Isolation）

- **做法：** 系统不共用一个大线程池，而是为每一个高风险的下游依赖**单独开辟一个固定大小的线程池**。
    
- **例子：** 订单接口有自己独立的 20 个线程，积分接口有自己独立的 10 个线程。
    
- **效果：** 哪怕积分接口因为下游故障卡死，疯狂涌入的请求最多把积分接口的 10 个线程占满。订单接口的 20 个线程完全不受影响，用户依然可以正常下单。
    
- **大牌代言人：** Netflix Hystrix 默认主推这种模式。

#### 2. 信号量隔离 / 并发线程数隔离（Semaphore Isolation）

- **做法：** 不创建新的线程池，大家还是在一个大池子里跑。但是，系统会给每个接口设一个“计数器”（类似于原子计数器或信号量），限制**同时正在处理该接口的线程总数**。
    
- **例子：** 规定“查询历史账单”接口同时最多只能有 5 个线程在执行。
    
- **效果：** 当第 6 个线程想去调用账单接口时，发现计数器满了，连进去的资格都没有，直接被拒绝返回。这就保证了账单接口无论多慢，**最多只会拖住大池子里的 5 个线程**，剩下的几百个线程还能去干别的事。
    
- **大牌代言人：** **Spring Cloud Alibaba Sentinel 默认推荐这种模式。**

### 三、 终极PK：线程池隔离 vs 信号量隔离

既然都是隔离，为什么 Sentinel 偏爱信号量，而老牌的 Hystrix 喜欢线程池？它们各有千秋：

| **维度**   | **线程池隔离 (Thread Pool)**                           | **信号量隔离 (Semaphore)**                 |
| -------- | ------------------------------------------------- | ------------------------------------- |
| **实现原理** | 独立开辟新的子线程池，通过线程池满额来拒绝                             | 基于计数器/信号量，限制并发的线程数量                   |
| **资源开销** | **较高**。由于排队、上下文切换（Context Switch）和额外线程创建，有一定性能损耗。 | **极低**。只是做简单的数字加减和判断，不创建新线程，无额外损耗。    |
| **异步支持** | 支持异步、超时控制（如果子线程卡死，主线程可以主动掐断）。                     | 不支持。调用者和被调用者在同一个主线程中，无法真正实现异步掐断。      |
| **适用场景** | 适合调用**第三方不稳定接口、网络极其恶劣**、或者需要强超时控制的场景。             | 适合**内部微服务之间高频调用**、QPS 极大、需要追求极致性能的场景。 |

### 四、 架构眼光：更高维度的“服务隔离”

除了上面针对代码和线程的隔离，在真正的微服务架构设计中，隔离其实无处不在：

1. **物理/部署隔离（Cluster Isolation）：**
    
    - 比如把“前台用户抢购系统”和“后台运营报表系统”部署在完全不同的服务器集群上。防止运营人员导出一个 10G 的大报表把服务器 CPU 跑满，导致前台用户连网页都打不开。
        
2. **数据隔离（Data Isolation）：**
    
    - 坚决执行“微服务核心原则”：一个服务一个数据库。订单服务只能读写订单库，商品服务只能读写商品库。如果商品库的硬盘满了或者锁表了，订单库依然能正常运转。
        
3. **租户隔离（Tenant Isolation）：**
    
    - 在 SaaS 系统中，大客户（VIP 客户）和普通小客户的请求和数据走不同的物理服务器和数据库，确保小客户高并发引起的故障不会波及大客户。


## 服务超时

如果说服务隔离是把系统拆分成独立的“吃水舱”，熔断是“紧急跳闸”，那么**服务超时（Service Timeout）** 就是整个微服务架构中**最基本、最基础的“心理防线”**。

它的核心思想非常朴素：**“我允许你变慢，但我绝不允许你让我无限期地等下去。”** ---

### 一、 为什么要设置服务超时？

在分布式系统中，网络是不可靠的。下游服务可能因为网络丢包、垃圾回收（Full GC）卡死、或者数据库死锁而无法响应。

如果没有设置超时时间（或者把超时设成了默认的无限大），就会发生以下惨剧：

1. **线程被死死拖住：** 上游服务发起网络调用后，主线程会一直处于阻塞（Block）状态，苦苦等待下游返回。
    
2. **资源迅速耗尽：** 新的请求源源不断地进来，每一个请求都占满一个线程并卡在那里。几秒钟内，系统的核心线程池就会被全部榨干。
    
3. **雪崩爆发：** 最终，由于这一个接口没有设置超时，导致整个系统的线程池崩溃，引发我们前面聊过的**服务雪崩**。

> **一句话总结：**
> 超时机制是**防止线程被无限期挂起**的最后底线。它通过主动放弃一次失败的调用，换取了整个系统线程资源的释放。

### 二、 必须要分清的两大超时指标

在日常开发和排查日志时，你经常会看到两个不同的超时参数，它们卡在不同的网络阶段：

``` plain
  [客户端] ──────(1. 建立TCP连接)──────> [服务端] ──────(2. 业务处理并返回)──────> [客户端]
             └─ Connect Timeout ─┘                 └──── Read Timeout ────┘
```

#### 1. 连接超时（Connect Timeout）

- **定义：** 指客户端与服务端**建立网络连接（TCP 三次握手）的最大等待时间**。
    
- **发生场景：** 通常发生在对方网络断开、服务器宕机、机房断电、或者由于防火墙拦截导致网络根本不通。
    
- **特点：** 这个超时通常设置得比较短（比如 1s - 3s），因为网络如果通的话，三次握手是非常快的。

#### 2. 读取/响应超时（Read Timeout / Socket Timeout）

- **定义：** 指连接已经成功建立了，客户端把请求发过去了，**等待服务端处理完业务并返回数据的最大时间**。
    
- **发生场景：** 对方服务器网络没问题，但是它的代码写了死循环、或者正在执行一条耗时 10 秒的奇葩慢 SQL。
    
- **特点：** 这个超时需要根据业务场景来评估。如果是查缓存，可能设为 500ms；如果是导出一个复杂的 Excel 报表，可能需要设为 30s 甚至更长。

### 三、 微服务中如何配置超时？

在 Spring Cloud 生态中，我们很少直接操作底层的 HttpClient，通常是在组件层进行配置。最常见的配置地方有三个：

#### 1. OpenFeign（最常用）

Feign 作为声明式的客户端，统一管理了微服务之间的调用超时。

``` yaml
feign:
  client:
    config:
      default: # default代表全局配置，也可以写具体的服务名进行精细化配置
        connectTimeout: 2000 # 2秒连接超时
        readTimeout: 5000    # 5秒读取超时
```
#### 2. Spring Cloud Gateway（网关层）

在最前端的网关层，也要防止路由转发时被下游微服务卡死。
``` yaml
spring:
  cloud:
    gateway:
      httpclient:
        connect-timeout: 1000 # 网关连接下游 1 秒超时
        response-timeout: 3000 # 网关等待下游响应 3 秒超时
```

### 四、 超时、熔断、隔离是怎么组合打连招的？

在真正的生产环境中，这几个高可用工具从来不是单兵作战，而是**环环相扣**的：

1. **超时（第一步）：** 订单服务调用积分服务，设置了 `readTimeout: 2s`。
    
2. **触发异常（第二步）：** 某一天积分服务卡顿，订单服务等了 2 秒没收到结果，瞬间抛出 `TimeoutException`。因为及时断开，订单服务的**线程被立刻释放**，没有被卡死。
    
3. **熔断器计数（第三步）：** 旁边的 **Sentinel** 默默记录下：“哦？刚刚发生了一次超时异常。”
    
4. **拉闸断开（第四步）：** 如果接下来的 10 个请求全部因为超时而报错，Sentinel 达到阈值，果断触发**熔断（Open状态）**。
    
5. **降级托底（第五步）：** 在接下来的 5 秒熔断期内，用户再来调用，Sentinel 连 2 秒都不让等了，直接 0 毫秒拦截，调用 **Fallback 降级方法**返回：“系统繁忙，积分稍后为您补上”。


# 流控规则

**流控规则（Flow Control Rules）**，就是我们前面提到的限流保安手里拿的那张“放行标准登记表”。

在 Sentinel 中，一条流控规则决定了什么样的流量能进系统，什么样的流量该被拦截。无论是通过 Sentinel 控制台（Dashboard）配置，还是写在代码/配置文件里，一条完整的流控规则都由以下几个**核心参数**组成。

### 一、 流控规则的核心参数大拆解

在 Sentinel 的底源码中，流控规则对应的是 `FlowRule` 这个类。它包含以下 5 个最重要的必填项：

| **参数名称** | **对应代码字段**        | **含义与可选值**                                                                                      |
| -------- | ----------------- | ----------------------------------------------------------------------------------------------- |
| **资源名**  | `resource`        | 唯一名称，可以是 API 路径（如 `/order/create`），也可以是 `@SentinelResource` 注解里定义的名称。                           |
| **针对来源** | `limitApp`        | Sentinel 可以识别调用方。默认是 `default`（代表不区分来源，谁来都限）。也可以写具体的微服务名字（如 `shop-frontend`），实现**只针对某个前端服务限流**。 |
| **阈值类型** | `grade`           | 两个选项：1. **QPS**（每秒请求数）2. **并发线程数**（同时有多少个线程在处理该接口）                                              |
| **单机阈值** | `count`           | 具体的数字。如果是 QPS 设为 10，代表每秒最多放行 10 个请求。                                                            |
| **流控模式** | `strategy`        | 流量怎么判断。可选值：**直接**（默认）、**关联**、**链路**。                                                            |
| **流控效果** | `controlBehavior` | 满了之后怎么处理。可选值：**快速失败**（默认）、**Warm Up（预热）**、**排队等待**。                                             |

### 二、 重点理解：三大“流控模式”（面试必考）

很多人配流控规则时，最容易对“流控模式”产生困惑。Sentinel 提供了三种非常有灵活性高级玩法：

#### 1. 直接模式（默认）

- **规则：** 当**当前资源**自己达到了限流阈值时，就限流自己。
    
- **场景：** 比如 `/getOrder` 接口每秒超过 100 次请求，直接让 `/getOrder` 报限流错误。最简单直观。
    
#### 2. 关联模式（大局观，为核心业务让路）

- **规则：** 当**与我关联的另一个资源**达到阈值时，限流**我自己**。
    
- **场景：** 比如系统里有两个接口，一个是 `/order/pay`（支付，核心赚钱业务），一个是 `/order/list`（查询历史订单，读操作）。它们共用同一个数据库。
    
- **配置方式：** 资源名写 `/order/list`，流控模式选关联，关联资源写 `/order/pay`。
    
- **效果：** 当“支付”接口流量暴增、数据库压力极大时，“历史订单查询”接口会自动开始自我限流、弹窗报错。**牺牲非核心业务，把宝贵的数据库性能全部留给“支付”服务。**
    
#### 3. 链路模式（精准打击调用源头）

- **规则：** 只记录从某个特定入口（Link）点进来的流量，如果超标了才限流。
    
- **场景：** 假设你的 Service 层有一个核心业务方法 `createOrder()`。这个方法既被“Web 前端 App 控制层”调用，也被“后台定时任务/批处理控制层”调用。
    
- **效果：** 你可以配置链路模式，指定入口为“Web 前端”。这样，哪怕定时任务在那边疯狂调用 `createOrder()` 把它跑出几万 QPS，前端用户的正常下单也不会受到任何影响。它只防范从前端入口进来的爆破流量。
    
### 三、 规则在代码中长啥样？

虽然我们一般在控制台配置，但了解它的 JSON 结构有助于我们做**规则持久化**。在后台，一条流控规则其实就是一个 JSON 对象：

``` json
[
  {
    "resource": "/order/create",    // 资源名
    "limitApp": "default",          // 来源应用
    "grade": 1,                     // 阈值类型：1 代表 QPS，0 代表并发线程数
    "count": 20,                    // 单机阈值：20 QPS
    "strategy": 0,                  // 流控模式：0 代表直接，1 代表关联，2 代表链路
    "controlBehavior": 0,           // 流控效果：0 代表快速失败，1 代表 Warm Up，2 代表排队等待
    "clusterMode": false            // 是否是集群限流（默认 false 单机限流）
  }
]
```

### 四、 避坑指南：流控规则重启丢失怎么办？

初学者使用 Sentinel 会遇到一个非常蛋疼的问题：**在控制台里辛辛苦苦配了几十条流控规则，结果微服务一重启，规则全没了！**

这是因为 Sentinel Dashboard 默认是把规则保存在微服务节点的**内存**里的。

- **生产环境正确姿势：** 必须做**规则持久化**。
    
- **主流方案：** 将流控规则配置在 **Nacos**、Apollo 或 Consul 等配置中心里。微服务启动时去 Nacos 读取规则，并在 Nacos 中修改时能实时同步到微服务和 Sentinel 控制台。这样不管怎么重启，规则永远都在。

# 流控效果

流控效果（Control Behavior）说白了就是：**当流量已经超过了你设定的阈值时，Sentinel 到底该用什么“姿势”来处置这些多出来的请求？**

在 Sentinel 中，流控规则里一共有三种流控效果。它们各司其职，分别应对不同的业务场景。

为了让你最直观地感受它们的区别，我们先来看一张这三种流控效果下**流量波形图**的对比：

### 1. 快速失败（Direct Reject）—— 默认，简单粗暴

- **怎么做：** 这是默认选项。一旦每秒的请求数（QPS）超过了设定阈值，Sentinel 会**立刻、毫不留情地拒绝**多余的请求，直接抛出 `FlowException`。
    
- **流量波形：** 像一刀切的平顶山。到了线，多出来的流量瞬间蒸发。
    
- **适用场景：** **绝大多数普通接口**。系统对响应时间（RT）要求极高，宁可让多余的用户看到“系统繁忙”，也绝不能在系统内部积压请求、拖慢整个调用链路。
    

### 2. Warm Up（预热/冷启动）—— 慢热型，保护刚重启的服务

- **原理底层：** 基于**令牌桶算法**的变种。
    
- **怎么做：** 系统刚启动或长期处于低负载时，如果突然涌入大流量，由于缓存未加载、JVM 还没热身，很容易直接把服务器冲死。
    
    - 当开启 Warm Up 并设置预热时间（比如 10 秒，最大阈值 30）后，Sentinel 会让通过的阈值从一个很低的“冷启动值”（默认是 **最大阈值 / 3**，即 10 QPS）开始。
        
    - 在接下来的 10 秒内，允许通过的 QPS 才会**缓慢、平滑地爬升**到最终的 30 QPS。
        
- **流量波形：** 像一个平缓上坡的阶梯，最后达到平顶。
    
- **适用场景：** **服务刚刚发布重启、或者每天清晨定时任务刚唤醒系统的场景**。给系统充足的时间去预热线程池、加载缓存、进行类加载。
    
### 3. 排队等待（Rate Limiter）—— 匀速排队，削峰填谷

- **原理底层：** 基于**漏桶算法**。
    
- **怎么做：** 严格控制请求通过的时间间隔。比如你设置了 QPS 阈值为 10，那么 Sentinel 就会强制规定：**每 100 毫秒只能通过 1 个请求**。
    
    - 如果在 100ms 内同时来了 5 个请求，第 1 个立刻通过；
        
    - 剩下的 4 个不会被直接拒绝，而是进入一个**虚拟队列里排队等待**；
        
    - 它们会以每 100ms 出来一个的速度匀速被执行。你还可以设置一个“最大等待超时时间”，如果在队列里排队的时间超过了这个值，才会被拒绝。
        
- **流量波形：** 把原本尖锐、突发的“惊涛骇浪”洪峰流量，强行梳理成一条**绝对平稳、匀速的直线**。
    
- **适用场景：** **秒杀系统、或者处理来自消息队列（MQ）的突发消费**。这种场景下，流量可能只在 1 秒内飙升，之后就没了。与其直接拒绝用户，不如让大家排个队，只要服务器能匀速把它们处理完即可。
    
### 三种流控效果大比拼（一览表）

| **流控效果**    | **核心核心算法** | **超过阈值怎么办** | **核心关注点**         | **经典业务场景**     |
| ----------- | ---------- | ----------- | ----------------- | -------------- |
| **快速失败**    | 滑动窗口       | 直接抛异常拒绝     | **响应速度**（立刻给结果）   | 普通商品查询、普通 API  |
| **Warm Up** | 令牌桶（冷系数）   | 低阈值逐步开放     | **系统保护**（防止冷启动瘫痪） | 刚发布/重启的服务、大促开幕 |
| **排队等待**    | 漏桶算法       | 排队匀速通过/超时拒绝 | **平滑流量**（削峰填谷）    | 秒杀抢购、MQ 消息拉取消费 |

# 熔断规则

**熔断规则（Circuit Breaking Rules）**，就是给系统安全员（Sentinel）下达的“跳闸判决书”。

它详细规定了在什么样的情况下，系统应该认定下游服务“生病了”，并果断拉闸断开连接。在 Sentinel 中，熔断规则由 `DegradeRule` 类来定义。

我们可以先通过这张图看看在 Sentinel 控制台配置熔断规则时的完整面板和它的核心参数：

### 一、 熔断规则的核心参数面板

无论你使用哪种熔断策略，都必须配置以下几个极其关键的“硬性指标”：

|**参数名称**|**对应代码字段**|**含义与避坑指南**|
|---|---|---|
|**资源名**|`resource`|需要保护的接口或资源名称（如 `/order/create`）。|
|**熔断策略**|`grade`|决定怎么判断接口不行了。可选：**慢调用比例**、**异常比例**、**异常数**。|
|**熔断时长**|`timeWindow`|**单位：秒**。跳闸后持续的“冷却时间”。在这段时间内，所有请求直接走降级（Fallback）。时间一到，进入半开状态（Half-Open）去试探。|
|**最小请求数**|`minRequestAmount`|**大坑！** 触发熔断的先决条件。在一秒内（或统计周期内），该接口的请求数必须**大于或等于**这个值，熔断规则才会生效。如果设为 5，即使前 4 个请求全报错，也不会熔断。|
|**统计时长**|`statIntervalMs`|**单位：毫秒**。统计上述指标的时间窗口大小。默认是 1000 毫秒（1秒）。|

### 二、 深度拆解：三大熔断策略（怎么才算不正常？）

Sentinel 提供了三种判定下游服务故障的“解题思路”：

#### 1. 慢调用比例 (Slow Call Ratio) —— 最推荐，专治各种“卡顿和变慢”

- **怎么判断：** 你需要设置两个值：**最大并发响应时间（RT）** 和 **比例阈值**。
    
- **规则配置举例：** * RT 设为 `500ms`
    
    - 比例阈值设为 `60%`
        
    - 最小请求数设为 `10`
        
- **大白话解释：** 如果在 1 秒内，进来了至少 10 个请求，其中有 **60% 以上**的请求处理时间都超过了 500 毫秒，Sentinel 就会大喊一声：“太慢了，拉闸！”，接下来的熔断时长内，谁也别想调这个接口。
    

#### 2. 异常比例 (Exception Ratio) —— 专治“大面积网络抖动或Bug”

- **怎么判断：** 统计规定时间内，下游接口**抛出异常的请求数占总请求数的比例**。
    
- **规则配置举例：** * 比例阈值设为 `50%`（取值范围 0.0 - 1.0）
    
    - 最小请求数设为 `5`
        
- **大白话解释：** 1 秒内至少有 5 个请求进来，结果有一半以上（$\ge 50\%$）都抛出异常报错了（比如报 500、NullPointerException 等），直接拉闸隔离。
    

#### 3. 异常数 (Exception Count) —— 简单粗暴的“容忍底线”

- **怎么判断：** 不管请求总数有多少，只看**抛出异常的绝对次数**。
    
- **规则配置举例：** * 异常数设为 `5`
    
    - 最小请求数设为 `5`
        
- **大白话解释：** 只要在 1 秒内，这个接口疯狂报错达到了 5 次，不管总共来了 5 个还是 500 个请求，Sentinel 都会立刻跳闸。
    

### 三、 熔断规则在代码/JSON 中长啥样？

如果你需要把熔断规则持久化到 Nacos 中，它在后台的 JSON 格式长这样：

``` json
[
  {
    "resource": "/user/info",       // 资源名
    "grade": 0,                     // 熔断策略：0代表慢调用比例，1代表异常比例，2代表异常数
    "count": 500,                   // 阈值：如果是慢调用代表500ms；如果是异常比例代表0.5；如果是异常数代表5
    "timeWindow": 10,               // 熔断降级的时间窗口：10秒冷却时间
    "minRequestAmount": 5,          // 触发熔断的最小请求数
    "statIntervalMs": 1000          // 统计时长：1000ms（1秒）
  }
]
```

### 💡 新手调试最容易犯的低级错误：

> “为什么我配了异常比例为 100% 熔断，我连续手动点浏览器刷新报错了 3 次，系统却依然没有熔断，还能继续访问？”

**原因就在于：你没有满足“最小请求数（minRequestAmount）”。** Sentinel 的默认最小请求数是 5。你只点了 3 次，哪怕 3 次全错，由于样本太少，Sentinel 会认为这可能只是偶发性的网络抖动，为了不误伤，它绝对不会盲目跳闸。调试时记得把“最小请求数”改小（比如改为 1 或 2），就能立刻看到熔断效果了。


# @SentinelResource注解

如果你把 Sentinel 当作系统的“安保系统”，那么 **`@SentinelResource` 注解** 就是你埋在代码里的“监控摄像头兼防盗门”。

通过这个注解，你可以把任何一个普通的 Java 方法定义为一个 Sentinel **“资源”**，并直接在代码里指定“如果被限流了该怎么办”或者“如果报错了该怎么兜底”。

### 一、 核心属性大拆解

`@SentinelResource` 里面有很多参数，但最核心、最常用的是以下几个：

| **属性名称**                 | **类型**                         | **含义与作用**                                                      |
| ------------------------ | ------------------------------ | -------------------------------------------------------------- |
| **`value`**              | `String`                       | **必填**。资源名称。Sentinel 控制台就是通过这个名字来找到该方法并配置流控/熔断规则的。             |
| **`blockHandler`**       | `String`                       | **限流/熔断时的处理方法名**。当流量超标或触发熔断（抛出 `BlockException`）时，会自动调用这个方法。   |
| **`blockHandlerClass`**  | `Class<?>`                     | 如果不想把处理方法和业务代码写在同一个类里，可以用这个属性指定一个外部类。                          |
| **`fallback`**           | `String`                       | **业务异常时的兜底方法名**。当方法内部抛出业务异常（如 `NullPointerException`）时，会调用此方法。 |
| **`fallbackClass`**      | `Class<?>`                     | 同上，指定外部的异常兜底类。                                                 |
| **`exceptionsToIgnore`** | `Class<? extends Throwable>[]` | 忽略哪些异常。被忽略的异常不会触发 `fallback`，而是直接往外抛。                          |

### 二、 灵魂拷问：`blockHandler` vs `fallback` 有啥区别？

这是面试中最喜欢问的细节。它们两个虽然都是“兜底”，但是**分工完全不同**。我们通过下面这张执行流程图来直观感受一下：

- **`blockHandler`：专治“违反规则”**
    
    - 它**只负责**处理 Sentinel 的规则限制。比如：QPS 满了被**限流**了、或者报错太多被**熔断**了。
        
    - 换句话说，只有系统抛出 `BlockException` 及其子类时，才会走这里。
        
- **`fallback`：专治“代码报错”**
    
    - 它负责处理**业务层面的 runtime 异常**。比如：你代码里不小心写出了空指针异常（`NullPointerException`）、或者是数据库连接超时。
        
    - 只要是代码自己运行出错，且没有被 Sentinel 熔断，就会走这里。
        

> **如果两个都配了，谁先触发？**
> 
> 如果一秒内来了 100 个请求导致接口**被限流**了，走 `blockHandler`；
> 
> 如果只来了一个请求，但因为数据库没数据抛了**空指针异常**，走 `fallback`；
> 
> 如果因为频繁抛空指针异常导致接口**被熔断**了，那么后续的请求再进来，会走 `blockHandler`（因为熔断本质上是违反了 Sentinel 规则）。

### 三、 实战：标准代码怎么写？

写 `@SentinelResource` 的时候，初学者最容易犯的错误就是**兜底方法的参数写错了**，导致 Sentinel 根本找不到方法而直接报错。

下面是一个教科书级别的标准写法：

``` java
@RestController
public class OrderController {

    @GetMapping("/getUserOrder")
    @SentinelResource(
        value = "getUserOrderResource", 
        blockHandler = "handleBlock", // 针对限流/熔断
        fallback = "handleFallback"   // 针对业务报错
    )
    public String getUserOrder(@RequestParam("userId") String userId) {
        if ("error".equals(userId)) {
            throw new RuntimeException("故意制造的业务异常");
        }
        return "用户 " + userId + " 的订单数据";
    }

    /**
     * 1. blockHandler 方法要求：
     * - 返回值类型、参数列表必须和原方法完全一致
     * - 必须在参数列表的最后，额外加上一个 BlockException 参数
     */
    public String handleBlock(String userId, BlockException ex) {
        return "【限流降级】当前访问人数过多，服务器被限流了，请稍后再试！错误原因：" + ex.getClass().getSimpleName();
    }

    /**
     * 2. fallback 方法要求：
     * - 返回值类型、参数列表必须和原方法完全一致
     * - 可以在参数列表的最后，额外加上一个 Throwable 参数（可选，用来拿报错堆栈）
     */
    public String handleFallback(String userId, Throwable throwable) {
        return "【业务异常】对不起，查询用户 " + userId + " 失败，代码抛出异常：" + throwable.getMessage();
    }
}
```

### 四、 避坑指南与最佳实践

1. **兜底方法必须是 `public` 的**：Sentinel 底层是基于 Spring AOP（动态代理）实现的，如果你的 `blockHandler` 或 `fallback` 方法写成了 `private`，Sentinel 将无法调用，直接把异常抛给前端。
    
2. **避免代码膨胀**：如果每个业务方法都把 `blockHandler` 写在同一个 Controller 类里，这个类很快就会变得臃肿不堪。建议使用 `blockHandlerClass` 和 `fallbackClass` 属性，把所有的兜底逻辑统一抽离到外部的 `OrderHandlerConfig` 类中（注意：抽离到外部类后，方法必须声明为 `static` 静态方法）。
    
3. **原方法参数要对齐**：兜底方法的参数不仅要包含原方法的全部参数，**顺序也必须一模一样**，最后才能加 `BlockException` 或 `Throwable`。


# 热点规则

如果说普通的流控规则是系统大门口的“全面戒严”（只要人多就全拦下），那么热点规则（Param Flow Rules，热点参数限流）就是手握名单的“精准打击”——**它专门用来揪出流量里那几个最火爆的“超级大咖”（热点参数），并对它们单独进行限制，同时不影响其他正常流量。**

在微博吃瓜（某个明星爆料）、电商秒杀（某个爆款手机抢购）或者防刷防爬（某个恶意 IP 疯狂刷接口）的场景下，热点规则是绝对的主角。

### 一、 为什么要用热点规则？（解决什么痛点）

假设你的电商系统有一个商品详情接口 `/goods/detail?id=XXX`，平时全局限流设的是 1000 QPS。

- **遭遇痛点：** 忽然，平台上架了一款特价 iPhone。瞬间，`id=1001`（iPhone）的请求飙到了 900 QPS。这时候，买衣服的、买零食的其他用户发现系统开始频繁报错、限流了。**因为热点商品把全局的坑位全占满了，导致无辜的普通业务跟着陪葬。**
    
- **热点规则的解法：** 针对 `/goods/detail` 接口，普通商品每秒允许访问 5 次；但针对 `id=1001` 这个爆款，单独网开一面，允许它每秒访问 500 次，超过 500 再限流它自己。这样既保护了核心爆款，又顺畅放行了普通商品。
    

### 二、 热点规则的核心参数面板

在 Sentinel 控制台配置热点规则时，你会看到以下几个稍微有点怪异的参数：

| **参数名称**       | **对应代码字段**       | **含义与大白话解释**                                              |
| -------------- | ---------------- | --------------------------------------------------------- |
| **资源名**        | `resource`       | 对应的 `@SentinelResource` 资源名称。                             |
| **参数索引 (必填)**  | `paramIdx`       | **大坑！** 代表你要针对该方法的**第几个参数**进行限流（从 0 开始算）。如果是第一个参数，就填 `0`。 |
| **单机阈值**       | `count`          | **默认的**参数限流阈值。比如设为 5，代表**任何一个**商品 ID 每秒最多只能访问 5 次。        |
| **统计窗口时长**     | `durationInSec`  | 统计的时间范围，默认是 1 秒。                                          |
| **参数例外项 (高级)** | `paramAllowList` | **核心灵魂。** 在这里添加“特例”。比如指定当参数值等于 `1001` 时，阈值提升到 `500`。      |

### 三、 实战：代码与控制台怎么配合？

**特别注意：** 热点规则**必须**配合 `@SentinelResource` 注解使用，普通的 URL 盲配是无法自动识别方法内部参数的。

#### 1. 编写业务代码
``` java
@RestController
public class ProductController {

    @GetMapping("/product")
    @SentinelResource(value = "hotspotResource", blockHandler = "handleHotspot")
    public String getProduct(
            @RequestParam("id") String id,       // 参数索引 0
            @RequestParam("type") String type    // 参数索引 1
    ) {
        return "查询商品详情，ID: " + id + ", 类型: " + type;
    }

    // 热点限流后触发的兜底方法
    public String handleHotspot(String id, String type, BlockException ex) {
        return "【热点限流】当前商品 (ID: " + id + ") 访问太频繁，请稍后再试！";
    }
}
```

#### 2. 在控制台“精细化打击”配置

如果你想实现：“默认每个商品每秒只能查 2 次，但 ID 为 `1001` 的爆款商品每秒能查 100 次”，你要怎么配？

1. **资源名：** 填 `hotspotResource`
    
2. **参数索引：** 填 `0`（因为我们要对 `id` 限流，它是第一个参数）
    
3. **单机阈值：** 填 `2`（这是对普通商品的兜底限制）
    
4. **展开【高级选项】 $\rightarrow$ 参数例外项：**
    
    - **参数类型：** 选择 `java.lang.String`
        
    - **参数值：** 填 `1001`
        
    - **限流阈值：** 填 `100`
        

配置完成后，神奇的效果就出现了：你疯狂刷新 `id=1002`，点到第 3 下立刻被限流；但你疯狂刷新 `id=1001`，可以畅通无阻地点到 100 下才会被限流！

### 四、 避坑指南：热点规则的隐藏限制

1. **只支持基本类型和 String：** 热点规则在底层是用传进来的参数作为 Map 的 Key 的。因此，作为限流对象的参数，必须是基本数据类型（如 int、long）、包装类或者 String。如果你传了一个复杂的自定义对象（如 `UserOrderDto`），Sentinel 默认是没法直接对其进行热点限流的（除非你对该对象重写了完美的 `equals` 和 `hashCode` 方法）。
    
2. **参数例外项的类型必须匹配：** 在控制台配置例外项时，一定要选对参数类型（比如 `int` 和 `java.lang.String` 是严格区分的），否则例外项规则会直接失效。
    
3. **内存热点防爆：** Sentinel 为了防止你传入的参数太多（比如几百万个不同的用户 IP）导致服务器内存爆掉，底层使用了 **LRU（最近最少使用）淘汰策略**。默认只会缓存前 4000 个不同的参数值，超过的会被淘汰。



# 授权规则

如果说流控规则是数人头的“计数保安”，热点规则是专抓大咖的“粉丝控评师”，那么**授权规则（Authority Rules，黑白名单控制）**就是系统大门口手持名单、拦截特定人员的**“门禁保安”**。

它的核心功能非常纯粹：**根据请求的“来源应用（Origin）”，决定是放行（白名单）还是直接拒绝（黑名单）。**

### 一、 授权规则的两大铁面策略

在 Sentinel 控制台配置授权规则时，你主要需要做两道选择题：

#### 1. 白名单 (Whitelist)

- **大白话：** “除了名单上写了名字的人，其他人一律不准进！”
    
- **核心场景：** **安全架构强固**。在微服务中，我们通常希望所有的外部请求都必须先经过 **网关（Gateway）**，由网关统一做鉴权、路由，然后再转发给内部的“订单服务”或“用户服务”。
    
- **做法：** 我们可以把订单服务设为“白名单模式”，只允许来源为 `gateway` 的请求通过。这样，如果有黑客绕过网关，尝试直接通过 IP 访问内部的订单服务，会瞬间被授权规则无情拦截。
    

#### 2. 黑名单 (Blacklist)

- **大白话：** “除了名单上这几个捣蛋鬼，其他人都可以进！”
    
- **核心场景：** **紧急风控与防刷**。比如在日常运维中，你突然发现有一个第三方的合作伙伴系统、或者某个特定渠道的应用（如 `bad-crawler-app`）正在疯狂调用你的核心接口，严重拖慢了整个系统，但还没达到全局流控的整体阈值。
    
- **做法：** 此时无需修改代码重启，直接在 Sentinel 控制台把 `bad-crawler-app` 丢进黑名单，该应用的所有请求就会在最外层直接被秒杀，而其他正常的网页端、手机端请求完全不受影响。
    
### 二、 初学者最大的坑：Sentinel 怎么知道请求从哪来？

很多同学在控制台配好了授权规则，比如流控应用填了 `gateway`，选了白名单，结果发现**根本不生效**，或者**所有请求都被拦截了**。

> **核心原因：** Sentinel 默认是一个“睁眼瞎”，它自己是拿不到请求来源的名字的！你必须在代码里实现一条 **“接头暗号”解析器**，告诉 Sentinel 怎么去认出调用方。

#### 💡 核心解法：实现 `RequestOriginParser` 接口

你需要在你的微服务代码里，硬编码或者通过拦截器定义一套规则（比如规定所有微服务互相调用时，必须在 HTTP Header 里带上 `S-Origin` 参数）：

``` java
@Component
public class MyRequestOriginParser implements RequestOriginParser {
    
    @Override
    public String parseOrigin(HttpServletRequest request) {
        // 1. 从请求头中获取名为 "S-Origin" 的参数
        String origin = request.getHeader("S-Origin");
        
        // 2. 如果没带，为了防止恶意绕过，可以返回一个默认值（比如 "unknown"）
        if (StringUtil.isEmpty(origin)) {
            return "unknown";
        }
        
        return origin;
    }
}
```

有了这段代码后，当网关转发请求给订单服务时，网关只需要在 Header 里加上 `S-Origin: gateway`。Sentinel 就能准确识别出：“哦！老熟人 `gateway` 来了，放行！”

### 三、 授权规则的参数面板与 JSON 结构

在后台持久化时，授权规则对应的是 `AuthorityRule` 类，其 JSON 结构非常简单：
``` json
[
  {
    "resource": "/order/create",   // 资源名
    "limitApp": "gateway,app-A",   // 流控应用：多个应用用逗号隔开
    "strategy": 0                  // 授权类型：0 代表白名单，1 代表黑名单
  }
]
```

### 四、 总结：高可用规则全家桶

走到这一步，Sentinel 的核心规则我们就全部集齐了。面对复杂的线上流量，你可以这样组合御敌：

1. 先用 **授权规则** 辨别身份（是不是网关来的？是不是黑名单用户？）。
    
2. 再用 **流控规则** 总体把关（每秒 QPS 是不是超标了？）。
    
3. 如果是特定爆款商品，用 **热点规则** 局部精细化制裁。
    
4. 下游服务如果真的卡顿了，用 **熔断规则** 果断拉闸，最后统一用 `@SentinelResource` 的 **`blockHandler`** 弹出一句温暖的“服务器开小差了，请稍后再试”。

### 五、 添加`S-Origin: gateway`

别慌，这其实就是微服务之间在“对暗号”。

简单来说，浏览器（用户）访问你的系统时，是不可能自觉在请求头里加上 `S-Origin: gateway` 的。这个动作必须由网关（Gateway）在转发请求的一瞬间，**偷偷盖上一个“戳”**，再传给后面的订单服务。

这个“戳”要加在你的**网关（Gateway）项目**里，而不是订单服务里。添加的方法主要有以下两种，你可以二选一：

### 方法一：直接在网关的配置文件（YAML）里加（最简单）

如果你的网关路由是通过 `application.yml` 配置的，Spring Cloud Gateway 自带了一个叫 `AddRequestHeader` 的过滤器，可以直接在转发时注入请求头。

找到你**网关项目**的 `application.yml`，在订单服务的路由配置下加上 `- AddRequestHeader=S-Origin, gateway`：

``` yaml
spring:
  cloud:
    gateway:
      routes:
        # 订单服务的路由
        - id: order_service_route
          uri: lb://order-service        # 转发到订单服务
          predicates:
            - Path=/order/**             # 只要是 /order/ 开头的请求都走这里
          filters:
            # 👇 就是在这里添加！网关转发时会自动在请求头里加上 S-Origin: gateway
            - AddRequestHeader=S-Origin, gateway
```

**优点：** 纯配置，不需要写一行代码。 **缺点：** 如果你有 20 个微服务，你得在每个服务的路由下面都写一遍，有点麻烦。

### 方法二：在网关项目里写一个全局过滤器（全自动，推荐）

如果你嫌在 YAML 里每个路由都配一遍太麻烦，希望**只要经过网关的请求，不论转发给谁，都统一加上这个暗号**，那就在**网关项目**里写一个 Java 类：
``` java
package com.example.gateway.filter; // 注意：这是在网关项目里

import org.springframework.cloud.gateway.filter.GatewayFilterChain;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.core.Ordered;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.stereotype.Component;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

@Component
public class SentinelOriginFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 悄悄在原本的请求头里，强行塞入 S-Origin = gateway
        ServerHttpRequest request = exchange.getRequest().mutate()
                .header("S-Origin", "gateway")
                .build();
        
        // 2. 把带有新请求头的请求继续往下传
        ServerWebExchange newExchange = exchange.mutate().request(request).build();
        return chain.filter(newExchange);
    }

    @Override
    public int getOrder() {
        // 过滤器执行顺序，数字越小越先执行
        return 0; 
    }
}
```

### 💡 整体的运作流程（连起来看）

配完之后，整个闭环就跑通了：

``` plain
[浏览器] 
   │  发出普通请求 (没有特殊 Header)
   ▼
[网关 Gateway] 
   │  1. 拦截到请求
   │  2. 根据上面的配置/代码，在 Header 里硬塞入：S-Origin = gateway
   ▼
[订单服务 Order-Service]
   │  1. 进门时，Sentinel 的 RequestOriginParser 开始解析
   │  2. 发现 Header 里有 "S-Origin"，提取出值是 "gateway"
   │  3. 授权规则校验：检查白名单里有没有 "gateway"
   │  4. 校验通过！放行进入具体的业务代码
```

这样一来，如果有黑客知道了你订单服务的真实 IP，绕过网关直接用浏览器或 Postman 访问 `http://订单服务IP:端口/order/create`。因为他的请求没经过网关，里面绝对没有 `S-Origin: gateway` 这个暗号，订单服务的 Sentinel 就会直接无情地把他拦截在门外！



# 规则持久化

在默认情况下，Sentinel 的所有规则（流控、熔断、热点等）都是保存在微服务节点的**内存**中的。这意味着**只要微服务一重启，你在控制台配置的规则就会瞬间烟消云散**。

在生产环境中，这显然是无法接受的。因此，我们必须做**规则持久化（Rule Persistence）**。

### 一、 持久化的两种模式：Pull vs Push

Sentinel 支持两种持久化模式，但在生产环境中，我们**几乎 100% 选择 Push 模式**。

#### 1. Pull（拉模式）—— 了解即可

- **原理：** 微服务本地挂载一个 JSON 文件。微服务定期去读取这个文件，或者在控制台修改规则时，控制台通知微服务，微服务再把规则写进本地文件。
    
- **缺点：** 无法及时同步。如果是集群部署，多台机器之间难以同步，且高并发下频繁读写本地文件性能较差。
    

#### 2. Push（推模式）—— 生产环境标准

- **原理：** 把规则统一存放在配置中心（如 **Nacos**）。
    
- **工作流：** 
		
	1. 运维/开发在 Nacos 中修改限流规则。
	    
    2. Nacos 作为一个“大喇叭”，实时把最新的规则**推给**对应的微服务。
	    
    3. 微服务拿到规则，更新内存，立刻生效。
    
- **优点：** 规则持久化存储，支持集群，时效性极高（秒级同步）。
    

### 二、 落地实战：基于 Nacos 实现 Push 模式持久化

下面以 **Spring Boot + Nacos** 为例，带你一步步实现流控规则的持久化。

#### 第一步：引入依赖

在你的微服务（比如订单服务）的 `pom.xml` 中，除了引入 Sentinel 核心依赖外，必须额外引入 **Nacos 数据源** 的动态扩展包：

``` xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>

<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-datasource-nacos</artifactId>
</dependency>
```

#### 第二步：在 application.yml 中配置数据源

你需要告诉微服务：“去哪里找 Nacos？去哪个 Data ID 下面拉取流控规则？”

``` yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848 # Nacos 注册中心地址
    sentinel:
      transport:
        dashboard: 127.0.0.1:8080 # Sentinel 控制台地址
      # 👇 配置 Sentinel 的数据源
      datasource:
        flow-rules: # 策略名（自定义，代表流控规则）
          nacos:
            server-addr: 127.0.0.1:8848 # Nacos 配置中心地址
            data-id: ${spring.application.name}-flow-rules # 规范命名：order-service-flow-rules
            group-id: SENTINEL_GROUP # 组名
            data-type: json          # 数据格式
            rule-type: flow          # 规则类型：flow代表流控，degrade代表熔断，param-flow代表热点
```

#### 第三步：在 Nacos 中创建配置文件

启动 Nacos 控制台，进入配置列表，新建一个配置：

- **Data ID:** `order-service-flow-rules` （必须和上面 yml 里的对应）
    
- **Group:** `SENTINEL_GROUP`
    
- **配置格式:** `JSON`
    
- **配置内容:** （填入我们之前学过的流控规则 JSON 数组）
    

``` json
[
  {
    "resource": "/order/create",
    "limitApp": "default",
    "grade": 1,
    "count": 5,
    "strategy": 0,
    "controlBehavior": 0,
    "clusterMode": false
  }
]
```

点击发布。此时启动你的微服务，访问 Sentinel 控制台，你会神奇地发现：**不需要你在 Sentinel 里配任何东西，`/order/create` 的流控规则就已经自动躺在里面了！** 即使重启微服务，规则依然在。

### 三、 生产环境不得不面对的“痛点”（进阶思考）

当你高高兴兴把上面这套方案用到生产环境时，很快就会发现一个很别扭的操作问题：

> **“我在 Nacos 里改规则，Sentinel 控制台能同步。但是如果我在 Sentinel 控制台页面上点了修改，Nacos 并不会同步，而且微服务一重启，在控制台改的部分又丢了！”**

这是因为官方开源的 Sentinel Dashboard 默认依然是往微服务的**内存**里推数据的（走的是 API 直连）。

#### 业界通用的完美解决方案：改装 Sentinel Dashboard

为了让运营和开发体验最好，通常需要**修改 Sentinel Dashboard 的源码**，将其底层改为：**无论是在控制台增删改查，都直接去读写 Nacos。**

1. **改造前：** 
		
	-  控制台 $\rightarrow$ 微服务内存（重启丢）
	    
    - Nacos $\rightarrow$ 微服务内存（重启不丢）
        
2. **改造后（双向绑定）：** 
	-  控制台 $\leftrightarrow$ **Nacos** $\rightarrow$ 微服务内存
    

> 改造 Sentinel Dashboard 源码并不复杂，官方其实在源码的 `test` 目录下已经写好了对接 Nacos 的类（`FlowControllerV2` 等），我们只需要把它挪到主源码里，修改一下前端页面的打包配置即可。




# OpenFeign和Sentinel集成实现fallback服务降级

把 **OpenFeign** 和 **Sentinel** 结合起来，是微服务架构中处理“服务间调用安全”的经典大招。

简单来说，OpenFeign 负责向远端微服务发送请求，而 Sentinel 负责在暗中观察。一旦远端服务挂了、超时了、或者触发了限流熔断，Sentinel 会立刻接管，让 OpenFeign 别再死等了，而是直接调用本地写好的**备用方案（Fallback）**。

下面是完整的集成与落地的四个标准步骤：

### 第一步：引入依赖

在需要调用别家服务的微服务（比如 `order-service` 订单服务）的 `pom.xml` 中，确保同时引入了 OpenFeign 和 Sentinel 的 Starter：

``` xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

### 第二步：修改配置文件（开启集成开关）

**核心关键点！** 默认情况下，OpenFeign 并不会理会 Sentinel。你必须在 `application.yml` 中明确告诉 OpenFeign：“请把你的生命安全交给 Sentinel 保护。”

``` yaml
feign:
  sentinel:
    enabled: true # 👈 必须开启！让 Sentinel 能够拦截 Feign 的远程调用
```

### 第三步：定义兜底方案（推荐使用 FallbackFactory）

实现降级有两种方式：`fallback` 和 `fallbackFactory`。

在生产环境中，**强烈推荐使用 `fallbackFactory`**，因为它可以抓住**导致这次失败的真实异常原因（Throwable）**，方便我们在日志里排查到底是超时、熔断还是 500 错误。

编写一个降级工厂类（记得贴上 `@Component` 让 Spring 管理）：

``` java
package com.example.order.feign.factory;

import com.example.order.feign.UserFeignClient;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.cloud.openfeign.FallbackFactory;
import org.springframework.stereotype.Component;

@Component
public class UserFeignClientFallbackFactory implements FallbackFactory<UserFeignClient> {
    
    private static final Logger log = LoggerFactory.getLogger(UserFeignClientFallbackFactory.class);

    @Override
    public UserFeignClient create(Throwable cause) {
        // 在这里可以捕获并打印真实的错误堆栈，方便排查
        log.error("Feign调用用户服务失败，触发降级保护！错误原因: ", cause);

        // 返回一个匿名内部类，实现原本的 Feign 接口，编写具体的错后处理
        return new UserFeignClient() {
            @Override
            public String getUserInfo(Long userId) {
                // 具体的降级托底逻辑
                return "【降级服务】无法获取用户 " + userId + " 的信息，该服务已被熔断。原因：" + cause.getMessage();
            }
        };
    }
}
```

### 第四步：在 Feign 接口上绑定降级工厂

回到你原本的 Feign 客户端接口上，在 `@FeignClient` 注解里通过 `fallbackFactory` 属性指定刚才写好的工厂类：

``` java
package com.example.order.feign;

import com.example.order.feign.factory.UserFeignClientFallbackFactory;
import org.springframework.cloud.openfeign.FeignClient;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;

// name 为服务名，fallbackFactory 指定刚才编写的降级工厂类
@FeignClient(name = "user-service", fallbackFactory = UserFeignClientFallbackFactory.class)
public interface UserFeignClient {

    @GetMapping("/user/info/{id}")
    String getUserInfo(@PathVariable("id") Long id);
}
```

### 💡 两种降级方式的对比

| **接入方式**                          | **编写方式**                      | **优缺点**                                                                                 |
| --------------------------------- | ----------------------------- | --------------------------------------------------------------------------------------- |
| **`fallback = Xxx.class`**        | 编写一个类直接实现 Feign 接口。           | **缺点：** 无法获取导致失败的 Exception 信息。代码出了 Bug 还是超时触发的降级，你完全不知道，两眼一抹黑。                         |
| **`fallbackFactory = Xxx.class`** | 编写一个类实现 `FallbackFactory<T>`。 | **优点：** 可以完美拿到 `Throwable cause`，不仅能打印精细的错误日志，还能针对不同的异常（如 `TimeoutException`）做差异化的业务返回。 |
### 🚀 避坑小贴士

1. **主启动类别漏了注解**：在微服务的启动类（如 `OrderApplication`）上，一定要加上 `@EnableFeignClients` 注解，否则 Spring 根本不会扫描到你的 FeignClient 接口。
    
2. **Sentinel 控制台的资源名长相**：当 OpenFeign 和 Sentinel 集成成功后，Sentinel 会自动将 Feign 的调用包装成资源。它默认的命名规则非常规范，长这样：
    
    `GET:http://user-service/user/info/{id}`
    
    你在控制台配置熔断规则或流控规则时，**资源名必须填这个完整的 Feign 路径**，降级方案才会生效。

# GateWay和Sentinel集成实现服务限流

把 **Spring Cloud Gateway** 和 **Sentinel** 结合起来，相当于给你的微服务集群在大门口架设了一挺“加特林机枪”。

在前面的调用中，我们是在各个微服务（如订单服务、用户服务）内部做限流。但最理想的状态是：**如果流量超标或者遭到恶意刷量，直接在最前端的网关层就把请求拍死，根本不让这些垃圾流量进入内网，从而绝对保护内部微服务的 CPU 和内存。**

由于 Spring Cloud Gateway 底层基于 **WebFlux（响应式编程）** 架构，它的集成方式和普通的 Controller 略有不同。以下是保姆级的四大集成步骤：

### 第一步：引入依赖（注意别引错包）

在你的 **网关（Gateway）项目** 的 `pom.xml` 中，除了网关核心依赖外，需要引入专门为 Gateway 定制的 Sentinel 适配包：

``` xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-alibaba-sentinel-gateway</artifactId>
</dependency>

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

### 第二步：修改配置文件（application.yml）

网关项目需要连接到 Sentinel Dashboard，并且配置好路由。

``` yaml
server:
  port: 10010 # 网关端口
spring:
  application:
    name: cloud-gateway
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
    sentinel:
      transport:
        dashboard: 127.0.0.1:8080 # Sentinel控制台地址
    gateway:
      routes:
        # 1. 订单服务路由
        - id: order_route
          uri: lb://order-service
          predicates:
            - Path=/order/**
        # 2. 用户服务路由
        - id: user_route
          uri: lb://user-service
          predicates:
            - Path=/user/**
```

### 第三步：Sentinel 控制台的“网关专属玩法”

启动网关并访问一次接口后，打开 Sentinel 控制台，你会发现左侧菜单栏和普通的微服务不一样，它多出了两个极其牛逼的专属菜单：**【请求链路端点】** 和 **【网关流控规则】**。

在网关限流中，你有两种玩法：

#### 玩法 A：针对“Route ID（路由条目）”直接限流

- **操作：** 鼠标点击【网关流控规则】 $\rightarrow$ 新增网关流控规则。
    
- **API类型：** 选择 `Route ID`。
    
- **API名称：** 填入你在 yml 里写的路由 id（比如 `order_route`）。
    
- **效果：** 只要每秒通过网关转发到订单服务的总流量超标，直接拦截。
    

#### 玩法 B：针对“API 分组”自定义精细化限流

- **痛点：** 哪怕同属于 `order_route`，我希望对 `/order/create`（下单，核心）和 `/order/query`（查订单，非核心）进行不同的限流，该怎么办？
    
- **操作：** 
	
	1. 先去 **【API 分组管理】** 里新建一个分组，取名 `order_create_group`，匹配路径设为 `/order/create`。
    
    2. 再回到 **【网关流控规则】** 里新增规则，API 类型选择 `API 分组`，名称选刚才建好的 `order_create_group`。
    
- **效果：** 网关会精准识别 URL，只给下单接口卡死阈值，不误伤查询接口。
    

### 第四步：核心避坑：定制网关的限流返回 JSON

默认情况下，网关如果被 Sentinel 限流了，前端会收到一个极其简陋的白屏报错：`Blocked by Sentinel (flow limiting)`。

因为网关底层是响应式 WebFlux，**普通的 `BlockExceptionHandler` 在这里会直接失效**。我们必须编写一个配置类，使用 `GatewayCallbackManager` 注册一个特殊的 `BlockRequestHandler`：

``` java
package com.example.gateway.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.http.HttpStatus;
import org.springframework.http.MediaType;
import org.springframework.web.reactive.function.server.ServerResponse;
import com.alibaba.csp.sentinel.adapter.gateway.sc.callback.GatewayCallbackManager;
import com.alibaba.csp.sentinel.adapter.gateway.sc.callback.BlockRequestHandler;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

import javax.annotation.PostConstruct;
import java.util.HashMap;
import java.util.Map;

@Configuration
public class GatewaySentinelConfig {

    @PostConstruct
    public void initBlockHandlers() {
        // 👈 注意：这里使用的是 GatewayCallbackManager
        GatewayCallbackManager.setBlockHandler(new BlockRequestHandler() {
            @Override
            public Mono<ServerResponse> handleRequest(ServerWebExchange exchange, Throwable t) {
                // 1. 组装你希望返回给前端的统一标准 JSON 格式
                Map<String, Object> errorMap = new HashMap<>();
                errorMap.put("code", 429);
                errorMap.put("msg", "【网关提示】前面的大侠请留步，当前访问人数过多，服务器挤爆啦！");
                errorMap.put("status", "error");

                // 2. 采用 WebFlux 的响应式编程方式友好返回
                return ServerResponse.status(HttpStatus.TOO_MANY_REQUESTS)
                        .contentType(MediaType.APPLICATION_JSON)
                        .bodyValue(errorMap);
            }
        });
    }
}
```

### 🛠️ 为什么网关限流更推荐配“热点/参数限流”？

在网关层，我们经常需要防止**防刷、恶意爬虫**。Sentinel 的网关规则支持很多高级的**参数提取项**：

- **Client IP：** 只要同一个 IP 每秒访问超过 10 次就直接封锁该 IP。
    
- **Request Header：** 识别带有特定 Token 的请求。
    
- **URL 参数：** 比如针对 `/order/pay?userId=XXX` 里的 `userId` 进行热点限流，防止单用户写脚本爆破接口。
    

这些都可以在 Sentinel 控制台的网关流控规则的“高级选项”里直接勾选，不需要在代码里写任何痕迹。
