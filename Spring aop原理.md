
搞懂了三级缓存，再来看 Spring AOP（面向切面编程）的原理就非常顺理成章了。因为 AOP 的本质，就是在 Bean 的生命周期中，**通过“动态代理”把你的业务代码默默地包装起来**。

用一句话总结 Spring AOP 的底层原理：**它是基于 Java 的动态代理机制，在 Bean 的初始化阶段，为目标对象生成一个“代理对象”，从而在不修改原代码的前提下，为方法偷偷“织入”增强逻辑（如日志、事务、安全检查等）。**

我们可以把它的核心拆解为两部分：**设计模式（怎么代理）** 和 **Spring 织入时机（何时代理）**。

## 一、 AOP 的核心底层：两种动态代理

Spring AOP 就像是一个“经纪人”。你写的业务类是“明星”，真正干活的是明星，但外界所有找明星的电话，都会先打到“经纪人（代理对象）”这里。经纪人可以在唱歌（业务代码）前后，帮你安排收钱、签合同（切面逻辑）。

Java 中有两种实现经纪人的方式，Spring 会根据你的类自动切换：

### 1. JDK 动态代理（基于接口）

- **原理**：利用反射机制，在内存中动态创建一个**实现了相同接口**的代理类。
    
- **适用场景**：你的业务类**实现了接口**（比如 `UserServiceImpl` 实现了 `UserService`）。
    
- **特点**：生成的代理对象和你的原始对象是“同门师兄弟”关系，它们都实现了同一个接口。
    

### 2. CGLIB 动态代理（基于继承）

- **原理**：底层利用 ASM 框架，在内存中动态构建一个**子类**。
    
- **适用场景**：你的业务类**没有实现任何接口**（就是一个普通的 `OrderService` 类）。
    
- **特点**：生成的代理对象是原始对象的“儿子”。因为它用的是继承，所以**如果你的类或者方法被 `final` 修饰，CGLIB 就无法代理**（因为 final 不能被继承/重写）。
    

> 💡 **Spring Boot 的小秘密**：
> 
> 在传统的 Spring 中，有接口用 JDK，没接口用 CGLIB。但到了 **Spring Boot 2.x 和 3.x 之后，默认全都直接使用 CGLIB 代理**，哪怕你有接口。这是为了避免由于接口和实现类类型转换引发的奇怪报错。

## 二、 运行时的“织入”原理（怎么把代码塞进去的）

当你调用一个被 AOP 代理的方法时（比如一个加了 `@Transactional` 事务的方法），调用流程其实变成了这样：

```
客户端调用 -> 代理对象 (Proxy) 
                ↓
           执行 前置通知 (Before Advice) -> 比如开启事务
                ↓
           执行 你的真实业务方法 (Target Method)
                ↓
           执行 后置/返回通知 (After/AfterReturning Advice) -> 比如提交事务
```

这一串流程在 Spring 内部被封装成了 **拦截器链（Interceptor Chain）**。Spring 会把通知（Advice）排序，做成一条链条，用责任链模式依次执行。

## 三、 Spring 到底在什么时候帮我们创建代理的？

这就能和我们之前聊的 **Bean 生命周期** 串联起来了！

在 Bean 生命周期的 **初始化（Initialization）** 阶段，最后一步是执行 `BeanPostProcessor`（Bean 后置处理器）的后置方法。

Spring AOP 的核心实现类叫 `AbstractAutoProxyCreator`（它就是一种 `BeanPostProcessor`）。

1. 你的 Bean 刚实例化完，也填好了属性。
    
2. 走到了生命周期的最后关头，`AbstractAutoProxyCreator` 探出头来扫描一遍。
    
3. 它拿你的类去匹配所有的 `@Aspect` 切面配置（看你的切入点 `Pointcut` 有没有命中这个类）。
    
4. **如果没有命中**：直接放行，返回原对象。
    
5. **如果命中了**：它就在内存里偷偷用 JDK 或 CGLIB 创建一个代理对象，**把原对象掉包**。最后塞进 Spring 一级缓存的，其实是这个代理对象。
    

_(注：如果在三级缓存那里发生了循环依赖，这个创建代理的动作就会被提前到注入阶段执行。)_

## 核心概念速查表

面试或看源码时，经常被这几个洋文绕晕，其实它们很简单：

- **Aspect（切面）**：一个大容器，里面装着“在哪里切（Pointcut）”和“切了干嘛（Advice）”。
    
- **Pointcut（切入点）**：具体的规则。比如“所有 service 包下的方法”。
    
- **Advice（通知/增强）**：具体要加的代码。比如“在方法执行前打印日志”。
    
- **Target（目标对象）**：你写的那个原原本本的业务类（明星本人）。
    
- **Proxy（代理对象）**：Spring 掉包后的那个对象（经纪人）。
    

简单来说，Spring AOP 就是**在 Bean 快要出厂（初始化）的时候，通过后置处理器发现它需要被增强，然后用 JDK 或 CGLIB 做了个代理对象把原对象给替换了。** 这样解释，有把 AOP 的底子摸透的感觉了吗？
