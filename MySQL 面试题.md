## 第 1 题：MySQL 为什么用 B+Tree 而不是 B-Tree 或 Hash？（必问）

**要点**：

| 数据结构       | 缺点                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------- |
| **Hash**   | 不支持范围查询（如 `>`、`<`、`between`）；不支持排序；哈希冲突时性能下降                                                |
| **B-Tree** | 叶节点和非叶节点都存数据，导致树更"胖"（相同高度下存的数据量少）；范围查询需要多次回溯B-Tree；因为非叶子节点也存数据，**每个节点能存的“指针”数量很少**，树很快就长高了。 |
| **B+Tree** | 只有叶节点存数据，非叶节点存索引 → 树更"矮胖"；叶节点用链表相连 → 范围查询友好                                                 |

**B+Tree 的核心优势**：
1. **磁盘 I/O 更少**：非叶节点不存数据，可存更多索引，树高通常 2-3 层
2. **范围查询高效**：叶子节点形成有序链表，查完一个值直接遍历后续节点
3. **查询稳定**：所有查询都要走到叶子节点，时间复杂度稳定 O(log n)

**面试回答模板**：
> "MySQL 用 B+Tree 而不是 B-Tree，主要有三个原因：
> 1. B+Tree 的非叶子节点只存索引不存数据，每个磁盘块能存更多索引项，树的高度更低，磁盘 I/O 次数更少（InnoDB 中树高通常 2-3 层，千万级数据 2-3 次 I/O 就能查到）
> 2. B+Tree 的叶子节点用双向链表连接，做范围查询时，查完一个值直接遍历链表即可，不需要像 B-Tree 那样回溯
> 3. B+Tree 所有查询都要到叶子节点，查询时间稳定；Hash 虽然单点查询 O(1)，但不支持范围查询和排序，所以 InnoDB 的自适应哈希索引只是作为 B+Tree 的补充"

## 第 2 题：聚簇索引和非聚簇索引（二级索引）的区别？（必问）

**要点**：

| 对比项      | 聚簇索引                                                | 二级索引            |
| -------- | --------------------------------------------------- | --------------- |
| **定义**   | 叶子节点存整行数据                                           | 叶子节点存主键值        |
| **数量**   | 一个表只能有一个                                            | 可以有多个           |
| **默认**   | 主键就是聚簇索引；没有主键就用第一个 NOT NULL UNIQUE 列；都没有就用隐藏 row_id | 普通索引、唯一索引都是二级索引 |
| **查询效率** | 直接查到数据                                              | 查到主键后再回表        |

**关键理解**：
```sql
-- 聚簇索引：主键查询，一次 I/O 直接拿到数据
SELECT * FROM user WHERE id = 1;

-- 二级索引：name 索引查到主键 id，再回表查聚簇索引拿整行数据
SELECT * FROM user WHERE name = '张三';
```

**面试回答模板**：
> "聚簇索引的叶子节点存储的是整行数据，数据和索引存储在一起，所以一个表只能有一个聚簇索引，默认是主键。二级索引的叶子节点存储的是主键值，查询时需要先查二级索引拿到主键，再回聚簇索引查完整数据，这个过程叫回表。
>
> 聚簇索引的优势是主键查询非常快，缺点是更新主键或页分裂时代价较高；二级索引的优势是可以建立多个来加速不同字段的查询，缺点是存在回表开销。"

## 第 3 题：什么是回表和覆盖索引？（必问）

**要点**：

**回表**：用二级索引查询时，先查到主键，再去聚簇索引查完整数据的过程。

```sql
-- name 是二级索引
SELECT * FROM user WHERE name = '张三';
-- 流程：name 索引 → 拿到 id → 聚簇索引 → 拿到完整行
```

**覆盖索引**：要查询的字段都在索引里，不需要回表。

```sql
-- name 是二级索引，id 是主键（name 索引的叶子节点有 id）
SELECT id, name FROM user WHERE name = '张三';
-- 不需要回表，因为 id 和 name 都能从二级索引直接拿到
```

**如何实现覆盖索引**：
```sql
-- 联合索引 (name, age)
CREATE INDEX idx_name_age ON user(name, age);

-- 查询 name 和 age 都从索引拿，不回表
SELECT name, age FROM user WHERE name = '张三';
```

**面试回答模板**：
> "回表是指通过二级索引查到主键后，再回到聚簇索引查整行数据的过程。覆盖索引是指查询需要的所有字段都在索引树里，不需要回表。
>
> 覆盖索引能大幅提升查询性能，因为省去了回表的 I/O 开销。实际开发中，我会尽量把高频查询的字段建到联合索引里，比如 `SELECT id, name, age` 三个字段，就建 `(name, age)` 索引，这样通过 name 查询时能直接拿到 age，不用回表。"
## 第 4 题：联合索引的最左前缀法则是什么？（超高频）

**要点**：

联合索引 `(a, b, c)` 相当于创建了三个索引：`(a)`、`(a,b)`、`(a,b,c)`。

**能走索引的情况**：
```sql
WHERE a = 1           -- ✅ 走索引
WHERE a = 1 AND b = 2 -- ✅ 走索引
WHERE a = 1 AND b = 2 AND c = 3 -- ✅ 走索引
WHERE a = 1 AND c = 3 -- ✅ 走索引（只用 a，c 用不到）
WHERE a > 1 AND b = 2 -- ✅ 走索引（a 范围查询后，b 用不到）
```

**不能走索引的情况**：
```sql
WHERE b = 2           -- ❌ 不走索引（没有最左列 a）
WHERE b = 2 AND c = 3 -- ❌ 不走索引
WHERE a = 1 OR b = 2  -- ❌ OR 可能导致不走索引
```

**关键规则**：
1. 查询条件必须包含最左列
2. 遇到范围查询（`>`、`<`、`between`、`like`）会停止匹配
3. 查询条件的顺序不影响 MySQL 优化器调整

**面试回答模板**：
> "最左前缀法则是联合索引的核心原则。对于联合索引 `(a,b,c)`，MySQL 会从左到右匹配查询条件，只有匹配了左边的列，才能使用右边的列。如果跳过中间的列，后面的列就用不到索引；如果遇到范围查询，右边的列也会停止匹配。
>
> 举个例子，索引 `(name, age, city)`，查询 `WHERE name = '张三' AND city = '北京'` 只能用 name 字段，city 用不到，因为跳过了 age。所以建联合索引时，要把等值查询的字段放左边，范围查询的字段放右边。"
## 第 5 题：哪些情况会导致索引失效？（必问，能说出 5 条以上加分）

**要点**（记住 6-8 条）：

| 失效场景                   | 示例                                           | 原因                          |
| ---------------------- | -------------------------------------------- | --------------------------- |
| **1. 对索引列用函数或计算**      | `WHERE LEFT(name,3) = '张'`                   | 索引存的是原值，不是函数结果              |
| **2. 隐式类型转换**          | `WHERE phone = 13800138000`（phone 是 varchar） | 相当于 `CAST(phone AS signed)` |
| **3. 使用 OR 且两边不全是索引列** | `WHERE name = '张三' OR age = 20`（age 无索引）     | 必须全表扫描                      |
| **4. 使用 != 或 <>**      | `WHERE status != 1`                          | 不走索引（除非值极少）                 |
| **5. LIKE 以 % 开头**     | `WHERE name LIKE '%三'`                       | 不知道前缀，无法定位                  |
| **6. 索引列参与 NOT IN**    | `WHERE id NOT IN (1,2,3)`                    | 数据量小时可能走全表                  |
| **7. 联合索引不满足最左前缀**     | `WHERE b = 1`（索引 `(a,b)`）                    | 缺少最左列 a                     |
| **8. 范围查询右边列**         | `WHERE a > 1 AND b = 2`                      | 范围后停止匹配                     |
| **9. 全表扫描比索引快**        | 小表或查询 30%+ 数据                                | 优化器选择                       |

**面试回答模板**：
> "索引失效的情况我总结了几种：
> 1. 对索引列用函数或计算，比如 `WHERE DATE(create_time) = '2024-01-01'`，应该改成 `WHERE create_time BETWEEN ...`
> 2. 隐式类型转换，比如 varchar 字段传了数字
> 3. LIKE 以 `%` 开头
> 4. OR 条件两边不全是索引列
> 5. 联合索引不满足最左前缀
> 6. 用 `!=` 或 `<>` 操作符
>
> 排查时我会先用 Explain 看 type 和 key，如果 type 是 ALL 或 index，key 是 NULL，说明索引没走。"
## 第 6 题：如何定位和优化慢查询？

**要点**：

**第 1 步：开启慢查询日志**
```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
-- 设置超过 1 秒就算慢查询
SET GLOBAL long_query_time = 1;
```

**第 2 步：用 Explain 分析 SQL**
```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**第 3 步：重点看 Explain 的关键列**

| 列                 | 好坏顺序                                                               | 含义                |
| ----------------- | ------------------------------------------------------------------ | ----------------- |
| **type**          | `const` > `ref` > `range` > `index` > `ALL`                        | ALL 表示全表扫描，要优化    |
| **possible_keys** | -                                                                  | 可能用的索引            |
| **key**           | -                                                                  | 实际用的索引（NULL 表示没用） |
| **rows**          | 越小越好                                                               | 预估扫描行数            |
| **Extra**         | `Using index`（覆盖索引）好；`Using filesort`（需要优化）；`Using temporary`（要警惕） | 额外信息              |

**第 4 步：针对性优化**
- type = ALL → 加索引
- Extra = Using filesort → 对排序字段加索引
- rows 很大 → 优化 where 条件，增加过滤性

**面试回答模板**：
> "定位慢查询我会先开启慢查询日志，设置 `long_query_time=1`。找到具体 SQL 后用 `EXPLAIN` 分析。重点看三个字段：
> - `type`：如果是 ALL 说明全表扫描，必须加索引
> - `key`：如果为 NULL 说明索引没生效
> - `Extra`：`Using filesort` 说明需要优化排序，`Using temporary` 说明需要优化分组
>
> 优化时我会遵循：给 where、order by、group by 的字段建索引；联合索引遵循最左前缀；避免索引失效的情况。"
## 第 7 题：什么是索引下推？（进阶加分题）

**要点**：

**索引下推（Index Condition Pushdown，ICP）**：MySQL 5.6 引入，**减少回表次数**。

**没有 ICP 时**：
```sql
-- 索引 (name, age)
SELECT * FROM user WHERE name LIKE '张%' AND age = 20;
```
1. 用 name 索引找到所有姓"张"的记录的**主键**
2. **回表**取完整行
3. 判断 age=20

**有 ICP 后**：
1. 用 name 索引找到姓"张"的记录
2. **在索引树里**直接判断 age=20（索引里有 age 字段）
3. 过滤后再回表

**效果**：减少回表次数，尤其适用于 like 查询。

**面试回答模板**：
> "索引下推是 MySQL 5.6 引入的优化，可以在索引遍历过程中直接过滤掉不符合条件的记录，减少回表次数。
>
> 比如联合索引 `(name, age)` 查询 `WHERE name LIKE '张%' AND age=20`，没有 ICP 时会把所有姓张的主键都回表查一遍；有 ICP 时会在索引里先用 age=20 过滤，只对符合条件的回表。这个优化对 like 前缀匹配非常有效。"

## 第 8 题：如何设计一个优秀的索引？（综合题）

**要点**（面试官喜欢问开放性问题）：

| 原则                     | 说明                           |
| ---------------------- | ---------------------------- |
| **1. 区分度高的列放前面**       | 性别（区分度低）不适合；手机号（区分度高）适合      |
| **2. 等值查询放左边，范围查询放右边** | `(name, age)`：name 等值，age 范围 |
| **3. 覆盖索引优先**          | 高频查询字段尽量建覆盖索引                |
| **4. 索引不是越多越好**        | 维护成本（增删改变慢），一般单表不超过 5-6 个索引  |
| **5. 长字段用前缀索引**        | `VARCHAR(255)` 可以只索引前 20 个字符 |
| **6. 避免重复索引**          | `(a)` 和 `(a,b)` 有重叠          |
| **7. 注意索引列的顺序**        | 最左前缀决定顺序                     |


**面试回答模板**：
> "设计索引我会遵循几个原则：
> 1. 先分析业务 SQL，找出高频的 where、order by、group by 字段
> 2. 区分度高的列优先，比如手机号比性别好
> 3. 联合索引把等值查询的放左边，范围查询的放右边
> 4. 尽量用覆盖索引，避免回表
> 5. 控制索引数量，一般单表不超过 5 个，否则影响写入性能
>
> 比如用户表有查询 `SELECT name, age FROM user WHERE phone = 'xxx'`，我会建 `(phone, name, age)` 联合索引，覆盖索引直接返回数据，不回表。"

## 高频追问汇总

### Q：主键为什么推荐用自增 ID，不用 UUID？

> 自增 ID 是顺序的，B+Tree 插入只在最右边操作，页分裂少；UUID 随机，插入位置随机，页分裂频繁，造成碎片。另外 UUID 占用 16 字节，ID 只用 4 或 8 字节，每页能存更多索引。

### Q：为什么不能用 Null 的列做索引？

> 索引中 Null 值会占用空间，且每次比较用 `IS NULL` 不走索引。还要考虑可空列在联合索引中的失效场景。

### Q：索引维护代价多大？

> 插入：B+Tree 结构调整；删除：页合并；更新：先删后插。索引越多，写入越慢。

## 第 1 题：如何定位慢 SQL？（必问）

**要点**：

### 方法一：开启慢查询日志

```sql
-- 查看慢查询状态
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';

-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
-- 设置阈值：超过 1 秒记录
SET GLOBAL long_query_time = 1;
-- 设置日志文件路径
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

### 方法二：使用 `SHOW PROCESSLIST`

```sql
-- 查看当前正在执行的 SQL
SHOW FULL PROCESSLIST;

-- 找出执行时间长的，kill 掉
KILL <connection_id>;
```

### 方法三：使用 `performance_schema`（MySQL 5.7+）

```sql
-- 查看最近执行的慢语句
SELECT * FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC LIMIT 10;
```

**面试回答模板**：
> "定位慢 SQL 我会先开启慢查询日志，设置 `long_query_time=1`，跑一段时间后分析慢日志。生产环境我还会用 `SHOW PROCESSLIST` 实时抓取执行时间长的 SQL。MySQL 5.7 以上可以用 `performance_schema` 查看语句汇总统计，按执行时间排序找出问题 SQL。"

## 第 2 题：Explain 执行计划的关键列怎么看？（超高频）

**要点**：拿到慢 SQL 后，第一步就是 `EXPLAIN`。

```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**输出示例**：
| id  | select_type | table | type | possible_keys  | key | key_len | ref  | rows  | Extra |
|----|-------- -----|-------|------|-----------------|-----|----------|-----|-------|-------|
| 1 | SIMPLE | user | ref | idx_name | idx_name | 303 | const | 1 | Using index condition |

### 重点列详解（按重要性排序）

#### 1. `type`（访问类型）—— **最重要的指标**

**好坏顺序**：
```
system > const > eq_ref > ref > range > index > ALL
（从左到右，越左越好）
```

| type       | 含义        | 优化建议                           |
| ---------- | --------- | ------------------------------ |
| **ALL**    | 全表扫描      | ❌ 必须加索引                        |
| **index**  | 全索引扫描     | ⚠️ 扫描整个索引树，比 ALL 好一点但也很差       |
| **range**  | 范围扫描      | ✅ `BETWEEN`、`>`、`<`、`IN` 等，可接受 |
| **ref**    | 非唯一索引等值查询 | ✅ 良好，常见于普通索引                   |
| **eq_ref** | 唯一索引等值查询  | ✅ 优秀，常见于主键/唯一键关联               |
| **const**  | 常量查询      | ✅ 最好，主键/唯一键单点查询                |
| **system** | 系统表查询     | ✅ 极少见                          |

#### 2. `key`（实际使用的索引）
- `NULL` 表示没走索引 → 需要优化
- 有值表示用了该索引

#### 3. `rows`（预估扫描行数）
- 越小越好，但不是绝对精准
- 如果 `rows` 很大但 `type` 是 `ref`，可能还是需要优化

#### 4. `Extra`（额外信息）
| Extra 值                   | 含义         | 优化建议                      |
| ------------------------- | ---------- | ------------------------- |
| **Using index**           | 覆盖索引 ✅     | 很好，不需要回表                  |
| **Using index condition** | 索引下推       | 可接受                       |
| **Using where**           | 需要回表过滤     | 看情况，可尝试覆盖索引               |
| **Using filesort**        | 需要额外排序 ⚠️  | 给排序字段加索引                  |
| **Using temporary**       | 使用临时表 ❌    | 严重，优化 group by / distinct |
| **Using join buffer**     | 没有索引的 join | 给关联字段加索引                  |

**面试回答模板**：
> "用 Explain 分析 SQL 时，我重点看三个字段：
> 1. **type**：如果是 ALL 或 index，说明在扫全表或全索引，必须优化到至少 range 级别
> 2. **key**：如果为 NULL，说明索引没生效，检查索引失效场景
> 3. **Extra**：看到 `Using filesort` 说明需要加排序索引，看到 `Using temporary` 说明要优化 group by 或 distinct，看到 `Using index` 说明覆盖索引，效果最好"

## 第 3 题：有哪些常见的 SQL 优化手段？（必问）

**要点**（记住 8-10 条）：

| 优化手段                     | 说明                              | 示例                                    |
| ------------------------ | ------------------------------- | ------------------------------------- |
| **1. 避免 `SELECT *`**     | 只查需要的列，减少网络传输和回表                | `SELECT id, name` 代替 `SELECT *`       |
| **2. 用小表驱动大表**           | join 时把小表放前面                    | `FROM small_table JOIN big_table`     |
| **3. `LIKE` 不要以 `%` 开头** | `%xxx` 不走索引                     | 改 `xxx%` 或用搜索引擎                       |
| **4. `OR` 改 `UNION`**    | 两边都有索引才走                        | 或用 `IN` 代替                            |
| **5. `IN` 里的值不要太多**      | 超过 1000 个可能不走索引                 | 分批查询                                  |
| **6. `EXISTS` 替代 `IN`**  | 子查询结果集大时                        | `WHERE EXISTS (SELECT 1 ...)`         |
| **7. 分页优化**              | 大 offset 慢                      | 用游标/子查询/延迟关联                          |
| **8. 避免 `!=`、`<>`**      | 不走索引                            | 改成 `IN` 或分开查询                         |
| **9. 避免函数操作索引列**         | `WHERE DATE(create_time) = ...` | 改成范围查询                                |
| **10. 批量操作**             | 避免循环单条 SQL                      | `INSERT INTO ... VALUES (...), (...)` |

### 重点展开两个：

**分页优化（大 offset 问题）**：
```sql
-- ❌ 慢：offset 越大越慢，要扫描前面 100000 条再跳过
SELECT * FROM user ORDER BY id LIMIT 100000, 10;

-- ✅ 快：先查 id，再 join
-- 延迟关联（Deferred Join）
SELECT * FROM user 
WHERE id > (SELECT id FROM user ORDER BY id LIMIT 100000, 1)
LIMIT 10;

-- ✅ 快：用上次查询的最大 id 做游标
-- 游标分页（Cursor-based Pagination / 游标法）
SELECT * FROM user WHERE id > 100000 ORDER BY id LIMIT 10;
```

**批量操作**：
```sql
-- ❌ 慢：循环 1000 次
for (userId in userIdList) {
    INSERT INTO user_log VALUES (userId);
}

-- ✅ 快：一次批量插入
INSERT INTO user_log VALUES (1), (2), (3), ... (1000);
```

**面试回答模板**：
> "SQL 优化我常用的手段有：
> 1. 永远不用 `SELECT *`，只查需要的字段
> 2. 分页用游标或延迟关联代替大 offset
> 3. 批量操作代替循环单条 SQL
> 4. 避免在索引列上用函数或计算
> 5. `LIKE` 不要以 `%` 开头
> 6. 用 `EXISTS` 代替数据量大的 `IN` 子查询
> 7. 小表驱动大表进行 join
>
> 比如有一个订单表分页查询，从第 10 万页开始查，用 `LIMIT 100000, 10` 会很慢，我会改成 `WHERE id > (SELECT id FROM order LIMIT 100000, 1) LIMIT 10`，速度能提升几十倍。"

## 第 4 题：`COUNT(*)`、`COUNT(1)`、`COUNT(列)` 有什么区别？哪个快？

**要点**：

| 写法         | 行为              | 是否统计 NULL     |
| ---------- | --------------- | ------------- |
| `COUNT(*)` | 统计行数            | 是（不管有没有 NULL） |
| `COUNT(1)` | 统计行数            | 是（和 `*` 完全一样） |
| `COUNT(列)` | 统计该列非 NULL 值的个数 | 否             |

**性能对比**（MySQL 中）：
- **InnoDB**：`COUNT(*)` 和 `COUNT(1)` 没区别，优化器会选最小的索引来统计
- **`COUNT(列)`**：如果列有 NOT NULL，和 `COUNT(*)` 一样；如果允许 NULL，会慢一些

**面试回答模板**：
> "在 InnoDB 中，`COUNT(*)` 和 `COUNT(1)` 性能完全一样，优化器会选最小的二级索引来统计行数。`COUNT(列)` 只统计该列非 NULL 的行，如果列允许 NULL，会比 `COUNT(*)` 慢一点。
>
> 实际开发中我推荐用 `COUNT(*)`，语义最清晰，性能也是最优的。如果需要统计某个字段的非 NULL 数量，再用 `COUNT(列)`。"
## 第 5 题：Join 优化有哪些要点？（高频）

**要点**：

### 1. 小表驱动大表
```sql
-- ✅ 好：小表 left join 大表
SELECT * FROM small_table s LEFT JOIN big_table b ON s.id = b.id;

-- ❌ 差：大表 left join 小表
SELECT * FROM big_table b LEFT JOIN small_table s ON b.id = s.id;
```

### 2. 确保 join 字段有索引
- 被驱动表（join 的右表）的关联字段必须有索引

### 3. 避免复杂的子查询，改写成 Join
```sql
-- ❌ 慢：子查询
SELECT * FROM user WHERE dept_id IN (SELECT id FROM dept WHERE status = 1);

-- ✅ 快：用 Join
SELECT u.* FROM user u INNER JOIN dept d ON u.dept_id = d.id WHERE d.status = 1;
```

### 4. 如果 Join 太多，考虑反范式设计
- 适当冗余字段，减少 Join 次数

**面试回答模板**：
> "Join 优化我遵循三个原则：
> 1. 小表驱动大表，减少被驱动表的扫描次数
> 2. 被驱动表的 join 字段必须建索引，最好走到 ref 级别
> 3. 能用 Join 写的子查询就用 Join，避免子查询多次执行
>
> 举个例子，用户表和部门表关联查询，我会把小表（部门表）放前面，并在用户表的 dept_id 字段上建索引。"

## 第 6 题：给你一个慢 SQL，说说你的调优步骤？（综合场景题，必问）

### 标准调优 6 步法

**第 1 步：开启慢查询日志，拿到具体 SQL**

```

**第 2 步：用 Explain 分析执行计划**

**第 3 步：识别问题**

看 Explain 输出，常见问题：
| 问题 | 对应 Explain 字段 |

| 全表扫描 | type = ALL |
| 没用索引 | key = NULL |
| 文件排序 | Extra = Using filesort |
| 临时表 | Extra = Using temporary |
| 扫描行数太多 | rows 很大 |

**第 4 步：根据问题制定优化方案**

| 问题              | 优化方案                      |
| --------------- | ------------------------- |
| 全表扫描            | 加索引（考虑联合索引）               |
| 没用索引            | 检查索引失效场景，改写 SQL           |
| Using filesort  | 给排序字段加索引，或让排序字段进入索引       |
| Using temporary | 优化 group by、distinct，或加索引 |
| rows 太大         | 增加过滤条件，或调整索引顺序            |

-- 原 SQL

-- 优化：建联合索引 (user_id, status, create_time)
-- 这样 where 和 order by 都能用到索引
ALTER TABLE order ADD INDEX idx_user_status_time (user_id, status, create_time);
```

**第 6 步：再次 Explain 验证，上线观察**

```sql

-- 期望：type=ref, key=idx_user_status_time, Extra=Using index condition
```

> "遇到慢 SQL，我的调优步骤是：
> 
> 第一步，先用 `EXPLAIN` 分析执行计划，看 `type`、`key`、`rows`、`Extra` 四个关键字段。
> 
> 第二步，根据问题分类解决：
> - 如果 `type=ALL` 或 `key=NULL`，说明没走索引，我会检查 where 和 join 字段有没有索引，有没有索引失效的情况
> - 如果 `Extra=Using filesort`，我会在 `ORDER BY` 的字段上建索引，或者把它加到联合索引里
> - 如果 `rows` 很大，我会检查过滤条件是否充分，或者调整联合索引的列顺序
> 
> 第三步，加完索引或改完 SQL 后，再用 `EXPLAIN` 验证，看到 `type` 变成 `ref` 或 `range`，`rows` 明显变小，就算优化完成。
> 
> 举个例子，一个订单表查询 `WHERE user_id=123 AND status=1 ORDER BY create_time`，我会建联合索引 `(user_id, status, create_time)`，这样三个条件都走索引，避免回表和文件排序。"

## 附加题：Order By 优化

**要点**：

```sql
-- ❌ 慢：索引 (user_id)，order by 不走索引
SELECT * FROM order WHERE user_id = 123 ORDER BY create_time;

-- ✅ 快：索引 (user_id, create_time)，order by 也用索引
CREATE INDEX idx_user_time ON order(user_id, create_time);
SELECT * FROM order WHERE user_id = 123 ORDER BY create_time;
```

**原则**：
- 让 `ORDER BY` 的字段是**联合索引的最后一列**
- `ORDER BY` 的顺序要和索引列顺序一致
- `ASC` 和 `DESC` 要统一，否则不会走索引
## 面试官常问的追问

### Q：`LIMIT 100000, 10` 为什么慢？怎么优化？

> 慢是因为要扫描 100010 条数据，然后扔掉前 100000 条。优化方法：
> 1. 用 `WHERE id > 上次最大id` 做游标分页
> 2. 用子查询先取 id：`SELECT * FROM user WHERE id IN (SELECT id FROM user LIMIT 100000, 10)`

### Q：`GROUP BY` 怎么优化？

> 给 `GROUP BY` 的字段加索引，或者用 `WHERE` 先过滤数据再分组，减少分组的数据量。

### Q：`UNION` 和 `UNION ALL` 的区别？

> `UNION` 会去重（代价高），`UNION ALL` 不去重。如果结果集不会有重复，用 `UNION ALL`。

# **事务与隔离级别**

## 事务与隔离级别知识地图

```
事务
├── ACID 特性
├── 并发问题（脏读、不可重复读、幻读）
├── 隔离级别（4 种）
├── MVCC 原理（核心！）
└── 事务日志（Redo/Undo）
```
## 第 1 题：什么是事务？ACID 是什么？（必问）

**要点**：

**事务**：一组 SQL 操作的集合，要么全部成功，要么全部失败。

**ACID 四特性**：

| 特性                    | 含义               | 实现方式           |
| --------------------- | ---------------- | -------------- |
| **原子性 (Atomicity)**   | 事务内操作要么全成功，要么全失败 | Undo Log（回滚日志） |
| **一致性 (Consistency)** | 事务前后数据状态一致       | 其他三个特性共同保证     |
| **隔离性 (Isolation)**   | 事务之间互不干扰         | MVCC + 锁机制     |
| **持久性 (Durability)**  | 事务提交后数据永久保存      | Redo Log（重做日志） |

**面试回答模板**：
> "事务是数据库执行的最小工作单元，满足 ACID 四个特性。原子性由 Undo Log 保证，持久性由 Redo Log 保证，隔离性由 MVCC 和锁实现，一致性是最终目标。比如银行转账，扣款和加款必须在同一个事务里，要么都成功，要么都失败。"
## 第 2 题：并发事务会带来哪些问题？（必问）

**要点**：三个典型问题

| 问题        | 定义                          | 示例                                              |
| --------- | --------------------------- | ----------------------------------------------- |
| **脏读**    | 读到其他事务**未提交**的数据            | 事务 A 改余额 100→200 未提交，事务 B 读到 200，A 回滚，B 读到的是脏数据 |
| **不可重复读** | 同一事务内，两次读同一行数据，结果**不一致**    | 事务 A 两次读余额：第一次 100，期间事务 B 改成 200 并提交，A 第二次读 200 |
| **幻读**    | 同一事务内，两次查询**记录数**不一致（多了或少了） | 事务 A 查用户数=10，事务 B 插入一条新用户，A 再查=11，像出现幻觉         |

**区别图解**：

```text
不可重复读（针对已有数据）：
事务A第1次读：余额 = 100
     ↓
事务B：UPDATE 余额 SET = 200（提交）
     ↓
事务A第2次读：余额 = 200  ← 同一行数据变了

幻读（针对记录数）：
事务A第1次查：WHERE age > 18 → 10条记录
     ↓
事务B：INSERT 新用户 age=20（提交）
     ↓
事务A第2次查：WHERE age > 18 → 11条记录  ← 多了一条
```

**面试回答模板**：
> "并发事务会带来三个问题：脏读、不可重复读、幻读。
> - 脏读：读到未提交的数据，一旦事务回滚就出问题
> - 不可重复读：同一事务内多次读同一行，数据变了，通常由 UPDATE 引起
> - 幻读：同一事务内多次查询，记录数变了，通常由 INSERT/DELETE 引起
>
> 不可重复读关注的是值的变化，幻读关注的是行数的变化。"
## 第 3 题：MySQL 的隔离级别有哪些？分别解决什么问题？（超高频）

**要点**：MySQL 有 4 种隔离级别（从低到高）

| 隔离级别                        |  脏读  | 不可重复读 |        幻读        |
| --------------------------- | :--: | :---: | :--------------: |
| **读未提交 (Read Uncommitted)** | ✅ 可能 | ✅ 可能  |       ✅ 可能       |
| **读已提交 (Read Committed)**   | ❌ 解决 | ✅ 可能  |       ✅ 可能       |
| **可重复读 (Repeatable Read)**  | ❌ 解决 | ❌ 解决  | ⚠️ InnoDB 用间隙锁解决 |
| **串行化 (Serializable)**      | ❌ 解决 | ❌ 解决  |       ❌ 解决       |

> **注意**：MySQL 默认隔离级别是 **可重复读（Repeatable Read）**，而 Oracle、SQL Server 默认是读已提交。

```sql
-- 查看当前隔离级别
SELECT @@transaction_isolation;

-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

**面试回答模板**：
> "MySQL 有四种隔离级别：
> 1. 读未提交：会出现脏读、不可重复读、幻读，基本不用
> 2. 读已提交：解决了脏读，但还会出现不可重复读和幻读，Oracle 默认是这个
> 3. 可重复读：解决了脏读和不可重复读，MySQL 默认级别，InnoDB 用间隙锁解决了幻读
> 4. 串行化：所有问题都解决，但性能最差，相当于所有事务串行执行
>
> MySQL 默认用可重复读，因为要兼容主从复制；Oracle 用读已提交，因为性能更好。"
## 第 4 题：MVCC 是什么？怎么实现可重复读？（超高频，重点）

**要点**：

**MVCC（Multi-Version Concurrency Control，多版本并发控制）**：  
> 通过保存数据的多个版本，实现**读不加锁、读写不冲突**，从而提升并发性能。

### MVCC 的核心机制

**第 1 步：隐藏列**

InnoDB 每行数据背后有三个隐藏列：

| 隐藏列             | 说明                           |
| --------------- | ---------------------------- |
| **DB_TRX_ID**   | 最后修改该行的事务 ID（6 字节）           |
| **DB_ROLL_PTR** | 回滚指针，指向 Undo Log 中的旧版本（7 字节） |
| **DB_ROW_ID**   | 行 ID，没有主键时用（6 字节，可选）         |


**第 2 步：Undo Log 版本链**

```text
┌─────────────────────────────────────────────────────────┐
│ 某行数据的不同版本（通过 DB_ROLL_PTR 串联）                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  最新版本 (trx_id=102)                                  │
│  ┌─────────────┐                                        │
│  │ name = '李四'│ ← DB_ROLL_PTR ──┐                     │
│  │ trx_id=102  │                   │                     │
│  └─────────────┘                   ↓                     │
│                           ┌─────────────┐               │
│                           │ name = '张三'│ ← DB_ROLL_PTR ─┐
│                           │ trx_id=101  │                 │
│                           └─────────────┘                 ↓
│                                                  ┌─────────────┐
│                                                  │ name = '王五'│
│                                                  │ trx_id=100  │
│                                                  └─────────────┘
└─────────────────────────────────────────────────────────────────┘
```

**第 3 步：Read View（读视图）**

事务开始时，会生成一个 Read View，包含以下信息：

| 字段                 | 含义                   |
| ------------------ | -------------------- |
| **m_ids**          | 当前活跃（未提交）的所有事务 ID 列表 |
| **min_trx_id**     | m_ids 中的最小值          |
| **max_trx_id**     | 系统下一个要分配的事务 ID       |
| **creator_trx_id** | 当前事务自己的 ID           |

**第 4 步：可见性判断**

读数据时，根据 DB_TRX_ID 和 Read View 判断哪个版本可见：

```text
DB_TRX_ID < min_trx_id ? 
    → 是：该事务已提交，可见 ✅
    → 否：继续判断

DB_TRX_ID >= max_trx_id ?
    → 是：事务在 Read View 之后开始，不可见 ❌
    → 否：继续判断

DB_TRX_ID 在 m_ids 中？
    → 是：事务未提交，不可见 ❌（用 Undo 找旧版本）
    → 否：事务已提交，可见 ✅
```

**面试回答模板（完整）**：
> "MVCC 是多版本并发控制，InnoDB 通过它实现可重复读隔离级别，核心是读不加锁。
>
> 原理是：每行数据都有隐藏列 `DB_TRX_ID`（最后修改的事务ID）和 `DB_ROLL_PTR`（指向旧版本的指针），旧版本存在 Undo Log 中，形成版本链。
>
> 事务开始时，会生成一个 Read View，记录当前活跃的事务 ID 列表。读取数据时，根据 `DB_TRX_ID` 和 Read View 的规则判断应该读取哪个版本：只读已提交的数据，或者读历史版本。
>
> 在可重复读级别下，Read View 是在**事务的第一个 SELECT 时创建**的，整个事务期间复用这个 Read View，所以每次读到的都是同一个快照，保证了可重复读。而在读已提交级别下，每次 SELECT 都会重新生成 Read View。"
## 第 5 题：可重复读级别下，InnoDB 如何解决幻读？

**要点**：

**可重复读级别下，普通 SELECT 用 MVCC 快照读**，不会产生幻读（因为读的是历史快照）。

但如果是**当前读**（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`），会产生幻读问题，InnoDB 用 **间隙锁（Gap Lock）** 解决。

**示例**：

```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE age BETWEEN 20 AND 30 FOR UPDATE;
-- 此时，InnoDB 会锁住 age 在 20-30 之间的间隙
-- 事务B 插入 age=25 的数据会被阻塞
```

**间隙锁锁的是"间隙"**，防止其他事务在这个区间插入数据。

**面试回答模板**：
> InnoDB 在可重复读级别下，通过两种方式解决幻读：
> 1. 快照读（普通 SELECT）：利用 MVCC 读历史快照，本身不会有幻读
> 2. 当前读（SELECT FOR UPDATE、UPDATE、DELETE）：使用**间隙锁**锁住查询条件范围内的间隙，阻止其他事务插入新数据
>
> 比如查询 `age BETWEEN 20 AND 30`，InnoDB 会锁住 20-30 这个区间，其他事务无法插入 age=25 的记录，从而避免幻读。
## 第 6 题：读已提交和可重复读的本质区别是什么？

**要点**：

| 对比项                | 读已提交 (RC)         | 可重复读 (RR)                |
| ------------------ | ----------------- | ------------------------ |
| **Read View 创建时机** | 每条 SELECT 语句都重新生成 | 事务内第一条 SELECT 时生成，整个事务复用 |
| **间隙锁**            | 没有间隙锁             | 有间隙锁（解决幻读）               |
| **并发性能**           | 更高                | 稍低（间隙锁会增加阻塞）             |
| **是否出现幻读**         | 会                 | 不会                       |
| **默认数据库**          | Oracle、SQL Server | MySQL InnoDB             |

**面试回答模板**：
> "两者的本质区别是 **Read View 的创建时机**不同：
> - 读已提交：每次 SELECT 都会生成新的 Read View，所以能看到其他事务已提交的更新，导致不可重复读
> - 可重复读：事务内第一次 SELECT 时生成 Read View，整个事务期间复用，所以读到的数据始终一致
>
> 另外，可重复读有间隙锁机制防止幻读，读已提交没有间隙锁。"

## 第 7 题：事务隔离级别怎么选择？

**要点**：

| 场景             | 推荐隔离级别 | 原因                |
| -------------- | ------ | ----------------- |
| 普通 Web 应用      | 读已提交   | 性能好，不可重复读大多数场景可接受 |
| 对数据一致性要求高（如金融） | 可重复读   | 避免不可重复读和幻读        |
| 超高并发、允许脏读      | 读未提交   | 极少用               |
| 一致性要求极高、并发低    | 串行化    | 几乎不用，性能太差         |

**阿里巴巴规范**：默认使用 **读已提交（RC）**。

**面试回答模板**：
> "隔离级别的选择要在数据一致性和性能之间做权衡。
> - 读未提交：会产生脏读，基本不用
> - 读已提交：大多数互联网公司用这个，因为不可重复读在大多数业务场景下可以接受，性能好
> - 可重复读：金融、账务类系统用，避免不可重复读，MySQL 默认就是这个级别
> - 串行化：性能最差，基本不用
>
> 实际开发中，我会根据业务场景选择：普通查询用读已提交；涉及金额、库存扣减等，用可重复读并在代码中加分布式锁兜底。"
## 高频追问汇总

### Q：事务中 SELECT 会加锁吗？

> 普通 SELECT 不加锁（快照读）。`SELECT ... FOR UPDATE` 或 `SELECT ... LOCK IN SHARE MODE` 才会加锁（当前读）。

### Q：长事务有什么问题？

> 1. Undo Log 会膨胀，占用大量磁盘
> 2. 锁等待和死锁概率增加
> 3. 主从复制延迟
> 建议：事务尽量短小，避免用户交互操作在事务中

### Q：MySQL 默认隔离级别为什么是可重复读？

> 历史原因：早期 MySQL 主从复制基于语句（Statement）时，如果使用读已提交，可能导致主从数据不一致。现在主从复制推荐用行（Row）模式，这个问题不存在了。但 MySQL 默认级别一直保留为可重复读。

## 第 1 题：MySQL 锁有哪些分类？（必问）

**要点**（按不同维度分类）：
### 一、按粒度分类

| 锁类型     | 粒度  | InnoDB | MyISAM | 说明          |
| ------- | --- | ------ | ------ | ----------- |
| **表锁**  | 最大  | ✅      | ✅      | 锁整张表，并发低    |
| **行锁**  | 最小  | ✅      | ❌      | 锁一行，并发高     |
| **页锁**  | 中间  | ❌      | ❌      | BDB 引擎用，很少见 |
| **间隙锁** | 间隙  | ✅      | ❌      | 锁住记录之间的空隙   |

### 二、按模式分类（S/X 锁）

| 锁模式     | 名称      | 兼容           | 说明                                    |
| ------- | ------- | ------------ | ------------------------------------- |
| **S 锁** | 共享锁（读锁） | S 兼容 S，不兼容 X | `SELECT ... LOCK IN SHARE MODE`       |
| **X 锁** | 排他锁（写锁） | X 不兼容任何锁     | `SELECT ... FOR UPDATE`、UPDATE、DELETE |
### 三、按意向分类（表级）

| 锁类型             | 说明                   |
| --------------- | -------------------- |
| **IS 锁**（意向共享锁） | 事务想对某行加 S 锁，先在表上加 IS |
| **IX 锁**（意向排他锁） | 事务想对某行加 X 锁，先在表上加 IX |

> **意向锁的作用**：避免表级锁和行级锁冲突。比如要给表加表锁时，先检查是否有意向锁，有则说明有行锁在占用，需等待。

**面试回答模板**：
> "MySQL 的锁可以从多个维度分类：按粒度分有表锁、行锁、间隙锁；按模式分有共享锁（S 锁）和排他锁（X 锁）；按意图分有意向共享锁（IS）和意向排他锁（IX）。
>
> InnoDB 默认使用行锁，支持更高的并发。MyISAM 只有表锁。我们项目中主要用 InnoDB，利用它的行锁和 MVCC 实现高并发。"

## 第 2 题：InnoDB 的行锁有哪些类型？（超高频）

**要点**：InnoDB 的行锁有 3 种，面试官最爱问区别

| 锁类型 | 英文 | 锁住的对象 | 产生条件 |
|--------|------|-----------|----------|
| **记录锁** | Record Lock | 锁住**一条记录** | 等值查询**唯一索引**（命中记录） |
| **间隙锁** | Gap Lock | 锁住记录之间的**间隙**（不包括记录本身） | 范围查询、等值查询没命中 |
| **临键锁** | Next-Key Lock | **记录锁 + 间隙锁** | 默认行锁，范围查询命中记录时 |

> **面试官最常问**：临键锁 = 记录锁 + 间隙锁（锁住记录本身 + 记录前面的间隙）

### 三种锁的产生条件

```sql
-- 1. 记录锁：等值查询唯一索引且命中
SELECT * FROM user WHERE id = 5 FOR UPDATE;
-- 只锁住 id=5 这一条记录

-- 2. 间隙锁：等值查询没命中
SELECT * FROM user WHERE id = 6 FOR UPDATE;
-- id=6 不存在，锁住 (5,7) 这个间隙，阻止插入 id=6

-- 3. 临键锁：范围查询命中记录
SELECT * FROM user WHERE id > 3 AND id < 7 FOR UPDATE;
-- 锁住 (3,5] 和 (5,7] 两个临键区间
```

**面试回答模板**：
> "InnoDB 的行锁有三种：
> 1. **记录锁**：锁住索引记录本身，等值查询唯一索引且命中时产生
> 2. **间隙锁**：锁住记录之间的空隙，防止幻读，只在可重复读隔离级别下存在
> 3. **临键锁**：记录锁 + 间隙锁，是 InnoDB 的默认行锁，范围查询命中记录时产生
>
> 间隙锁是 MySQL 解决幻读的关键，比如事务 A 查询 `id BETWEEN 1 AND 5`，InnoDB 会锁住这个区间，事务 B 无法在中间插入 id=3 的记录。"
## 第 3 题：什么是间隙锁？为什么要用它？（高频）

**要点**：

**间隙锁**：锁住索引记录之间的**空隙**，不锁记录本身。

**为什么需要间隙锁**：**解决幻读问题**

### 场景演示

```sql
-- 当前数据：id = 1, 2, 5

-- 事务A
BEGIN;
SELECT * FROM user WHERE id = 3 FOR UPDATE;  -- id=3 不存在
-- 此时 InnoDB 加了间隙锁，锁住 (2,5) 这个区间

-- 事务B
INSERT INTO user VALUES (3, 'xxx');  -- ❌ 被阻塞！因为 (2,5) 被锁了

-- 事务A 再次查询
SELECT * FROM user WHERE id = 3 FOR UPDATE;  -- 还是查不到
COMMIT;

-- 如果没有间隙锁：
-- 事务B 插入 id=3 成功，事务A 再次查询时突然出现了一条，产生幻读 ✅
```

**间隙锁的特点**：

| 特点                 | 说明            |
| ------------------ | ------------- |
| **只存在于可重复读级别**     | 读已提交级别没有间隙锁   |
| **唯一索引等值查询没命中时产生** | 锁住查询值所在区间     |
| **对性能有影响**         | 间隙锁会阻塞插入，降低并发 |
| **可能产生死锁**         | 多个事务互相持有间隙锁   |

**面试回答模板**：
> "间隙锁是 InnoDB 在可重复读隔离级别下用来解决幻读的机制。它锁住的是索引记录之间的空隙，而不是记录本身。
>
> 比如查询 `SELECT * FROM user WHERE id = 6 FOR UPDATE` 没有命中记录时，InnoDB 会锁住 id 在 5 和 7 之间的间隙，防止其他事务在这个区间插入 id=6 的记录，从而避免幻读。
>
> 注意：间隙锁只在可重复读级别存在，读已提交级别没有间隙锁。另外，间隙锁之间是兼容的，因为锁的空隙不存在冲突，但插入操作会被阻塞。"
## 第 4 题：什么是当前读和快照读？（必问）

**要点**：

| 读类型     | 英文            | 说明           | 是否加锁 |
| ------- | ------------- | ------------ | ---- |
| **快照读** | Snapshot Read | 读取历史版本（MVCC） | 不加锁  |
| **当前读** | Current Read  | 读取最新版本       | 加锁   |

### 快照读（不加锁）

```sql
-- 普通 SELECT 就是快照读
SELECT * FROM user WHERE id = 1;
-- 读的是 MVCC 的历史快照，不加任何锁
-- 在 RR 级别下，读的是事务开始时的快照
```

### 当前读（加锁）

```sql
-- 以下都是当前读，读取最新数据并加锁
SELECT * FROM user WHERE id = 1 FOR UPDATE;           -- 加 X 锁
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;   -- 加 S 锁
UPDATE user SET name = '李四' WHERE id = 1;           -- 加 X 锁
DELETE FROM user WHERE id = 1;                        -- 加 X 锁
INSERT INTO user VALUES ...;                          -- 加 X 锁
```

**面试回答模板**：
> "当前读和快照读是 InnoDB 的两种读取方式。
> - **快照读**：普通 SELECT 语句，读的是 MVCC 的历史版本，不加锁，不阻塞其他事务
> - **当前读**：`SELECT FOR UPDATE`、`UPDATE`、`DELETE`、`INSERT` 等操作，读的是数据的最新版本，会加锁（X 锁或 S 锁）
>
> 在可重复读级别下，快照读读到的是事务开始时的快照；当前读读到的是最新的数据，并通过间隙锁防止幻读。"

## 第 5 题：InnoDB 什么时候加表锁？（高频追问）

**要点**：虽然 InnoDB 主要是行锁，但某些情况会退化为表锁

| 场景                      | 说明                             |
| ----------------------- | ------------------------------ |
| **不走索引的 UPDATE/DELETE** | 无法确定哪些行，锁全表                    |
| **DDL 操作**              | `ALTER TABLE`、`DROP TABLE` 加表锁 |
| **LOCK TABLES 显式加锁**    | `LOCK TABLES user WRITE`       |
| **间隙锁升级**               | 范围太大时可能升表锁                     |

```sql
-- ❌ 危险：name 没有索引，会锁全表
UPDATE user SET status = 1 WHERE name = '张三';

-- ✅ 安全：id 有索引，只锁相关行
UPDATE user SET status = 1 WHERE id = 1;
```

**面试回答模板**：
> "InnoDB 主要用行锁，但以下几种情况会退化为表锁：
> 1. UPDATE/DELETE 语句的 WHERE 条件**不走索引**时，无法确定要锁哪些行，InnoDB 会锁全表
> 2. DDL 操作如 `ALTER TABLE`、`DROP TABLE`
> 3. 显式执行 `LOCK TABLES` 语句
>
> 所以在开发中，UPDATE 和 DELETE 一定要走索引，否则可能锁全表，导致严重性能问题。"

## 第 6 题：什么是死锁？如何排查和解决？（高频）

**要点**：

### 死锁产生的四个必要条件

| 条件    | 说明            |
| ----- | ------------- |
| 互斥    | 资源只能同时被一个事务占用 |
| 持有并等待 | 事务持有锁的同时等待其他锁 |
| 不可剥夺  | 锁不能被强制剥夺      |
| 循环等待  | 事务间形成锁等待环     |

### 经典死锁场景

```sql
-- 场景：两个事务互相等待对方持有的锁

-- 事务A                          -- 事务B
BEGIN;                            BEGIN;
UPDATE user SET age=20            UPDATE order SET status=1
WHERE id=1;    -- 锁住 user(id=1)   WHERE id=1;  -- 锁住 order(id=1)

UPDATE order SET status=1         UPDATE user SET age=20
WHERE id=1;    -- 等待B释放order     WHERE id=1;  -- 等待A释放user
                                   ↓
                              【死锁！互相等待】
```

### 排查死锁的方法

```sql
-- 1. 查看最近一次死锁信息（最常用）
SHOW ENGINE INNODB STATUS\G
-- 重点关注 "LATEST DETECTED DEADLOCK" 部分

-- 2. 查看当前正在进行的锁
SELECT * FROM information_schema.INNODB_LOCKS;

-- 3. 查看当前等待锁的事务
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 4. 查看当前运行的事务
SELECT * FROM information_schema.INNODB_TRX;
```

### 解决死锁的方法

| 方法                                  | 说明                           |
| ----------------------------------- | ---------------------------- |
| **kill 阻塞的事务**                      | `KILL <trx_mysql_thread_id>` |
| **统一加锁顺序**                          | 多个事务按相同顺序访问资源                |
| **减小事务粒度**                          | 缩短事务时间，减少锁持有时间               |
| **添加合适的索引**                         | 避免不走索引导致表锁                   |
| **使用 `SELECT ... FOR UPDATE` 主动加锁** | 提前锁定需要访问的资源                  |
| **降低隔离级别**                          | 使用读已提交，减少间隙锁                 |
| **重试机制**                            | 应用层捕获死锁异常后重试                 |


**面试回答模板**：
> "死锁是指两个或多个事务互相持有对方需要的锁，形成循环等待。
>
> 排查死锁时，我会执行 `SHOW ENGINE INNODB STATUS` 查看最近的死锁信息，找到具体 SQL 和事务 ID。
>
> 解决死锁的方法包括：
> 1. 统一多个事务访问资源的顺序，比如都先更新 user 表再更新 order 表
> 2. 尽量缩短事务，减少锁持有时间
> 3. 为 WHERE 条件字段加索引，避免表锁
> 4. 如果业务允许，降低隔离级别到读已提交
> 5. 应用层做重试，死锁后自动重试事务
>
> MySQL 也会自动检测死锁，并回滚代价较小的事务。"
## 第 7 题：乐观锁和悲观锁的区别？（必问）

**要点**：

| 对比项        | 乐观锁          | 悲观锁                     |
| ---------- | ------------ | ----------------------- |
| **核心思想**   | 假设不会冲突，更新时检查 | 假设一定会冲突，先加锁             |
| **实现方式**   | 版本号、时间戳      | `SELECT ... FOR UPDATE` |
| **适用场景**   | 读多写少         | 写多读少                    |
| **并发性能**   | 高（不加锁）       | 低（加锁阻塞）                 |
| **是否数据库锁** | 否（应用层实现）     | 是（数据库行锁）                |


### 乐观锁实现示例

```sql
-- 表结构：增加 version 字段
CREATE TABLE product (
    id INT PRIMARY KEY,
    stock INT,
    version INT
);

-- 查询商品
SELECT id, stock, version FROM product WHERE id = 1;
-- 结果：stock=10, version=1

-- 更新时检查 version
UPDATE product 
SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 1;  -- 版本号匹配才更新

-- 影响行数 = 0 表示版本已变，重试
```

### 悲观锁实现示例

```sql
-- 先加锁，再更新
BEGIN;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 加 X 锁
-- 判断 stock 是否充足
UPDATE product SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

**面试回答模板**：
> "乐观锁和悲观锁是两种不同的并发控制策略。
>
> **乐观锁**假设冲突很少发生，不加锁，只在更新时检查版本号是否被修改。适合读多写少的场景，比如文章点赞数更新。用版本号或时间戳实现。
>
> **悲观锁**假设一定会发生冲突，操作前先加锁阻塞其他事务。适合写多读少的场景，比如库存扣减。用 `SELECT ... FOR UPDATE` 实现。
>
> 库存扣减通常用乐观锁，因为并发高且大部分情况不会冲突；如果冲突严重，比如秒杀，可以用悲观锁配合 Redis 做兜底。"
## 附加题：锁升级和锁降级

### 锁升级

> InnoDB 会**主动升级锁**：从一条记录的范围扩大到更大范围。

```sql
-- 查询范围太大，可能从行锁升级为表锁
SELECT * FROM user WHERE id > 10000 FOR UPDATE;  -- 可能升表锁
```

### 锁降级

> InnoDB **不支持锁降级**。比如持有 X 锁（写锁）的事务，不能降级为 S 锁（读锁）。

# **日志系统**

## 日志系统知识地图

```
日志系统
├── Redo Log（重做日志）—— 崩溃恢复
├── Binlog（归档日志）—— 主从复制、数据恢复
├── Undo Log（回滚日志）—— 事务回滚、MVCC
├── Relay Log（中继日志）—— 主从复制中间件
└── 两阶段提交 —— Redo Log + Binlog 一致性
```
## 第 1 题：MySQL 有哪些主要的日志？各自的作用是什么？（必问）

**要点**：

| 日志类型          | 所属引擎     | 作用            | 关键特点              |
| ------------- | -------- | ------------- | ----------------- |
| **Redo Log**  | InnoDB   | 崩溃恢复，保证事务持久性  | 物理日志，循环写，固定大小     |
| **Binlog**    | Server 层 | 主从复制、数据恢复     | 逻辑日志，追加写，可归档      |
| **Undo Log**  | InnoDB   | 事务回滚、MVCC 快照读 | 逻辑日志，记录反向操作       |
| **Relay Log** | Server 层 | 主从复制的中继       | 从库接收主库 Binlog 后写入 |


**面试回答模板**：
> MySQL 主要有三种核心日志：
> 1. **Redo Log**（重做日志）：InnoDB 特有，记录数据页的物理修改，用于崩溃恢复，保证事务持久性。它是循环写入的，空间固定。
> 2. **Binlog**（归档日志）：Server 层，记录所有 DDL 和 DML 语句的逻辑日志，用于主从复制和数据恢复。它是追加写入的，可以归档。
> 3. **Undo Log**（回滚日志）：InnoDB 特有，记录事务修改前的旧值，用于事务回滚和 MVCC 快照读。
>
> 另外主从复制中还有 Relay Log（中继日志），存在从库上。

## 第 2 题：Redo Log 和 Binlog 的区别是什么？（超高频）

**要点**：

| 对比项 | Redo Log | Binlog |
|--------|----------|--------|
| **所属层** | InnoDB 引擎层 | MySQL Server 层 |
| **日志类型** | 物理日志（"在某个数据页的某个偏移量，把值改成 X"） | 逻辑日志（"把 id=1 的 name 改成 '张三'"） |
| **写入方式** | 循环写，空间固定，写满会覆盖 | 追加写，写满会新建文件 |
| **作用** | 崩溃恢复（crash-safe） | 主从复制、基于时间点的恢复 |
| **记录时机** | 事务执行过程中持续写 | 事务提交时写 |

**图解 Redo Log 循环写**：

```text
Redo Log 文件组（固定大小，比如 4 个文件，每个 1GB）
┌─────────────────────────────────────────────────────────────────┐
│  file1   │  file2   │  file3   │  file4   │                     │
│  (已写)   │  (已写)   │  (当前写) │  (空闲)  │                     │
└─────────────────────────────────────────────────────────────────┘
      ↑                              ↑
   write pos                    checkpoint
   （当前写的位置）                （已刷盘的位置）

写入规则：
- 写入：write pos 向前推进
- 刷盘：checkpoint 向前推进
- 空间：write pos 追到 checkpoint 时，需要停写刷盘
```

**图解 Binlog 追加写**：

```text
Binlog 文件（不断新增）
┌─────────────────────────────────────────────────────────────────┐
│ binlog.000001 │ binlog.000002 │ binlog.000003 │ ...            │
│   (已写满)     │   (已写满)     │   (当前写)    │                 │
└─────────────────────────────────────────────────────────────────┘
                                      ↑
                                 追加写入，永不覆盖
```

**面试回答模板**：
> "Redo Log 和 Binlog 有三个主要区别：
> 1. **所属层级不同**：Redo Log 是 InnoDB 引擎层的，Binlog 是 Server 层的，所以 MyISAM 只有 Binlog 没有 Redo Log
> 2. **日志类型不同**：Redo Log 是物理日志，记录的是数据页的物理修改；Binlog 是逻辑日志，记录的是 SQL 语句或行变更
> 3. **写入方式不同**：Redo Log 是循环写，空间固定，写满会覆盖；Binlog 是追加写，写满会新建文件，可以归档
>
> 作用也不同：Redo Log 用于崩溃恢复，Binlog 用于主从复制和基于时间点的数据恢复。"

## 第 3 题：什么是两阶段提交？为什么需要它？（超高频，重点）

**要点**：

**问题背景**：Redo Log 和 Binlog 是两个独立的日志，如果不一致，会导致主从数据不一致或崩溃恢复后数据丢失。

**两阶段提交**：将 Redo Log 的写入拆成两个阶段：prepare 和 commit，中间写入 Binlog。

### 流程图解

```text
事务执行过程
─────────────────────────────────────────────────────────────────→

   ①执行SQL               ②写Redo Log        ③写Binlog        ④提交事务
      │                      (prepare)           │               │
      ▼                      ▼                   ▼               ▼
┌─────────┐            ┌─────────────┐      ┌─────────┐     ┌─────────┐
│ 业务SQL │            │ Redo Log    │      │ Binlog  │     │ Redo Log│
│ 执行    │────────────│ 状态:prepare│─────→│ 写入    │────→│ 状态:   │
└─────────┘            └─────────────┘      └─────────┘     │ commit  │
                                                            └─────────┘

                         └─────────── 两阶段提交 ────────────┘
```

### 崩溃恢复场景分析

| 崩溃时机                   | Redo Log 状态 | Binlog 状态 | 恢复策略 | 结果       |
| ---------------------- | ----------- | --------- | ---- | -------- |
| ① 写 Redo Log prepare 前 | 无           | 无         | 事务回滚 | 数据不丢失 ✅  |
| ② prepare 后、写 Binlog 前 | prepare     | 无         | 回滚事务 | 主从不一致被避免 |
| ③ 写 Binlog 后、commit 前  | prepare     | 已写        | 提交事务 | 数据一致 ✅   |
| ④ commit 后             | commit      | 已写        | 正常   | 数据一致 ✅   |

**关键规则**：
> 崩溃恢复时，如果 Redo Log 是 prepare 状态，就去查 Binlog：如果 Binlog 完整就提交，否则回滚。

**面试回答模板**：
> "两阶段提交是为了保证 Redo Log 和 Binlog 的一致性。因为这两个日志是独立写入的，如果只写其中一个就崩溃，会导致主从不一致或数据丢失。
>
> 两阶段提交把 Redo Log 的写入分成两步：
> 1. **prepare 阶段**：先写 Redo Log 并标记为 prepare
> 2. **commit 阶段**：写完 Binlog 后，再把 Redo Log 标记为 commit
>
> 崩溃恢复时，如果 Redo Log 是 prepare 状态，就去检查 Binlog 是否完整：完整则提交，不完整则回滚。这样就保证了两个日志的逻辑一致性。"
## 第 4 题：Redo Log 是怎么保证崩溃恢复的？（追问）

**要点**：

**Redo Log 的核心机制**：WAL（Write-Ahead Logging，预写日志）

> **WAL 原则**：修改数据前，必须先写日志。

### 数据更新流程

```text
用户执行 UPDATE user SET name = '李四' WHERE id = 1

                    ┌─────────────────────────────────────────┐
                    │            Buffer Pool                 │
                    │  ┌─────────────────────────────────┐   │
                    │  │ 数据页（旧数据: '张三'）           │   │
                    │  └─────────────────────────────────┘   │
                    └─────────────────────────────────────────┘
                           │                    │
         ③ 标记为脏页       │                    │ ① 先读入内存
         等待后台刷盘       │                    │
                           ▼                    ▼
┌─────────────────────────────────────────────────────────────┐
│                         Redo Log                             │
│  ② 立即写入：把"修改操作"记入 Redo Log（顺序写，很快）         │
│     "在 page 1 offset 100 处，把 '张三' 改成 '李四'"          │
└─────────────────────────────────────────────────────────────┘
```

### 崩溃恢复过程

```text
MySQL 崩溃重启
      │
      ▼
┌─────────────────────────────────────────────────────────────┐
│ 1. 读取 Redo Log，找到所有已 prepare/commit 但未刷盘的修改    │
│ 2. 把这些修改重新应用到数据页（重做）                          │
│ 3. 数据恢复到崩溃前的状态，已提交的事务不会丢失                │
└─────────────────────────────────────────────────────────────┘
```

**关键参数**：`innodb_flush_log_at_trx_commit`

| 值 | 含义 | 性能 | 安全性 |
|----|------|------|--------|
| 0 | 每秒写一次 Redo Log 并刷盘 | 最高 | 可能丢失 1 秒数据 |
| 1 | 每次事务提交都写并刷盘（默认） | 最低 | 最安全 |
| 2 | 每次事务提交都写，每秒刷盘 | 中等 | 操作系统崩溃可能丢数据 |

**面试回答模板**：
> "Redo Log 通过 WAL（预写日志）机制保证崩溃恢复。修改数据时，先写 Redo Log（顺序写，很快），再更新内存中的数据页，后台线程再异步刷盘。
>
> 如果 MySQL 崩溃重启，会读取 Redo Log，把已写入日志但未刷盘的修改重新执行一遍，保证已提交的事务不丢失。这就是 Redo Log 的'重做'能力，也是 Redo 这个名字的由来。"

## 第 5 题：Undo Log 是用来做什么的？和 MVCC 什么关系？（高频）

**要点**：

### Undo Log 的两个作用

| 作用           | 说明                                   |
| ------------ | ------------------------------------ |
| **事务回滚**     | 事务执行过程中如果出错或主动回滚，用 Undo Log 里的旧值恢复数据 |
| **MVCC 快照读** | 读事务需要读取历史版本时，通过 Undo Log 找到旧版本数据     |

### Undo Log 版本链

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Undo Log 版本链                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   当前数据页（最新版本）                                          │
│   ┌─────────────────────────┐                                   │
│   │ id=1, name='李四'        │                                   │
│   │ DB_TRX_ID=102           │                                   │
│   │ DB_ROLL_PTR ──────────────────┐                              │
│   └─────────────────────────┘      │                              │
│                                     ▼                             │
│                          ┌─────────────────────────┐             │
│                          │ Undo Log 版本1           │             │
│                          │ id=1, name='张三'        │             │
│                          │ DB_TRX_ID=101           │             │
│                          │ DB_ROLL_PTR ──────────────────┐         │
│                          └─────────────────────────┘      │         │
│                                                           ▼         │
│                                                ┌─────────────────┐  │
│                                                │ Undo Log 版本2   │  │
│                                                │ id=1, name='王五'│  │
│                                                │ DB_TRX_ID=100   │  │
│                                                └─────────────────┘  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**和 MVCC 的关系**：
- Read View 决定了当前事务应该看到哪个版本的数据
- Undo Log 提供了历史版本的数据
- 两者配合实现 MVCC

**面试回答模板**：
> "Undo Log 有两个作用：一是事务回滚，二是 MVCC 快照读。
>
> 当事务修改一行数据时，InnoDB 会先把修改前的旧值写入 Undo Log，这样如果事务需要回滚，就能用 Undo Log 恢复到修改前的状态。
>
> 对于 MVCC，当 Read View 判断当前行版本不可见时，就会通过 DB_ROLL_PTR 指针沿着 Undo Log 的版本链往下找，直到找到可见的版本。这就是 InnoDB 能实现无锁快照读的原因。"

## 第 6 题：Binlog 有哪几种格式？有什么区别？（高频）

**要点**：

Binlog 有三种格式：

| 格式            | 说明                           | 优点       | 缺点                           |
| ------------- | ---------------------------- | -------- | ---------------------------- |
| **STATEMENT** | 记录原始 SQL 语句                  | 日志量小     | 可能有主从不一致（如 `NOW()`、`UUID()`） |
| **ROW**       | 记录每行数据的变更前后值                 | 最安全，主从一致 | 日志量大                         |
| **MIXED**     | 自动切换，默认用 STATEMENT，不安全时用 ROW | 兼顾       | 复杂                           |

```sql
-- 查看当前格式
SHOW VARIABLES LIKE 'binlog_format';

-- 设置格式
SET GLOBAL binlog_format = 'ROW';
```

**STATEMENT 的不安全问题示例**：

```sql
-- 主库执行
DELETE FROM user WHERE name = '张三' LIMIT 5;
-- 从库执行同样的 SQL，如果数据不同，删除的行可能不一致

-- 使用 ROW 格式
-- 直接记录哪些行被删了，主从结果完全一致
```

**面试回答模板**：
> "Binlog 有三种格式：
> 1. **STATEMENT**：记录原始 SQL 语句，日志量小，但包含 `NOW()`、`UUID()` 这类不确定函数的 SQL 会导致主从不一致
> 2. **ROW**：记录每行数据的变更前后值，最安全，主从完全一致，但日志量大
> 3. **MIXED**：默认用 STATEMENT，遇到不安全 SQL 时自动切换成 ROW
>
> 生产环境推荐使用 **ROW 格式**，虽然日志量大，但保证了主从复制的数据一致性。MySQL 8.0 默认就是 ROW 格式。"
