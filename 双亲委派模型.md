
在 Java 的面试和高级进阶中，双亲委派机制（Parents Delegation Mechanism）**和**“打破双亲委派”绝对是绕不开的硬核大山。

如果用大白话来形容，双亲委派机制就是 JVM 内部的“逐级请示审批制度”**，而打破它，则是为了在特定场景下进行**“越级特批”。

## 一、 什么是双亲委派机制？

Java 中的类并不是一次性全部加载到内存的，而是由 **类加载器（ClassLoader）** 按需加载。JVM 默认自带了三个老员工（类加载器），它们之间存在着明确的“上下级”树状关系：

- **Bootstrap ClassLoader（启动类加载器）：** 最高层级的 CEO。由 C++ 编写，负责加载 Java 最核心的类库（如 `rt.jar`、`java.lang.*`）。
    
- **Extension ClassLoader / Platform ClassLoader（扩展/平台类加载器）：** 核心高管。负责加载 Java 的一些扩展类库（`ext` 目录下的类）。
    
- **Application ClassLoader（系统/应用类加载器）：** 基层主管。负责加载你在项目中自己写的代码，以及 Maven 引入的第三方 Jar 包。
    
- **Custom ClassLoader（自定义类加载器）：** 你自己派出的办事员。
    

### 核心工作流程：逐级请示

当一个类加载器收到“加载某个类（比如 `User` 类）”的任务时，它绝对不会自己先去加载，而是像一个**极度严谨（推卸责任）的职场打工人**，遵循以下流程：

1. **向上请示：** 办事员问主管：“老大，你能加载这个类吗？”主管说：“等等，我问问高管。”高管又问 CEO：“大佬，这个类您能加载吗？”
    
2. **向下分派：** * CEO 翻了翻自己的核心类库，说：“这属于业务类，我不认识，你们自己搞定。”（抛出 `ClassNotFoundException`）。
    
    - 高管一看，自己也不认识，继续往下推。
        
    - 最后任务落回基层主管（Application）手里，基层主管在你的项目里找到了 `User.class`，成功加载。
        

> **为什么要这么麻烦？（双亲委派的好处）**
> 
> 1. **安全性（防止核心类被篡改）：** 假设有个黑客在项目中写了一个恶意类 `java.lang.String` 企图窃取密码。由于双亲委派机制，这个加载请求一路传到 CEO（Bootstrap）那里，CEO 一看：“这个类我早就加载过了！”直接返回官方的 String。黑客的恶意代码永远没机会被执行。
>     
> 2. **避免重复加载：** 保证了在整个 JVM 生命周期中，核心类只会被加载一次，全网唯一。
>     

## 二、 为什么要打破双亲委派？有什么好处？

既然这个机制这么完美，为什么还要去打破它？因为在很多复杂的工程实践中，“一夫多妻”和“版本冲突”是不可避免的。打破它的核心好处，主要体现在以下两个经典战场：

### 1. 战场一：Tomcat 等 Web 容器（实现“应用隔离”）

- **痛点：** 假设你的 Tomcat 服务器上同时运行着两个 Web 应用。应用 A 依赖的是 **Spring 4.0**，应用 B 依赖的是 **Spring 5.0**。
    
- **如果遵循双亲委派：** 它们都会把加载 Spring 核心类的请求交给最顶层的 Application ClassLoader。基层主管一看，先加载了 Spring 4.0 的类，当应用 B 启动时，主管直接返回了已经加载好的 Spring 4.0 的类——**结果应用 B 因为版本不兼容直接当场崩溃！**
    
- **打破的好处：** Tomcat 自定义了 `WebappClassLoader`。当收到加载请求时，它**不往上请示**，而是先在自己应用的目录下找。这样应用 A 和应用 B 就能各自加载属于自己的 Spring 版本，互不干扰，实现了完美的**应用隔离**。
    

### 2. 战场二：JDBC 等 SPI 技术（解决“长辈无法指挥晚辈”的尴尬）

- **痛点：** Java 官方定义了一套数据库驱动的标准接口（SPI），比如 `java.sql.Driver`，这个接口在核心包里，是由最高级别的 CEO（Bootstrap）加载的。但是，具体的实现类（如 MySQL 驱动、Oracle 驱动）是第三方厂商写的，放在 Classpath 里，属于基层主管（Application）管辖。
    
- **如果遵循双亲委派：** CEO（Bootstrap）在运行核心代码时，需要实例化具体的 MySQL 驱动。但由于双亲委派只能“从下往上请示”，**上层加载器默认是看不到下层加载器管理的类的**（CEO 怎么能随意去底层的工位翻东西呢？）。
    
- **打破的好处：** Java 引入了**线程上下文类加载器（Thread Context ClassLoader）**。CEO 通过这个通道，直接向下“借用”基层主管（Application）去加载 MySQL 驱动，完美的解决了解耦和倒挂调用的问题。
    

## 三、 怎么打破双亲委派机制？

在代码层面，双亲委派的逻辑其实全部写在 `java.lang.ClassLoader` 的 `loadClass()` 方法里。

### 💡 核心规则：

- 如果你只想**扩展**类加载器（不打破委派）：只需要重写 **`findClass(String name)`** 方法。
    
- 如果你想要**打破**双亲委派：你必须重写 **`loadClass(String name, boolean resolve)`** 方法，把里面“先找爸爸”的逻辑直接删掉。
    

### 🛠️ 实战：Tomcat 风格的“双亲委派打破者”代码

下面是一个简易的自定义类加载器，它会**优先加载自己指定目录下的类**，找不到才向上请示：

``` java
public class MyBrokenClassLoader extends ClassLoader {
    private String classPath;

    public MyBrokenClassLoader(String classPath) {
        this.classPath = classPath;
    }

    // 重写 loadClass，强行扭转双亲委派的“请示流程”
    @Override
    public Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            // 1. 首先，检查该类是否已经被加载过了
            Class<?> c = findLoadedClass(name);
            
            if (c == null) {
                // 核心防线：如果是 java 核心类库（如 java.lang.String），绝对不能打破！
                // 否则 JVM 会直接抛出 SecurityException 安全异常
                if (name.startsWith("java.")) {
                    c = getSystemClassLoader().loadClass(name);
                }
            }

            if (c == null) {
                try {
                    // 2. 叛逆的一步：不找爸爸！自己先尝试去本地路径加载
                    c = findClass(name);
                } catch (ClassNotFoundException e) {
                    // 自己找不到，没关系，再认贼作父，让父类加载器去兜底
                }
            }

            if (c == null) {
                // 3. 如果自己和核心库都搞不定，才走传统的双亲委派残余逻辑
                c = super.loadClass(name, resolve);
            }
            
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 自定义读取本地磁盘字节码文件（.class）并转换为 Class 对象的逻辑
        byte[] data = loadClassData(name); 
        return defineClass(name, data, 0, data.length);
    }
}
```

### 总结大局观

双亲委派机制是 Java 为了江山稳固（安全性）而设计的标准官僚体制。但在遇到 **Web 应用多版本共存（Tomcat）** 或 **标准接口调用底层实现（JDBC SPI）** 时，这种死板的体制就成了阻碍。通过覆写 `loadClass` 或利用 `ContextClassLoader`，Java 赋予了开发者在规则之外进行“特事特办”的自由。
