# 概述

## 一、 为什么叫“新一代”网关？（历史演进）

在旧版的 Spring Cloud 生态中，默认的网关组件是 Netflix 公司的 **Zuul 1.x**。但后来 Spring 官方毅然决然地抛弃了 Zuul，自己动手写了 **Spring Cloud Gateway**，并冠以“新一代”的名号。

它们之间有着本质的技术代差：

| **特性**   | **老一代 Zuul 1.x**                       | **新一代 Spring Cloud Gateway**             |
| -------- | -------------------------------------- | ---------------------------------------- |
| **底层架构** | 基于 **Servlet 2.5** 阻塞 I/O              | 基于 **Spring WebFlux + Netty** 非阻塞 I/O    |
| **线程模型** | **一请求一线程**。一个请求卡住，整个线程就死等，高并发下极易耗尽线程池。 | **反应式异步线程模型**。用极少数的线程就能应对海量的并发连接，性能高出数倍。 |
| **动态路由** | 原生支持较弱，配置繁琐。                           | 天生支持强大的动态路由、断言和过滤器配置。                    |
| **官方血统** | Netflix 停止维护的边缘产品。                     | Spring 亲儿子，与 Spring Cloud 生态完美融合。        |

## 二、 网关的核心使命：它需要解决哪些问题？

如果让你裸奔微服务（不用网关），前端就必须记住每一个微服务的 IP 和端口，这简直是灾难。网关的出现，统一了微服务的入口，并顺手解决了以下核心问题：

1. **统一路由（Routing）：** 外部请求只需要访问网关的唯一地址（如 `api.company.com`）。网关根据请求的路径，动态地把流量转发给背后不同的微服务。
    
2. **统一鉴权（Authentication）：** 像 Token 校验、用户登录状态检查等安全工作，没必要在每个微服务里都写一遍。网关在最外层直接统一拦截，不合法的请求直接轰走。
    
3. **安全防护与限流（Rate Limiting）：** 防止恶意刷接口、DDoS 攻击。网关可以限制每个 IP 每秒只能请求多少次，保护内部微服务不被打垮。
    
4. **日志与监控（Metrics）：** 统一记录所有进出系统的流量日志，监控接口的整体响应时间和错误率。

## 三、 掌握 Gateway 的三大核心支柱

学习 Spring Cloud Gateway，本质上就是学习三个词：**Route（路由）**、**Predicate（断言）**、**Filter（过滤器）**。这三者织成了网关的整张过滤网。

### 1. Route（路由）

- **含义：** 路由是网关的最基本组织单元。它由一个 **ID**、一个**目标 URI**、一组断言（Predicates）**和一组**过滤器（Filters）组成。
    
- **通俗解释：** 它定义了“如果请求满足什么条件，就把它送到哪里去”。
    

### 2. Predicate（断言 / 谓词）

- **含义：** 借用了 Java 8 的 `Predicate` 概念。它是一个**匹配规则**。
    
- **通俗解释：** 它是网关的“判官”。它去检查请求的各种细节（比如请求时间是不是在12点后？Header里有没有带Token？请求路径是不是`/order/`？）。如果检查通过，说明路由匹配成功。
    

### 3. Filter（过滤器）

- **含义：** 在请求被转发给真正的微服务**之前（Pre）**或者**之后（Post）**，对请求或响应进行修改的组件。
    
- **通俗解释：** 它是流水线上的“工人”。可以在请求飞过去之前，帮你偷偷加个请求头、改个参数；或者在微服务返回结果后，帮你在响应头里加个 Cookie。
    

## 四、 Gateway 的核心工作流程

当一个请求从外部敲响网关的大门时，它会经历以下丝滑的流转步骤：

1. **Gateway Client** 向网关发起 HTTP 请求。
    
2. **Gateway Handler Mapping** 开始干活：它去配置文件里翻看所有的 **Route（路由）**，并把请求交给各个路由的 **Predicate（断言）** 挨个匹配。
    
3. 一旦某一条路由匹配中了，**Gateway Web Handler** 就会把这个请求接管过来。
    
4. 请求开始穿过一条由无数个 **Filter（过滤器）** 组成的“过滤器链”。
    
    - 先执行所有过滤器的 **Pre** 逻辑（如：检查 Token 是否有效，往请求头里塞入用户 ID）。
        
5. 过滤器放行，请求真正到达背后的**微服务实例**（利用 LoadBalancer 动态选择 IP）。
    
6. 微服务处理完业务，把响应结果扔回来。
    
7. 请求再次穿过过滤器链，执行所有过滤器的 **Post** 逻辑（如：记录这次调用的耗时日志）。
    
8. 最终，网关把清爽的响应结果吐回给前端用户。


# Gateway三大核心

如果把 **Spring Cloud Gateway** 比作一座戒备森严的“海关大楼”，那么支撑这座大楼正常运转的，就是它的**三大核心支柱：Route（路由）、Predicate（断言）和 Filter（过滤器）**。

这三者不是孤立存在的，而是层层嵌套、环环相扣的。一句话总结它们的关系：**路由（Route）是一个大容器，里面装着“目的地（URI）”、“谁能通过（Predicate）”以及“怎么通关（Filter）”。**

## 一、 Core 1：Route（路由）—— 流量的导航地图

### 1. 什么是路由？

路由是网关的**最基本组织单元**。外部请求打到网关时，网关必须决定这个请求接下来要去哪台机器、哪个微服务。这个“导航”的规则就是路由。

### 2. 一个完整的路由包含什么？

在代码或配置文件中，一个路由由以下四部分组成：

- **ID：** 路由的唯一标识（就像身份证号，不能重复，通常起名为 `order-service-route` 这种可读性高的名字）。
    
- **URI（Destination URI）：** 目标地址。外部请求匹配成功后，流量要转发到哪里去。在微服务中，我们通常写 `lb://service-name`（例如 `lb://order-service`），表示结合 LoadBalancer 进行动态负载均衡转发。
    
- **Predicates（断言集合）：** 匹配条件，决定什么样的请求能走这条路由。
    
- **Filters（过滤器集合）：** 加工工序，对匹配成功的请求进行修改。
    

## 二、 Core 2：Predicate（断言）—— 铁面无私的判官

### 1. 什么是断言？

断言借用了 Java 8 中 `Predicate` 的概念。简单来说，它就是一堆 **`if-else` 条件判断**。

当请求到达网关时，断言去检查这个请求的各个细节（如路径、时间、Header、Cookie、IP 等），如果满足条件就返回 `true`（放行到下一关），不满足就返回 `false`（对不起，你走错路了）。

### 2. 主流的内置断言工厂（Route Predicate Factories）

Spring Cloud Gateway 贴心地提供了 10 多种内置断言，涵盖了 HTTP 请求的方方面面。以下是实战中最常用的几种：

- **Path 断言（最常用）：** 根据请求路径匹配。
    
    ``` yaml
    predicates:
      - Path=/api/order/** # 只要请求路径是以 /api/order/ 开头的，就满足条件
    ```
    
- **Method 断言：** 根据 HTTP 请求方式（GET/POST/PUT）匹配。
    
    ``` yaml
    predicates:
      - Method=GET,POST # 只有 GET 或 POST 请求才能通过
    ```
    
- **After/Before/Between 断言（定时炸弹）：** 根据请求发生的时间匹配。
    
    ``` yaml
    predicates:
      - After=2026-05-31T23:59:59.000+08:00[Asia/Shanghai] # 只有在2026年5月31日午夜之后，这条路由才生效（常用于秒杀活动准时开启）
    ```
    
- **Header 断言：** 检查请求头里有没有某个属性，或者属性值是否符合正则表达式。
    
    ``` yaml
    predicates:
      - Header=X-Request-Id, \d+ # 请求头必须包含 X-Request-Id，且值必须是纯数字
    ```

## 三、 Core 3：Filter（过滤器）—— 流水线上的加工工人

### 1. 什么是过滤器？

当断言考核通过，请求被确认要发给某个微服务之前或之后，**Filter（过滤器）** 就会出手。它可以在流量传输的生命周期中，对请求（Request）或响应（Response）进行任意的修改。

### 2. 过滤器的两个生命周期（Lifecycle）

- **PRE（前置阶段）：** 在请求**发给目标微服务之前**执行。
    
    > 💡 **实战场景：** 统一登录校验、Token 解析、安全黑名单拦截、往请求头里塞入当前登录用户的 `userId`。
    
- **POST（后置阶段）：** 在目标微服务执行完毕、**响应返回给前端之前**执行。
    
    > 💡 **实战场景：** 统一计算接口耗时、统一在响应头里加上跨域支持（CORS Header）、修改响应体的格式。
    

### 3. 过滤器的两个作用域（Scope）

根据生效范围的不同，过滤器分为两类：

- **GatewayFilter（局部过滤器）：** 只对**某一条特定路由**生效。需要在 YAML 的具体 route 下配置。
    
- **GlobalFilter（全局过滤器）：** 极其霸道，不需要配置，**对所有路由、所有进出网关的请求全部生效**。通常用来写全链路的鉴权日志。

## 四、 三大核心如何珠联璧合？（代码实战演示）

我们把这三大核心装进 `application.yml` 配置文件里，看看它们是如何协同作战的：

``` yaml
spring:
  cloud:
    gateway:
      routes:
        # ==========================================
        # 1. 这是一个完整的【Route (路由)】，ID 叫 order_route
        # ==========================================
        - id: order_route
          # 流量最终的目的地：动态转发到订单微服务集群
          uri: lb://order-service
          
          # ==========================================
          # 2. 这里的【Predicates (断言)】是门禁
          # ==========================================
          predicates:
            # 条件一：路径必须是 /order/ 开头
            - Path=/order/**
            # 条件二：必须是 GET 请求
            - Method=GET
            
          # ==========================================
          # 3. 这里的【Filters (过滤器)】是加工流水线
          # ==========================================
          filters:
            # 动作一：在请求发给 order-service 之前，悄悄往请求头里加个参数
            - AddRequestHeader=X-Gateway-From, NettyGateway
            # 动作二：剥离路径。把请求路径里的第一层删掉（比如把 /order/list 变成 /list 发给后端）
            - StripPrefix=1
```

### 💡 绝佳的现实比喻：去游乐场玩过山车

- **Route（路由）：** 就是游乐场里的“过山车项目（Order_Project）”，它有明确的终点（uri）。
    
- **Predicate（断言）：** 就是项目入口处的“检票员与身高测量尺”。他去检查你：身高是否超过 $1.4\text{m}$？（Path判断）、有没有买票？（Header判断）。只有全部满足（返回 true），你才能进去。
    
- **Filter（过滤器）：** 就是你检票进去后的“工作人员”。
    
    - **PRE 过滤器：** 在你坐上车出发前，工作人员强行塞给你一把雨伞，并帮你扣紧安全带（修改请求）。
        
    - **POST 过滤器：** 当过山车到终点你准备离场时，工作人员在出口处递给你一张刚才抓拍的抓拍照片（修改响应）。

# GateWay工作流程


掌握了 **Route（路由）**、**Predicate（断言）** 和 **Filter（过滤器）** 这三大核心组件后，我们就能把它们连成线，完整地走一遍 **Spring Cloud Gateway 的核心工作流程**了。

网关的底层架构是基于 **Spring WebFlux** 和 **Netty** 的异步非阻塞模型，它的请求处理流程非常严密。我们可以将其生动地比喻为“机场国际安检与出入境通道”。

## 二、 核心流转步骤：一个请求的“通关”之旅

当客户端发出了一个类似 `http://localhost:8080/order/list` 的请求，网关内部会依次惊动以下几大核心组件：

### 1. 流量入口：Gateway Client

外部客户端（如手机 App 或前端浏览器）向网关发起 HTTP 请求。网关的底层 Netty 服务器接收到网络连接。

### 2. 第一关：Gateway Handler Mapping（寻路/对号入座）

- **职责：** 它是网关的“导视牌”。
    
- **动作：** 它会拿着这个请求，去配置文件里比对所有写好的 **Route（路由）配置**。它会调用路由里的 **Predicate（断言）** 逐一进行判断。
    
- **结果：** “噢！你的路径是 `/order/list`，正好对上了我这里 `id: order_route` 的门禁条件！” 匹配成功后，它会将请求转发给对应的 `Gateway Web Handler`。

### 3. 第二关：Gateway Web Handler（组装流水线）

- **职责：** 它是网关的“总调度室”。
    
- **动作：** 既然知道了要去哪个微服务，它就会把**专属于这条路由的局部过滤器（GatewayFilter）** 和**全网通用的全局过滤器（GlobalFilter）** 全部召集过来，按照配置的**优先级（Order）** 进行排序，组装成一条**过滤器链（Filter Chain）**。

### 4. 第三关：Filter Chain - PRE 阶段（安检与加工）

请求开始顺序穿过过滤器链。此时执行的是过滤器的 **PRE（前置）逻辑**。

- **过滤器 A (安全鉴权)：** 检查你的 Header 里有没有带 Token，没有就直接轰走（返回 401）。
    
- **过滤器 B (限流防护)：** 检查你的 IP 访问频率是否超标。
    
- **过滤器 C (路径裁剪)：** 执行 `- StripPrefix=1`，把 `/order/list` 咔嚓剪成 `/list`。

### 5. 第四关：HttpWebHandlerAdapter / Proxied Service（核心转发）

通过了所有的 PRE 过滤器后，请求终于来到了网关的边界。 网关会利用 **Spring Cloud LoadBalancer** 动态决定具体的微服务物理 IP（例如 `192.168.1.11:8081`），并使用高性能的底层 HTTP 客户端将裁剪后的纯净请求（`/list`）正式发送给后端的微服务。

### 6. 第五关：Filter Chain - POST 阶段（回执与包装）

后端的微服务业务处理完毕，把结果（如 JSON 数据）返回给网关。 请求原路返回，但这次触发的是过滤器链的 **POST（后置）逻辑**（注意：执行顺序通常与 PRE 阶段**相反**，类似于剥洋葱，从内向外）。

- **过滤器 C (日志记录)：** 记录本次接口调用的最终耗时。
    
- **过滤器 B (跨域处理)：** 在响应头里塞入 `Access-Control-Allow-Origin: *`。

### 7. 终点：Response 返回

网关把经过全套 POST 过滤器包装洗礼后的最终响应结果，通过 Netty 吐回给前端用户。整个调用闭环完美结束。

## 三、 深度思考：这个流程对我们开发有什么价值？

很多新同学觉得“流程懂个大概就行了，能跑通不就好了嘛”。其实不然，吃透这个核心流程，能直接帮你解决以下两个高级开发痛点：

### 1. 搞懂“过滤器优先级（Order）”的排兵布阵

在编写自定义过滤器时，我们经常要指定它的 `Order` 值（值越小，优先级越高）。

- **PRE 阶段：** Order 值越小的过滤器，**越先执行**。也就是说，如果鉴权过滤器 `Order = -1`，日志过滤器 `Order = 0`，那么系统会先鉴权，再记日志。
    
- **POST 阶段：** 恰恰相反，Order 较大的过滤器反而**先执行**返回。

### 2. 完美的“全链路接口耗时统计”方案

如果你想监控微服务到底慢不慢，利用这个流程的两个阶段可以写出极其优雅的代码：

``` java
public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
    // 1. 【PRE 阶段】在请求发过去之前，记录一个开始时间戳
    exchange.getAttributes().put("startTime", System.currentTimeMillis());
    
    return chain.filter(exchange).then(Mono.fromRunnable(() -> {
        // 2. 【POST 阶段】在微服务返回结果后，捞出时间戳算差值
        Long startTime = exchange.getAttribute("startTime");
        if (startTime != null) {
            long executeTime = System.currentTimeMillis() - startTime;
            log.info("接口：{} , 耗时：{} ms", exchange.getRequest().getURI().getPath(), executeTime);
        }
    }));
}
```
