### **1. 基础概念**

**Q: Spring Security 是什么？它的核心功能有哪些？**

Spring Security 是 Spring 生态中提供**认证（Authentication）**和**授权（Authorization）**的安全框架。

**核心功能**：
- 用户认证（登录）
- 权限控制（RBAC）
- 防护常见攻击（CSRF、XSS、会话固定攻击等）
- 方法级权限控制（@PreAuthorize）
- 与 OAuth2、JWT、OAuth2 Login 集成

---

### **2. 核心原理**

**Q: Spring Security 的工作原理是什么？（FilterChain）**

Spring Security 的核心是**过滤器链（FilterChain）**。

- 通过 `DelegatingFilterProxy（委托过滤器代理）` 委托给 `FilterChainProxy（过滤链代理）`。
- 默认会添加一系列 `SecurityFilter（安全过滤器）`（约15个左右），按顺序执行。
- **最重要几个过滤器**（面试必问）：
  1. `UsernamePasswordAuthenticationFilter` —— 处理表单登录
  2. `ExceptionTranslationFilter` —— 处理认证/授权异常
  3. `FilterSecurityInterceptor` —— 最后进行权限校验
  4. `AnonymousAuthenticationFilter` —— 匿名用户处理

	DelegatingFilterProxy 是 Spring Web 提供的一个Servlet Filter，它的核心作用是：把 Servlet 容器（Tomcat/Jetty）的 Filter 机制和 Spring 的 IOC 容器（Bean 管理）连接起来。
	
	FilterChainProxy 是 Spring Security 的核心入口类，它是一个实现了 Filter 接口的代理。它的主要职责是：管理并执行 Spring Security 的整个安全过滤器链（Security Filter Chain）。
	
	UsernamePasswordAuthenticationFilter 是 Spring Security 中处理表单登录（用户名 + 密码认证） 的核心过滤器。作用：负责采集用户名和密码，封装成 Authentication 对象，然后进行认证。
	
	ExceptionTranslationFilter 是 Spring Security 中异常翻译和处理的核心过滤器。它主要处理两类异常：AuthenticationException认证异常（未登录、Token无效）AccessDeniedException授权异常（已登录但权限不足）
	
	FilterSecurityInterceptor 是 Spring Security 过滤器链中最后一个核心过滤器，也是授权（Authorization） 的最终执行者。作用：对当前用户是否有权限访问该资源进行最终校验。
		
	AnonymousAuthenticationFilter 是 Spring Security 中处理匿名用户的过滤器。它的主要作用是：当当前请求中没有认证信息（即用户未登录）时，自动创建一个匿名 Authentication 对象，放入 SecurityContextHolder 中。

``` 

请求进入 FilterChainProxy 后：

1. UsernamePasswordAuthenticationFilter  
   → 处理登录请求（/login），如果不是登录请求则直接放行

2. AnonymousAuthenticationFilter  
   → 如果用户未登录，则创建一个匿名 Authentication 对象放入 SecurityContext

3. ...（其他过滤器，如 Jwt 过滤器、RememberMe 等）

4. ExceptionTranslationFilter  
   → 负责捕获后面过滤器抛出的认证/授权异常

5. FilterSecurityInterceptor  
   → 最终进行权限校验
      ├── 有权限 → 放行到 Controller
      └── 无权限 → 抛出 AccessDeniedException（被前面第4步的 ExceptionTranslationFilter 捕获）
```


---

**Q: Authentication 和 Authorization 的区别？**

- **Authentication（认证）**：你是谁？（验证身份）→ `Authentication` 对象
- **Authorization（授权）**：你能干什么？（验证权限）→ `AuthorizationManager` / `AccessDecisionManager`

### **3. 认证流程（最高频）**

**Q: Spring Security 的登录认证流程是怎样的？**

1. 用户提交用户名密码 → `UsernamePasswordAuthenticationFilter`
2. 封装成 `UsernamePasswordAuthenticationToken`
3. 交给 `AuthenticationManager`（通常是 `ProviderManager`）
4. `DaoAuthenticationProvider` 调用 `UserDetailsService.loadUserByUsername()`
5. 比对密码（`PasswordEncoder`）
6. 成功后生成 `Authentication` 对象，放入 `SecurityContextHolder`
7. 返回成功响应

```
[ 用户发起 HTTP 请求 ]
                 │
                 ▼
==========================================================================================
 1. 【UsernamePasswordAuthenticationFilter】 (登录拦截器)
    │
    ├── 是登录请求 (/login 且为 POST) ──► 触发【核心认证子流程】 (见下方单独展开的方框)
    │                                          │
    │                                          ├── 认证成功 ──► 放入 SecurityContextHolder ──► 返回成功响应 (流程结束)
    │                                          └── 认证失败 ──► 触发 AuthenticationFailureHandler ──► 返回错误响应 (流程结束)
    │
    └── 不是登录请求 (普通业务请求) ──► 直接放行 (chain.doFilter)
                 │
                 ▼
 2. 【AnonymousAuthenticationFilter】 (匿名过滤器)
    │
    ├── ContextHolder 中已有认证信息 (已登录) ──► 毫无作为，直接放行
    └── ContextHolder 中无认证信息 (未登录) ──► 创建一个匿名"AnonymousAuthenticationToken"放入上下文 ──► 放行
                 │
                 ▼
 3. 【ExceptionTranslationFilter】 (异常守卫官)
    │   
    └── 核心逻辑：本身不处理业务，只用 try-catch 包裹后面的过滤器 ──► 默默放行，等待捕获异常
                 │
                 ▼
 4. 【FilterSecurityInterceptor】 (权限终审官)
    │   (从 SecurityContextHolder 中取出 Authentication，对比当前路由所需的权限)
    │
    ├── 权限校验通过 ──► 【放行进入 Controller 业务层】 ──► 返回业务数据 (200 OK)
    │
    └── 权限校验失败 ──► 抛出 AccessDeniedException 
                 │
                 │ (异常向上抛出)
                 ▼
 🔙 【ExceptionTranslationFilter】 成功拦截到异常！
    │
    ├── 情况 A：当前是“匿名用户”访问受保护资源 ──► 触发 AuthenticationEntryPoint ──► 提示去登录 (401 Unauthorized)
    └── 情况 B：当前是“已登录用户”但权限不够 ──► 触发 AccessDeniedHandler ──► 提示权限不足 (403 Forbidden)
==========================================================================================


  ┌────────────────────────────────────────────────────────────────────────┐
  │                   【 核心认证子流程 】 (登录时触发)                     │
  │                                                                        │
  │  [ 用户提交的用户名/密码 ]                                              │
  │            │                                                           │
  │            ▼                                                           │
  │    封装为【UsernamePasswordAuthenticationToken】 (未认证状态)          │
  │            │                                                           │
  │            ▼                                                           │
  │    提交给【AuthenticationManager】 (通常是 ProviderManager)            │
  │            │                                                           │
  │            ▼                                                           │
  │    委派给【DaoAuthenticationProvider】 (负责具体账号密码校验)          │
  │            │                                                           │
  │            ├──► 调用【UserDetailsService.loadUserByUsername()】        │
  │            │    └──► 从数据库/内存加载出【UserDetails】(包含正确密码和权限)     │
  │            │                                                           │
  │            └──► 调用【PasswordEncoder.matches()】                      │
  │                 └──► 比对用户输入的明文密码与数据库加密密码             │
  │                          │                                             │
  │                          ├── 密码错误 ──► 抛出 BadCredentialsException  │
  │                          └── 密码正确 ──► 重新封装 Token (已认证状态)    │
  └────────────────────────────────────────────────────────────────────────┘
```

**Q: SecurityContextHolder 有几种存储策略？**

- `MODE_THREADLOCAL`（默认）：线程绑定
- `MODE_INHERITABLETHREADLOCAL`：子线程可继承
- `MODE_GLOBAL`：全局静态（不推荐）

### **4. 授权（RBAC）**

**Q: 如何实现基于角色的权限控制（RBAC）？**

```java
// 方法级
@PreAuthorize("hasRole('ADMIN')")
@PreAuthorize("hasAnyAuthority('USER:ADD', 'USER:DELETE')")

// URL级（配置）
.antMatchers("/admin/**").hasRole("ADMIN")
```

### **5. 密码加密**

**Q: 常用 PasswordEncoder 有哪些？推荐使用哪个？**

- `NoOpPasswordEncoder` —— 明文（测试用）
- `BCryptPasswordEncoder` —— **推荐**（加盐哈希）
- `SCryptPasswordEncoder`
- `DelegatingPasswordEncoder` —— Spring Security 5+ 默认（支持多算法）

**最佳实践**：使用 `PasswordEncoderFactories.createDelegatingPasswordEncoder()`

### **6. JWT 集成（简历高频）**

**Q: 如何在 Spring Security 中集成 JWT？**

核心步骤：
1. 禁用 Session（`sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)`）
2. 自定义 `JwtAuthenticationFilter` 继承 `OncePerRequestFilter`
3. 在过滤器中解析 Token，生成 `UsernamePasswordAuthenticationToken`
4. 放入 `SecurityContextHolder`
5. 添加 `JwtAuthenticationEntryPoint` 处理未认证请求

### **7. 常见配置（Spring Boot）**

**Q: Spring Boot 中如何配置 Spring Security？**

```java
@Configuration
@EnableWebSecurity
@RequiredArgsConstructor
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.csrf().disable()
            .sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            .and()
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**", "/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }
}
```

### **8. 高频进阶问题**

**Q: @EnableGlobalMethodSecurity 和 @EnableMethodSecurity 的区别？**

- `@EnableGlobalMethodSecurity`（旧版，已过时）
- `@EnableMethodSecurity`（Spring Security 5.6+ 新注解，推荐使用）

**Q: 如何防止 CSRF 攻击？**

- 默认开启，表单提交需带 `_csrf` token
- 前后端分离 + JWT 时通常关闭（`.csrf().disable()`）

**Q: UserDetailsService 和 UserDetails 的作用？**

- `UserDetails`：用户信息载体（用户名、密码、权限、是否过期等）
- `UserDetailsService`：根据用户名加载用户（通常从数据库查）
