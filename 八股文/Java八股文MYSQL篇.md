### 1. MySQL 基础与存储引擎

**1. MySQL 的逻辑架构是怎样的？（从上到下）**

- **连接层**：客户端连接处理、权限认证、连接池等。
- **SQL 层**（服务层）：查询解析、优化、缓存（MySQL 8.0 已移除查询缓存）、内置函数、存储过程、触发器、视图等。
- **存储引擎层**：负责数据的存储和提取，支持插件式（如 InnoDB、MyISAM）。
- **文件系统层**：数据、日志、索引等物理存储。  黄

**2. InnoDB 和 MyISAM 的主要区别？（必问，高频）**

|          |                         |                   |
| -------- | ----------------------- | ----------------- |
| 维度       | InnoDB                  | MyISAM            |
| **默认引擎** | MySQL 5.5+ 默认           | 早期默认              |
| **事务支持** | 支持（ACID）                | 不支持               |
| **锁粒度**  | 行级锁（默认）、表级锁             | 仅表级锁              |
| **外键支持** | 支持                      | 不支持               |
| **崩溃恢复** | 支持（redo log + undo log） | 不支持               |
| **MVCC** | 支持（实现可重复读）              | 不支持               |
| **适用场景** | 高并发、事务、行级锁需求            | 读多写少、非事务场景（如报表）   |
| **索引实现** | 聚簇索引（主键）+ 非聚簇索引         | 非聚簇索引（所有索引都是）     |
| **文件存储** | .ibd（数据+索引）、redo/undo   | .MYD（数据）、.MYI（索引） |

InnoDB 适合生产环境，MyISAM 适合只读或低并发场景。

**3. MySQL 5.7 和 8.0 的主要区别？（版本追问）**

- **数据字典**：8.0 使用事务型数据字典（InnoDB 表），不再是文件系统元数据，提升可靠性。
- **查询缓存**：8.0 完全移除（因写操作频繁失效，命中率低）。
- **默认字符集**：8.0 从 latin1 改为 utf8mb4。
- **索引增强**：8.0 支持降序索引、隐藏索引、函数索引。
- **InnoDB 改进**：更高的并发、原子 DDL、并行复制优化、更强的 JSON 支持。
- **安全性**：8.0 默认使用 caching_sha2_password 插件，更安全。
- **其他**：8.0 GROUP BY 不再隐式排序；支持窗口函数等。

生产升级需注意兼容性（SQL 语法、参数）。

### 2. 索引（核心高频模块）

**4. 为什么 MySQL 索引使用 B+ 树，而不是 B 树或 Hash？**

- **B 树**：每个节点存键 + 数据，非叶子节点也存数据，适合随机访问，但范围查询需遍历更多节点。
- **B+ 树**：非叶子节点只存键（索引），叶子节点存数据 + 双向链表指针。优点：

- 层级更矮（相同数据量，IO 次数少）。
- 范围查询高效（叶子节点链表顺序扫描）。
- 叶子节点存全部数据，查询更稳定。

- **Hash**：等值查询 O(1)，但不支持范围查询、排序、模糊查询，且碰撞问题多。MySQL 只在 Memory 引擎等场景用 Hash。

InnoDB 默认使用 B+ 树。

**5. 聚簇索引 vs 非聚簇索引（二级索引）？什么是回表？**

- **聚簇索引**（Clustered Index）：叶子节点直接存放整行数据。一张表**只能有一个**（通常是主键，如果无主键则选唯一非空键，否则隐式生成 ROWID）。数据物理顺序与索引顺序一致。
- **非聚簇索引**（Secondary Index）：叶子节点存放**索引列值 + 主键值**。查询时需通过主键回聚簇索引取完整数据，这个过程叫 **回表**（两次 B+ 树查询，性能损耗）。

**覆盖索引**（Covering Index）：查询所需的所有字段都在索引（二级索引）的叶子节点中，直接从索引取数据，无需回表。优化手段：将常用查询列加入联合索引。

**6. 最左前缀原则是什么？索引会失效的场景有哪些？**  
联合索引 (a, b, c) 时，查询必须从最左列开始匹配（最左优先）。

- 生效：`WHERE a=1 AND b=2`、`WHERE a=1 AND b>2`（范围后列失效，但 a 仍用）。
- 失效常见场景：

- 违反最左前缀：`WHERE b=2 AND c=3`（跳过 a）。
- 索引列上用函数/表达式：`WHERE DATE(create_time) = '2025-01-01'`（用 `create_time >= ...` 代替）。
- LIKE 以 % 开头：`WHERE name LIKE '%abc'`。
- OR 连接且部分列无索引。
- 类型不匹配（隐式转换，如字符串字段传数字）。
- 范围查询后列失效。

**索引下推（Index Condition Pushdown）**：MySQL 5.6+ 优化，联合索引中，在回表前先用索引过滤部分条件，减少回表次数。

**7. 什么时候创建/不创建索引？如何优化索引？**  
创建：高区分度列、WHERE/GROUP BY/ORDER BY 常用列、联合索引覆盖多查询。  
不创建：更新频繁、区分度低（如性别）、数据量小、长文本列。

优化技巧：

- 主键自增（避免页分裂）。
- 前缀索引（长字符串只索引前 N 字符）。
- 覆盖索引减少回表。
- 联合索引顺序：区分度高、查询频率高的放左边。
- 定期分析表（ANALYZE TABLE）、删除冗余索引。

### 3. 事务与隔离级别（超级高频）

**8. 事务的 ACID 特性是什么？**

- **A（Atomicity）**：原子性，要么全成功，要么全失败（undo log 实现回滚）。
- **C（Consistency）**：一致性，指多个副本或多次读取看到的数据是否“对齐”。常见说法：

- **强一致性**：写入成功后，任何地方立刻读到最新值（或表现得像单副本）。
- **最终一致性**：短时间内可能读到旧值，但在没有新写入后，最终会收敛到同一值。
- 还有“读己之写”“单调读”等更细的模型。

- **I（Isolation）**：隔离性，并发事务互不影响（通过锁 + MVCC 实现）。
- **D（Durability）**：持久性，提交后数据永久保存（redo log 实现）。

**9. MySQL 的四种事务隔离级别？各自解决什么问题？（必背 + 追问）**

|   |   |   |   |   |
|---|---|---|---|---|
|隔离级别|脏读|不可重复读|幻读|备注|
|**READ UNCOMMITTED**|可能|可能|可能|几乎不用|
|**READ COMMITTED (RC)**|不可能|可能|可能|Oracle 默认，生产有时用|
|**REPEATABLE READ (RR)**|不可能|不可能|可能（InnoDB 通过间隙锁解决）|MySQL InnoDB **默认**|
|**SERIALIZABLE**|不可能|不可能|不可能|串行执行，性能差|

- **脏读**：读到未提交数据。
- **不可重复读**：同一事务多次读同一行，结果不同（别人已提交修改）。
- **幻读**：同一事务多次查询同一范围，结果行数不同（别人插入/删除）。

InnoDB 在 RR 下用 **MVCC + Next-Key Lock（间隙锁 + 行锁）** 基本解决幻读。

**10. MVCC（多版本并发控制）是怎么实现的？（经典追问）**  
MVCC 让读不阻塞写，实现非阻塞读。核心：

- **隐藏字段**：每行记录有 `DB_TRX_ID`（事务 ID）、`DB_ROLL_PTR`（回滚指针指向 undo log）。
- **undo log**：构建版本链（旧版本链表）。
- **ReadView**：事务启动时生成快照（包含当前活跃事务 ID 列表 min_trx_id、max_trx_id、creator_trx_id）。
- **可见性判断**：比较当前记录版本的事务 ID 与 ReadView，若可见则返回，否则沿 undo log 找旧版本。

RC 级别每次读都生成新 ReadView；RR 级别事务开始时生成一次 ReadView（保证可重复读）。

### 4. 锁机制

**11. MySQL 有哪些锁？按粒度/类型分类**

- **按粒度**：表锁、行锁、页锁（InnoDB 主要行锁 + 表锁）。
- **按类型**：共享锁（S 锁，读锁）、排他锁（X 锁，写锁）。
- **意向锁**：表级意向共享/排他锁（IS/IX），提高加锁效率。
- **Next-Key Lock**：间隙锁（Gap Lock） + 行锁，防止幻读（锁定区间）。
- **记录锁**：锁具体行。
- **间隙锁**：锁两个记录之间的间隙，防止插入。

死锁：两个或以上事务互相等待对方释放锁。避免：统一加锁顺序、超时设置、用 RC 隔离级别（间隙锁少）。

**12. 行锁是怎么实现的？什么时候会锁表？**  
InnoDB 行锁通过在索引上加锁实现（无索引则锁表）。更新/删除时加排他锁，查询在 RR 当前读时加锁。

### 5. SQL 优化与执行计划

**13. 如何定位和优化慢查询？**

- 开启慢查询日志（slow_query_log），设置 long_query_time。
- 用 **EXPLAIN** 查看执行计划（重点看 type、key、rows、Extra）：

- type：system > const > eq_ref > ref > range > index > ALL（全表扫描最差）。
- Extra：Using index（覆盖索引）、Using where、Using temporary、Using filesort（需优化）。

- 优化方向：加合适索引、避免 * 查询、拆分复杂 SQL、分页优化（延迟关联）、批量操作。

- **redo log、undo log、binlog 的作用和区别？**（WAL 机制：Write-Ahead Logging，先写日志再写磁盘）。

- redo：物理日志，Crash-Safe 恢复。
- undo：逻辑日志，回滚 + MVCC。
- binlog：逻辑日志，主从复制、备份。

- **一条 SQL 的执行过程？**（连接 → 解析 → 优化 → 执行引擎 → 存储引擎）。
- **分库分表如何做？**（垂直/水平、Sharding 策略、路由、中间件如 ShardingSphere）。
- **主从复制原理？**（异步/半同步，binlog + relay log）。
- **如何防止 SQL 注入？**（预编译 PreparedStatement、ORM 参数绑定）。
